#### UIレイヤー

- **責務**: ユーザー操作の受付、フローエディタの描画、ノードライブラリ・プロパティパネルの表示
- **許可される操作**: アプリケーションレイヤーの呼び出し
- **禁止される操作**: IPC・バックエンドへの直接アクセス、画像処理ロジックの実装

#### アプリケーションレイヤー

- **責務**: 編集状態管理（Zustand）、永続データ管理（TanStack Query）、Undo/Redo、フローバリデーション、プロセッサ定義レジストリの管理
- **編集状態（Zustand）**: 編集中フローの作業状態（ノード・接続・パラメータ）をメモリ内で保持。ビューポート位置・ズーム、ノード選択状態、ドラッグ中の一時状態、Undo/Redo操作履歴もここで管理。パラメータ変更・ノード移動はメモリ内で即座に反映される
- **永続データ（TanStack Query）**: プロジェクト一覧の取得、保存済みフローのSQLiteへの読み書きを管理。保存・読み込みのタイミングでZustandの編集状態と同期する
- **許可される操作**: IPCレイヤーを通じたバックエンドの呼び出し
- **禁止される操作**: UIレイヤーへの依存、ファイルI/O・DB操作の直接実行

#### IPCレイヤー

- **責務**: フロントエンドとバックエンド間のコマンド・イベントの仲介、データのシリアライズ/デシリアライズ
- **許可される操作**: Tauri Commandsの定義と呼び出し、Eventsの発行と購読
- **禁止される操作**: ビジネスロジックの実装

#### バックエンドレイヤー（Rust）

- **責務**: SQLiteデータベースの管理、プロジェクトデータのCRUD、自動バックアップ、画像ファイルI/O、C++画像処理エンジンの呼び出し制御
- **許可される操作**: SQLiteアクセス、ファイルシステムアクセス（Tauriスコープ内）、C++ FFI呼び出し
- **禁止される操作**: UIレイヤーへの依存

#### 画像処理レイヤー（C++）

- **責務**: 画像処理アルゴリズムの実行（各プロセッサの処理本体）
- **許可される操作**: OpenCVを使用した画像処理、メモリ上の画像データ操作
- **禁止される操作**: ファイルI/O（Rustバックエンド経由で行う）、UI・状態管理への依存

## プロセッサレジストリ設計

データモデル設計書で定義した2層分離を実現するための仕組み。

### レジストリの役割

- 全 `ProcessorDefinition` を管理し、`typeId` による検索を提供する
- フロントエンド側（ノードライブラリ表示、バリデーション）とバックエンド側（処理実行）の両方で参照される
- フロントエンド側: ProcessorDefinition のメタデータ（名前、ポート構成、パラメータ定義）を保持
- バックエンド側: typeId に対応する C++ 処理関数へのディスパッチ

### 拡張方法

新しいプロセッサ種別を追加する際の手順:
1. `ProcessorDefinition`（メタデータ）をレジストリに登録
2. C++ 側に処理関数を実装
3. typeId とC++処理関数のマッピングを追加

グラフモデル層の変更は不要。

## データ永続化戦略

### ストレージ方式

| データ種別 | ストレージ | フォーマット | 理由 |
|---|---|---|---|
| プロジェクトデータ | SQLite | テーブル | トランザクションによるデータ整合性、構造化クエリ、将来の検索・集計への拡張性 |
| フロー定義（ノード・接続・パラメータ） | SQLite | テーブル | プロジェクトと一元管理、リレーショナルな参照整合性 |
| 検査画像 | ローカルファイルシステム | 画像ファイル | バイナリデータはファイルシステムで管理。SQLiteにはパスのみ保持 |
| プロセッサ定義 | アプリケーション組み込み | TypeScript/Rust定数 | バージョン管理をアプリと一体化 |
| アプリ設定 | ローカルファイルシステム | JSON | Tauri標準の設定管理方式 |

### SQLiteデータベース構成

- DBファイルはTauriのアプリデータディレクトリに配置（例: `{app_data}/visualinspect.db`）
- Rustバックエンドが `rusqlite` または `sqlx` でアクセス
- フロントエンドはTauri Commandsを経由してCRUD操作を行い、TanStack Queryでキャッシュ・同期を管理

### フロントエンドの状態管理パターン

```
┌───────────────────────────────────────────────────────────┐
│  UIコンポーネント                                           │
│                                                            │
│  ┌──────────────────────────────┐  ┌───────────────────┐  │
│  │ Zustand                      │  │ TanStack Query    │  │
│  │ (編集状態 + UI状態)           │  │ (永続データ)       │  │
│  │                              │  │                   │  │
│  │ 【編集中フロー作業状態】       │  │ - プロジェクト一覧 │  │
│  │ - ノード配置・接続・パラメータ │  │ - 保存済みフロー   │  │
│  │ - ダーティフラグ              │  │   の読み書き      │  │
│  │                              │  │                   │  │
│  │ 【UI状態】                    │  │                   │  │
│  │ - ビューポート位置            │  │                   │  │
│  │ - 選択中ノード                │  │                   │  │
│  │ - ドラッグ状態                │  │                   │  │
│  │ - Undo/Redo履歴             │  │                   │  │
│  │ - モーダル開閉                │  │                   │  │
│  └──────────┬───────────────────┘  └───────┬───────────┘  │
│             │                              │               │
│             │  保存 / 自動保存 /            │               │
│             │  プロジェクト切替時に同期      │               │
│             ▼                              ▼               │
│        ┌──────────────────────────────────────┐            │
│        │  Tauri Commands (IPC)                 │            │
│        └──────────────────┬───────────────────┘            │
│                           ▼                                │
│        ┌──────────────────────────────────────┐            │
│        │  Rust Backend (SQLite CRUD)           │            │
│        └──────────────────────────────────────┘            │
└───────────────────────────────────────────────────────────┘
```

**Zustand（編集状態 + UI状態）**:
- 編集中フローのノード・接続・パラメータをメモリ内に保持し、ユーザー操作を即座に反映する
- Undo/Redoはメモリ内のコマンド履歴で完結し、SQLiteへの往復を伴わない
- ダーティフラグで未保存の変更を追跡する

**TanStack Query（永続データ）**:
- プロジェクト一覧の取得、保存済みフローのSQLiteへの読み書きを `useQuery` / `useMutation` で管理
- Tauri Commandsをクエリ関数として使用

**同期タイミング**:
- フローの読み込み時: TanStack Query → Zustand（SQLiteから取得したフロー定義を編集状態にロード）
- 明示的な保存（Ctrl+S等）: Zustand → TanStack Query → SQLite
- 自動保存（5分ごと）: Zustand → TanStack Query → SQLite（ダーティフラグがtrueの場合のみ）
- プロジェクト切替・アプリ終了時: 未保存変更がある場合は保存確認ダイアログを表示

### バックアップ戦略

- **自動保存**: 5分ごとにZustandの編集状態をSQLiteに書き出し（ダーティフラグがtrueの場合のみ）
- **DB整合性**: SQLiteのWAL（Write-Ahead Logging）モードにより、書き込み中のクラッシュでもデータ整合性を保証
- **定期バックアップ**: SQLiteデータベースファイルのコピーを定期的に作成
- **保存先**: アプリデータディレクトリ内の `backups/` に最新3世代を保持
- **復元方法**: アプリ起動時にメインDBの整合性チェックを実施。破損検出時はバックアップからの復元を提案

### プロジェクトのエクスポート/インポート

- SQLiteに保存されたプロジェクトデータをJSON形式（.vinsp）でエクスポート可能
- エクスポートファイルは他のPCへのプロジェクト移行や、バージョン管理（Git）に利用
- インポート時はJSONからSQLiteにデータを投入

|---|---|---|

## セキュリティアーキテクチャ

### データ保護

- **ローカルデータの原則**: プロジェクトデータ・検査画像はローカルに保存し、明示的なクラウド連携機能（将来フェーズ）を有効化しない限り外部送信しない
- **ファイルアクセス制限**: Tauriのスコープ機能により、ユーザーが明示的に選択したディレクトリのみにアクセスを限定
- **機密情報**: MVPではユーザー認証なし。機密情報（パスワード等）は扱わない

### 入力検証

- **データベース整合性**: SQLiteの外部キー制約・CHECK制約によりデータ整合性を保証。インポート時はJSONスキーマによる構造検証を実施
- **パラメータのバリデーション**: ParameterDefinitionのconstraintsに基づくリアルタイム検証
- **画像入力のバリデーション**: サポートする画像形式（PNG, JPEG, BMP, TIFF）のみ受け付け

## スケーラビリティ設計

### データ増加への対応

- **フロー規模**: 1フローあたり最大200ノードの編集・実行をサポート
- **プロジェクト数**: 1アプリインスタンスで最大50プロジェクトを管理
- **パフォーマンス劣化対策**: 大規模フロー（100ノード超）ではキャンバスの仮想化描画（ビューポート外のノードの描画省略）を適用

### 機能拡張性

- **プロセッサレジストリ**: 新しいノード種別はProcessorDefinitionの追加とC++処理関数の実装で対応。グラフモデル層の変更不要
- **PortDataType拡張**: 将来のAIノードに向けてPortDataTypeに新しい型を追加可能な設計
- **フェーズ2以降のAI統合**: Python（AI学習）との連携はRustバックエンドからのプロセス起動で対応予定

## テスト戦略

### ユニットテスト

- **フレームワーク**: Vitest（フロントエンド）、Google Test（C++）
- **対象**:
  - フロントエンド: Zustand stores（編集状態管理・Undo/Redo）、TanStack Queryのクエリ関数、フローバリデーション、プロセッサレジストリ
  - C++: 各画像処理プロセッサの処理結果の正確性
- **カバレッジ目標**: 80%以上（UIコンポーネントを除くロジック部分）

### 統合テスト

- **方法**: Tauri IPCを介したフロントエンド⇔バックエンド間のコマンド・レスポンス検証
- **対象**: SQLite CRUD操作、プロジェクトのエクスポート/インポート、フロー実行パイプライン

### E2Eテスト

- **ツール**: Playwright（Tauri + WebDriver）
- **シナリオ**:
  - プロジェクト作成 → ノード配置 → 接続 → パラメータ設定 → フロー実行 → 結果確認
  - プロジェクトエクスポート → 別環境でインポート → フロー整合性確認

## 技術的制約

### 環境要件

- **OS**: Windows 10以降（MVP）
- **最小メモリ**: 8GB
- **推奨メモリ**: 16GB以上
- **必要ディスク容量**: 1GB（アプリ本体 + 依存ライブラリ）
- **ディスプレイ**: 1280x720以上
- **必要な外部依存**: WebView2ランタイム（Windows。Tauriが自動検出・インストール案内）

### パフォーマンス制約

- フロントエンド⇔Rustバックエンド間のIPC通信にシリアライズ/デシリアライズのオーバーヘッドが発生する。大きな画像データの転送はBase64エンコードではなく、共有メモリまたはファイルパス参照で最適化する
- C++ FFI呼び出しのオーバーヘッドを最小化するため、画像処理パイプラインは可能な限りC++側でまとめて実行する
- リアルタイムプレビュー時のデータフローはSQLiteを経由しない。Zustandのメモリ内フロー定義からIPC経由でC++処理を直接呼び出し、200ms以内のプレビュー更新を実現する

### セキュリティ制約

- Tauriのセキュリティモデルに従い、フロントエンドからのファイルシステムアクセスは全てRustバックエンドのTauri Commandsを経由する
- 工場ネットワーク内でのオフライン動作を前提とし、外部ネットワークへの接続は行わない
