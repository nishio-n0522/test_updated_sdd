# スペック駆動開発（SDD）の取り組み報告

## 1. 背景：AIコーディングにおける課題

### 1-1. AI活用の現状と期待

- GitHub Copilot、Claude Code等のAIコーディングツールの急速な普及
- 「AIにコードを書かせれば開発が速くなる」という期待
- 実際にはコード生成だけでは解決しない問題が多い

### 1-2. 既存の困りごと

- **仕様の曖昧さ**: 口頭やチャットベースの指示では、AIが意図と異なるコードを生成しがち
- **コンテキストの断絶**: AIとのセッションが変わると、前回の文脈が失われ同じ説明を繰り返す
- **品質のばらつき**: 同じ機能でもセッションごとに異なるアーキテクチャ・命名規則で実装される
- **ドキュメント不在**: AIが生成したコードの設計意図が残らず、保守時に困る
- **手戻りの多さ**: 仕様が固まらないまま実装に入り、大幅な手戻りが発生する

### 1-3. 根本的な問題意識

- AIは「何を作るか」を決めてくれない。決めるのは人間の仕事
- しかし「何を作るか」を構造化して伝える仕組みがない
- **→ AIの能力を最大限引き出すには、仕様（スペック）を先に整備する必要がある**

---

## 2. 取り組み内容：スペック駆動開発（SDD）の構築

### 2-1. コンセプト

- **「仕様ドキュメントを書いてから実装する」をAIとの協働で実現する開発手法**
- AIが仕様ドキュメントの作成を支援し、そのドキュメントに基づいてAIが実装する
- 人間はレビューと意思決定に集中する

### 2-2. 仕組みの概要

#### ドキュメント体系（`docs/`）

プロジェクト全体の「北極星」となる永続的な仕様ドキュメント群を定義:

| ドキュメント | 役割 |
|---|---|
| PRD（プロダクト要求定義書） | 何を作るか・なぜ作るか |
| 機能設計書 | 機能分解・ドメインモデル・UI概要 |
| データモデル設計書 | エンティティ定義・ER図 |
| 画面仕様書 | 詳細な画面レイアウト・操作挙動 |
| アーキテクチャ設計書 | 技術構成・非機能要件 |
| 開発ガイドライン | コーディング規約・開発プロセス |
| デザインパターン | 共通の実装パターン |
| 用語集 | ユビキタス言語 |

#### プロジェクト立ち上げフロー（`/new-project`）

1. アイデアメモからAIが仕様ドキュメントを順次生成
2. 各ドキュメントは**自動レビュー → 人間の承認**を経て確定
3. 承認済みドキュメントからissueを自動分解・GitHub登録

#### 自動実装フロー（`/autopilot`）

1. 登録済みissueを番号順に自動取得
2. issueごとにブランチを作成し、計画→実装→レビュー→PR作成を自律実行
3. 全issue完了後に完了レポートを出力

### 2-3. 技術基盤

- **Claude Code**: AIコーディングアシスタントとしてのベースツール
- **Skills/Agents**: 20以上の専門スキル・エージェントを定義し、役割分担を実現
- **DevContainer**: 環境の再現性を保証
- **GitHub CLI連携**: issue管理・PR作成を自動化

### 2-4. 工夫したポイント

- **承認ゲート**: 1ドキュメントずつ人間が確認し、AIの暴走を防ぐ
- **ドキュメント駆動のエージェント間連携**: エージェント同士は`.issue/`配下のファイルを介して情報を受け渡す
- **レビューの多層化**: コーディング規約・アーキテクチャ・i18nなど観点別にレビューエージェントを分離
- **段階的UI設計**: 概要→詳細の2段階で設計を深掘りし、早すぎる詳細化を回避

## 3. 実践から得た知見

### 3-1. 最大の学び

書籍の内容（ドキュメントを作成して開発を進める手法）自体は既知であった。最も重要な学びは、**AIがより良い出力をするためにはドキュメントの書き方・コンテキスト管理などのトライ&エラーを通じたエンジニアリングプロセスが不可欠である**ということだった。

### 3-2. 直面した課題と解決策

#### コンテキスト肥大化による精度低下

- **現象**: 1セッションでドキュメント読み込み→UI差分設計→実装を一括実行すると、コンテキストが肥大化しAIの出力精度が著しく低下。特にASCIIワイヤーフレームやMarkdownでの詳細なUI指示はトークン消費が大きく、意図しないUI変更しかできなかった
- **解決策**: サブエージェントの境界でコンテキストをリセットし、各agentが必要なドキュメントだけを読み込む設計に変更。すべての変更はまずissueとして登録・ドキュメント化してから実行する方式に移行

#### ワークフロー実行時のステップスキップ

- **現象**: `/new-project` 実行時に、機能設計書と画面仕様書がスキップされ、実装仕様設計に直接移行してしまう
- **原因**: オーケストレーターの指示に強制力のある順序制御がなく、AIがステップを省略
- **解決策**: 実行ルールの厳守セクション追加、各ステップへの前提条件・出力ファイル存在確認の導入

### 3-3. スペック駆動開発の3つの柱

#### デザインパターンによる実装の再現性

- フロントエンド開発はデータフェッチ・フォーム送信など、パターン化しやすい実装が多い
- これらをデザインパターンとして明文化し、AIが従うべき実装テンプレートを提供する
- 新しい技術スタック導入時はPoC検証しながらパターンを確立する
- パターンがない場合AIの出力がブレるため、先にパターンを作ることが重要

#### 直列自動実行によるスループット最大化

- 並列実行すると変更が競合する可能性があり、完全なコントロールが難しい（実践で確認済み）
- issueに作業内容を記載し、autopilotで直列に全自動実行する方式に移行
- AIが24時間体制でコードを出力し、ユーザーはPRで適宜確認するだけでよい
- 「並列実行 × 競合リスク」より「直列実行 × 24時間稼働」の方が確実に生産性が高い

#### コンテキスト肥大化の制御

- 1セッションに多くの処理を詰め込むと、コンテキストが肥大化してAIの出力精度が著しく低下する
- サブエージェントの境界でコンテキストをリセットし、各agentが必要なドキュメントだけを読み込む
- すべての変更はissueとして登録・ドキュメント化してから実行する

## 4. 今後に向けて

### 4-1. 現在の成果と手応え

- テンプレートリポジトリとして仕組みが整備完了
- アイデアメモから仕様ドキュメント群の生成・issueへの分解が一気通貫で動作
- 「人間が考え、AIが書き、人間が確認する」ワークフローの確立

### 4-2. 今後の課題

- 実プロジェクトでの適用・検証（実際のプロダクト開発での効果測定）
- 既存プロジェクトへの段階的導入方法の確立
- チーム開発での運用フロー整備（複数人での承認・レビュープロセス）

### 4-3. 重要な注意事項

**スペック駆動開発はまだまだ発展途上である。** 本報告の内容はフロントエンド開発に最適化された一例であり、万能な正解ではない。フロントエンドはUIパターンが定型化しやすくコンポーネント単位で分割しやすいため上記のアプローチが有効に機能したが、バックエンド（C++等）やインフラなど異なる領域では同じアプローチが通用するとは限らない。

重要なのは特定のワークフローを真似ることではなく、**各チームが自分たちの開発プロセスに合わせてAIへの指示体系をエンジニアリングすること**である。そのプロセス自体がトライ&エラーの繰り返しであり、各チームが自分たちに合った形を模索する必要がある。

### 4-4. 展望

- SDDテンプレートの社内標準化・横展開
- プロジェクト特性に応じたドキュメント体系のカスタマイズ
- AI技術の進化に合わせたワークフローの継続的改善
