# 画面改善 (変更仕様承認後に自動実行)

**重要:** このワークフローは、Step 3（変更仕様書の作成）でユーザーの承認を得た後、実装完了・画面仕様書更新まで自動で実行されます。Step 3の承認ゲートを除き、各ステップは完了後ただちに次のステップへ移行してください。

**引数:** 変更内容（任意で画面ID指定。例: `/improve-screen SCR-001 保存後にトースト通知を追加`、`/improve-screen タイムラインにプルダウンリフレッシュを追加`）

**`/add-feature` との使い分け:**

| | /add-feature | /improve-screen |
|---|---|---|
| 起点 | PRDの機能要件 | 画面仕様書 |
| 対象 | 新機能（バックエンド含む） | 画面UI/UX改善 |
| 変更仕様書 | なし | あり（Before/After） |
| 承認ゲート | なし（完全自動） | あり（Before/After承認） |
| 画面仕様更新 | 手動 | ワークフローに組み込み |

---

## ステップ1: 準備とコンテキスト設定

1. 現在のタスクコンテキストを確立する:

- 変更内容: `[引数で与えられた変更内容]`
- 日時: `[現在の日時をYYYYMMDDHHmmss形式で取得]`
- ステアリングディレクトリパス: `.steering/[日時]-improve-[変更内容を英語で短縮した名前]/`
  - 例: `/improve-screen SCR-001 保存後にトースト通知を追加` → `.steering/20260219120000-improve-toast-notification/`
  - 例: `/improve-screen タイムラインにプルダウンリフレッシュを追加` → `.steering/20260219120000-improve-pull-to-refresh/`

2. 上記ステアリングディレクトリを作成する。
3. 以下の4つの空ファイルを作成する:

- `[ステアリングディレクトリパス]/requirements.md`
- `[ステアリングディレクトリパス]/design.md`
- `[ステアリングディレクトリパス]/tasklist.md`
- `[ステアリングディレクトリパス]/change-spec.md`

## ステップ2: 現状把握

1. `CLAUDE.md`を読み、プロジェクトの全体像を把握する。
2. `docs/screen-specification/index.md` を読み、画面全体の構造を把握する。
3. 引数で画面IDが指定されている場合、該当する画面仕様ファイル（`docs/screen-specification/scr-[番号]-[name].md`）を読み込む。
4. 引数で画面IDが指定されていない場合、`docs/screen-specification/index.md` の画面一覧と変更内容を照合して影響を受ける画面を特定し、該当する画面仕様ファイルを読み込む。
5. `docs/product-requirements.md` から関連する機能要件を確認する。
6. 影響を受ける画面の**ソースコードを調査**し、現在の実装構造を把握する:
   - 変更対象の画面名・ビュー名に関連するキーワードでソースコードを Grep 検索する
   - 変更対象のビューファイル・コンポーネントファイルを特定して読み込む
   - 変更仕様書の実現可能性を判断するための実装上の制約を把握する

## ステップ3: 変更仕様書の作成（★ ユーザー承認必須）

1. `Skill('screen-improvement')` をロードする。
2. スキルのガイドに従い、`[ステアリングディレクトリパス]/change-spec.md` にBefore/After変更仕様書を作成する。

**ユーザー承認（必須）**:

```
「変更仕様書（change-spec.md）の作成が完了しました。
Before/Afterの内容を確認してください。

[変更のサマリを簡潔に記述]

承認いただけたら実装に進みます。修正が必要な場合はお知らせください。」
```

**承認されるまで絶対に次に進まないこと。** ユーザーからの修正指示があれば変更仕様書を修正し、再度承認を求める。

## ステップ4: 計画フェーズ（ステアリングファイルの自動生成）

1. `Skill('steering')` を**計画モード**で実行し、ステップ1で作成した3つのファイル（`requirements.md`, `design.md`, `tasklist.md`）の内容を生成する。その際、以下を計画の前提として使用すること:
   - **主要入力**: `[ステアリングディレクトリパス]/change-spec.md`（承認済みの変更仕様）
   - **変更対象**: ステップ2で特定した画面仕様ファイルとソースコード

2. **重要**: `tasklist.md` の最終フェーズに以下のタスクを必ず含めること:

```markdown
## フェーズN: 画面仕様書の更新
- [ ] change-spec.md の After セクションを元に画面仕様書を更新する
- [ ] index.md の更新が必要な場合は更新する
```

3. `requirements.md` には変更仕様書（`change-spec.md`）への参照を含めること。

4. **このステップが正常に完了したら、決して停止せず、ただちにステップ5に進むこと。**

## ステップ5: 実装ループ（tasklist.mdの完全消化）

**このステップは、`tasklist.md`の全タスクが `[x]` になるまで自動で繰り返されるループ処理です。**
**ただし、`## フェーズN: 画面仕様書の更新` 見出し配下のタスクは除外してループする（画面仕様書の更新はステップ7で実施）。** この見出しはステップ4で `tasklist.md` に追加した最終フェーズを指す。
**このステップが正常に完了したら、決して停止せず、ただちにステップ6に進むこと。**

**ループ開始:**

1. タスクリストの読み込み:

- `[ステアリングディレクトリパス]/tasklist.md`ファイルを読み込む。

2. 進捗の確認:

- ファイル内に未完了タスク (`[ ]`) が存在するか確認する（「画面仕様書の更新」フェーズを除く）。
- **もし未完了タスクが存在しない場合:** この実装ループは完了とみなし、ただちに**ステップ6**へ進む。
- **もし未完了タスクが存在する場合:** 次の処理（3. タスクの実行）に進む。

3. タスクの実行:

- `tasklist.md`の**先頭にある未完了タスク**を1つ特定する（「画面仕様書の更新」フェーズを除く）。
- そのタスクを完了させるために必要な実装作業を実行する。
- `Skill('steering')` を**実装モード**で使用する。
- `Skill('development-guidelines')` のコーディング規約を常に遵守する。

4. タスクリストの更新:

- 実行したタスクが完了したら、`Edit`ツールを使用して`tasklist.md`を更新し、該当タスクを `[ ]` から `[x]` に変更する。

5. ループ継続:

- **ステップ5の先頭 (1. タスクリストの読み込み) に戻り、処理を繰り返す。**

---

### ※ 実装ループ内の例外処理ルール ※

実装ループの実行中に以下の状況が発生した場合は、このルールに従って自律的に対処し、ループを継続すること。

- **ルールA: タスクが大きすぎる場合**
  - **対処法:** 現在のタスクをより小さな複数のサブタスクに分割する。`Edit`ツールを使い、元のタスクを削除し、その場所に新しいサブタスク（`[ ]`付き）を挿入する。その後、ループを継続する。

- **ルールB: 技術的理由でタスクが不要になった場合**
  - **条件:** 実装方針の変更、アーキテクチャの変更、依存関係の変更など、明確な技術的理由がある場合にのみ適用。
  - **対処法:** `Edit`ツールを使い、該当タスクを `[x] ~~タスク名~~ (理由: [具体的な技術的理由を簡潔に記述])` の形式で更新する。その後、ループを継続する。

- **❌ 絶対禁止の行為:**
  - 未完了タスクを「後でやる」「別タスクにする」などの理由で意図的にスキップすること。
  - 理由なく未完了タスクを放置してループを終了させること。
  - 実装ループ中にユーザーに判断を仰ぐこと（承認ゲートはステップ3・ステップ7のみ）。

---

## ステップ6: 実装検証（ビルド確認）

1. `tasklist.md`の実装タスクが全て完了したことを確認する（`## フェーズN: 画面仕様書の更新` 配下のタスクを除く）。
2. プロジェクトの技術スタックに応じたビルド・コンパイルチェックを実行し、エラーがないことを確認する。
   - `CLAUDE.md` および `docs/architecture.md` に記載されているビルドコマンドを参照する
3. `docs/architecture.md` に静的解析・リントコマンドが定義されている場合は、それらを実行してエラーがないことを確認する。
4. エラーがある場合は修正し、再度チェックを実行する。

**このステップが正常に完了したら、決して停止せず、ただちにステップ7に進むこと。**

## ステップ7: 画面仕様書の更新（★ 1ファイルずつユーザー承認）

1. `Skill('screen-specification')` を**更新モード**でロードする。
2. `change-spec.md` の After セクションの内容を元に、画面仕様書を更新する。

**更新の手順:**

1. `change-spec.md` を読み込み、更新が必要なファイルを特定する。
2. 各ファイルを**1ファイルずつ**更新する。
3. 各ファイルの「最終更新日」を更新する。

**1ファイルずつユーザーの承認を得てから次に進む:**

```
「[ファイル名] を更新しました。変更点を確認してください。
承認いただけたら次のファイルに進みます。」
```

4. 全ファイルの更新が完了したら、`tasklist.md` の「画面仕様書の更新」タスクを `[x]` に更新する。

## ステップ8: 振り返りとドキュメント更新

1. `Skill('steering')` を**振り返りモード**で実行し、`tasklist.md` に申し送り事項を記載する。

- 実装完了日
- 計画と実績の差分
- 学んだこと
- 次回への改善提案

2. 以下のいずれかに該当する場合、`docs/` 内の関連する永続ドキュメントを `Edit` ツールで更新する:
   - 新しいUI操作フローが追加された → `docs/product-requirements.md` の機能要件に影響
   - 既存の画面遷移が変更された → `docs/functional-design.md` の画面遷移図に影響
   - 新しいドメイン用語が生まれた → `docs/glossary.md` に影響
   - 該当しない場合は更新不要

## 完了条件

このワークフローは、以下の全ての条件を満たした時点で完了となる。

- ステップ3: 変更仕様書（change-spec.md）がユーザーに承認されている。
- ステップ5: `tasklist.md`の全ての実装タスクが完了状態（`[x]`または正当な理由でスキップ）になっている。
- ステップ6: プロジェクトのビルドがエラーなく成功する。
- ステップ7: 画面仕様書が変更仕様書のAfterに基づいて更新され、ユーザーに承認されている。
- ステップ8: `tasklist.md`に申し送り事項が記載されている。
