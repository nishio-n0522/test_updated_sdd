# 画面仕様書の作成・更新

**引数（任意）**: 対象画面または操作指示
- 引数なし: 全画面の仕様書を新規作成または全体更新
- 画面指定: 特定画面のみ作成・更新（例: `/screen-spec SCR-001`、`/screen-spec 入力画面`）

---

## ステップ1: screen-specification スキルのロード

1. `Skill('screen-specification')` をロードする。

## ステップ2: モード判定

`docs/screen-specification/` ディレクトリの存在を確認し、モードを決定する。

### 判定ルール

- **ディレクトリが存在しない、または `index.md` が存在しない場合** → **新規作成モード**（ステップ3Aへ）
- **ディレクトリと `index.md` が存在する場合** → **更新モード**（ステップ3Bへ）

判定結果をユーザーに通知する:

```
# 新規作成モードの場合
「docs/screen-specification/ が存在しません。画面仕様書を新規作成します。」

# 更新モードの場合
「docs/screen-specification/ が見つかりました。既存の画面仕様書を更新します。」

# 更新モードで引数がある場合
「docs/screen-specification/ が見つかりました。[引数の画面] の仕様を更新します。」
```

---

## ステップ3A: 新規作成モード

SKILL.md の「作成プロセス」に従って実行する。

### 3A-1: 入力ドキュメントの読み込み

以下を順序どおりに読み込む:

1. `docs/product-requirements.md`
2. `docs/functional-design.md`
3. `.steering/` 配下のディレクトリをフォルダ名の昇順（＝時系列順）に読み込む
   - 各ディレクトリ内の `requirements.md`、`design.md`、`tasklist.md` を読む

### 3A-2: 現在のアプリ状態の把握

読み込んだドキュメントから以下を整理する:

1. 画面一覧: 現在アプリに存在するすべての画面
2. 画面遷移: 画面間のナビゲーション構造
3. 実装済み機能: PRDの機能要件のうち、実装済みのもの
4. UI要素: 各画面に配置されているボタン・入力フィールド等

### 3A-3: index.md の作成

`./templates/index.md` テンプレートに従い、`docs/screen-specification/index.md` を作成する。

**ユーザー確認（必須）**:

```
「index.md（画面一覧・画面遷移図・共通挙動仕様）の作成が完了しました。内容を確認してください。
承認いただけたら各画面の詳細仕様に進みます。」
```

**承認されるまで次に進まないこと。**

### 3A-4: 各画面ファイルの作成

`./templates/screen.md` テンプレートに従い、画面ごとに1ファイルを作成する。

**作成順序**:
1. メイン画面（タブバーから直接アクセスする画面）
2. サブ画面（メイン画面から遷移する画面）
3. モーダル・システムダイアログ

**1ファイルずつユーザーの承認を得てから次の画面に進む**:

```
「SCR-[番号]（[画面名]）の画面仕様書を作成しました。内容を確認してください。
承認いただけたら次の画面に進みます。」
```

### 3A-5: components.md の作成

複数画面で使用される共通コンポーネントがある場合、`./templates/components.md` テンプレートに従い作成する。

**ユーザー確認（必須）**:

```
「components.md（共通コンポーネント仕様）の作成が完了しました。内容を確認してください。」
```

### 3A-6: レビューと完了

生成した仕様書を以下の観点でセルフレビューし、問題があれば修正提案をする:

1. PRDの機能要件との整合性
2. functional-design.md の概要UI設計との整合性
3. steering ファイルで実装済みの変更が反映されているか
4. すべての画面要素に操作挙動が定義されているか
5. 画面遷移に抜け漏れがないか

詳細なレビュー観点は `./guide.md` の「レビューチェックリスト」セクションを参照。

**完了後のステップ4へ進む。**

---

## ステップ3B: 更新モード

SKILL.md の「更新プロセス」に従って実行する。

### 3B-1: 現状の把握

1. `docs/screen-specification/index.md` を読み込む
2. 引数で指定された画面、または更新が必要な画面のファイルを読み込む

### 3B-2: 変更情報の収集

以下の情報源から変更内容を特定する:

- **引数がある場合**: 引数で指定された画面・操作に関する変更
- **引数がない場合**: 最新の steering ファイル、PRD の変更、ユーザーの口頭指示から変更内容を特定

変更内容をユーザーに確認する:

```
「以下の変更を画面仕様書に反映します:
- [変更内容1]
- [変更内容2]

対象ファイル:
- index.md（画面遷移図の更新）
- scr-001-input.md（操作挙動の追加）

この内容で更新を進めてよろしいですか？」
```

**承認されるまで次に進まないこと。**

### 3B-3: 影響範囲の確認と更新

以下の観点で影響範囲を確認し、1ファイルずつ更新する:

- **index.md**: 画面一覧・画面遷移図・PRD対応表の更新が必要か
- **該当画面ファイル**: ワイヤーフレーム・UI要素定義・操作挙動の変更が必要か
- **他画面ファイル**: 遷移元・遷移先の画面に影響があるか
- **components.md**: 共通コンポーネントの変更が必要か

各ファイルの「最終更新日」を更新する。

**1ファイルずつユーザーの承認を得てから次に進む**:

```
「[ファイル名] を更新しました。変更点を確認してください。
承認いただけたら次のファイルに進みます。」
```

### 3B-4: 整合性の確認

更新完了後、以下を確認する:

1. index.md の画面遷移図が更新後の状態と一致しているか
2. 変更した画面の遷移先・遷移元の仕様に矛盾がないか
3. PRD機能要件との対応表が最新か

**完了後のステップ4へ進む。**

---

## ステップ4: 完了報告

```
「画面仕様書の[作成/更新]が完了しました。

[作成/更新]したファイル:
✅ docs/screen-specification/index.md
✅ docs/screen-specification/scr-001-[name].md
✅ docs/screen-specification/scr-002-[name].md
...

今後の使い方:
- 特定画面の更新: /screen-spec SCR-001
- 全体の見直し: /screen-spec
- ドキュメントレビュー: /review-docs docs/screen-specification/*
」
```
