# リファクタリング (設計承認後に自動実行)

**重要:** このワークフローは、Step 4（リファクタリング設計の承認）でユーザーの承認を得た後、実装完了まで自動で実行されます。Step 4の承認ゲートを除き、各ステップは完了後ただちに次のステップへ移行してください。

**引数:** リファクタリング内容 (例: `/refactor 認証モジュールの責務分離`、`/refactor APIクライアントのエラーハンドリング統一`)

**`/add-feature` との使い分け:**

| | /add-feature | /refactor |
|---|---|---|
| 起点 | PRDの機能要件 | コード品質・設計上の課題 |
| 対象 | 新機能（バックエンド含む） | 既存コードの構造改善 |
| 外部から見た振る舞い | 変わる（機能追加） | 変わらない（振る舞い保持） |
| 承認ゲート | なし（完全自動） | あり（リファクタリング設計承認） |
| テストベースライン | なし | あり（リファクタリング前に確立） |

---

## ステップ1: 準備とコンテキスト設定

1. 現在のタスクコンテキストを確立する:

- リファクタリング内容: `[引数で与えられたリファクタリング内容]`
- 日時: `[現在の日時をYYYYMMDDHHmmss形式で取得]`
- ステアリングディレクトリパス: `.steering/[日時]-refactor-[内容を英語で短縮した名前]/`
  - 例: `/refactor 認証モジュールの責務分離` → `.steering/20260223120000-refactor-auth-module-separation/`
  - 例: `/refactor APIクライアントのエラーハンドリング統一` → `.steering/20260223120000-refactor-api-error-handling/`

2. 上記ステアリングディレクトリを作成する。
3. 以下の3つの空ファイルを作成する:

- `[ステアリングディレクトリパス]/requirements.md`
- `[ステアリングディレクトリパス]/design.md`
- `[ステアリングディレクトリパス]/tasklist.md`

## ステップ2: プロジェクト理解

1. `CLAUDE.md`を読み、プロジェクトの全体像を把握する。
2. `docs/`ディレクトリ内の永続ドキュメントを確認し、関連する設計思想やアーキテクチャを理解する。

## ステップ3: 現状分析（リファクタリング対象の調査）

1. Grepツールを使用し、リファクタリング対象に関連するキーワードでソースコード(`src/`)を検索する。

```bash
Grep('[リファクタリング対象に関連するキーワード]', 'src/')
```

2. リファクタリング対象のファイル・モジュールを特定し、**全てのソースコードを読み込む**。
3. 対象コードの以下を把握する:
   - 現在のコード構造と責務
   - 他モジュールとの依存関係（インポート元・インポート先）
   - 問題点（コードスメル、設計上の課題）
   - 影響範囲（変更が波及するファイル・モジュール）

4. **テストベースラインの確立**: プロジェクトの技術スタックに応じたテスト・ビルドコマンドを実行し、現在の状態を記録する。
   - `CLAUDE.md` および `docs/architecture.md` に記載されているテスト・ビルドコマンドを参照する
   - テスト結果を記録する（パス数、失敗数、スキップ数）
   - **テストが既に失敗している場合**: 失敗しているテスト名を記録し、リファクタリングによる回帰ではないことを識別可能にする

## ステップ4: リファクタリング設計（★ ユーザー承認必須）

1. ステップ3の分析結果に基づき、リファクタリングの設計を策定する。
2. `[ステアリングディレクトリパス]/design.md` に以下の内容を記述する:

```markdown
# リファクタリング設計書

## 対象コードの現状

### 問題点
- {コードスメルや設計上の問題を具体的に列挙}
- {ファイルパスと行番号を含めて具体的に記述}

### 影響範囲
- {リファクタリングにより変更されるファイル一覧}
- {間接的に影響を受けるモジュール}

## リファクタリング方針

### アプローチ
{どのようなリファクタリング手法を適用するか}
（例: Extract Method, Move Class, Replace Conditional with Polymorphism 等）

### Before/After概要
{主要な変更のBefore/Afterをコード構造レベルで示す}

### 振る舞いの保持
{リファクタリング前後で振る舞いが変わらないことをどう保証するか}

## リスクと対策
- {想定されるリスクと、その対策}
```

3. `[ステアリングディレクトリパス]/requirements.md` にリファクタリングの要求内容を記述する。

**ユーザー承認（必須）**:

```
「リファクタリング設計書（design.md）の作成が完了しました。
内容を確認してください。

[リファクタリングのサマリを簡潔に記述]

対象ファイル:
- [影響を受けるファイル一覧]

承認いただけたら実装に進みます。修正が必要な場合はお知らせください。」
```

**承認されるまで絶対に次に進まないこと。** ユーザーからの修正指示があれば設計書を修正し、再度承認を求める。

## ステップ5: 計画フェーズ（tasklist.mdの自動生成）

1. `Skill('steering')` を**計画モード**で実行し、`tasklist.md` の内容を生成する。その際、以下を計画の前提として使用すること:
   - **主要入力**: `[ステアリングディレクトリパス]/design.md`（承認済みのリファクタリング設計）
   - **変更対象**: ステップ3で特定したファイル・モジュール

2. **重要**: `tasklist.md` のフェーズ構成は以下のパターンに従うこと:

```markdown
## フェーズ1: 準備（テスト・型安全性の確認）
- [ ] リファクタリング対象の既存テストを確認する
- [ ] 不足しているテストがあれば追加する（振る舞い保持の検証用）

## フェーズ2: リファクタリング実装
- [ ] {具体的なリファクタリングタスク1}
- [ ] {具体的なリファクタリングタスク2}
- [ ] ...

## フェーズ3: 品質チェックと修正
- [ ] すべてのテストが通ることを確認
- [ ] リントエラーがないことを確認
- [ ] 型エラーがないことを確認
- [ ] ビルドが成功することを確認
```

3. **このステップが正常に完了したら、決して停止せず、ただちにステップ6に進むこと。**

## ステップ6: 実装ループ（tasklist.mdの完全消化）

**このステップは、`tasklist.md`の全タスクが `[x]` になるまで自動で繰り返されるループ処理です。**
**このステップが正常に完了したら、決して停止せず、ただちにステップ7に進むこと。**

**ループ開始:**

1. タスクリストの読み込み:

- `[ステアリングディレクトリパス]/tasklist.md`ファイルを読み込む。

2. 進捗の確認:

- ファイル内に未完了タスク (`[ ]`) が存在するか確認する。
- **もし未完了タスクが存在しない場合:** この実装ループは完了とみなし、ただちに**ステップ7**へ進む。
- **もし未完了タスクが存在する場合:** 次の処理（3. タスクの実行）に進む。

3. タスクの実行:

- `tasklist.md`の**先頭にある未完了タスク**を1つ特定する。
- そのタスクを完了させるために必要な実装作業を実行する。
- `Skill('steering')` を**実装モード**で使用する。
- `Skill('development-guidelines')` のコーディング規約を常に遵守する。
- **リファクタリング固有の注意**: 振る舞いを変えないこと。新機能の追加や仕様変更は行わない。

4. タスクリストの更新:

- 実行したタスクが完了したら、`Edit`ツールを使用して`tasklist.md`を更新し、該当タスクを `[ ]` から `[x]` に変更する。

5. ループ継続:

- **ステップ6の先頭 (1. タスクリストの読み込み) に戻り、処理を繰り返す。**

---

### ※ 実装ループ内の例外処理ルール ※

実装ループの実行中に以下の状況が発生した場合は、このルールに従って自律的に対処し、ループを継続すること。

- **ルールA: タスクが大きすぎる場合**
  - **対処法:** 現在のタスクをより小さな複数のサブタスクに分割する。`Edit`ツールを使い、元のタスクを削除し、その場所に新しいサブタスク（`[ ]`付き）を挿入する。その後、ループを継続する。

- **ルールB: 技術的理由でタスクが不要になった場合**
  - **条件:** 実装方針の変更、アーキテクチャの変更、依存関係の変更など、明確な技術的理由がある場合にのみ適用。
  - **対処法:** `Edit`ツールを使い、該当タスクを `[x] ~~タスク名~~ (理由: [具体的な技術的理由を簡潔に記述])` の形式で更新する。その後、ループを継続する。

- **❌ 絶対禁止の行為:**
  - 未完了タスクを「後でやる」「別タスクにする」などの理由で意図的にスキップすること。
  - 理由なく未完了タスクを放置してループを終了させること。
  - 実装ループ中にユーザーに判断を仰ぐこと（承認ゲートはステップ4のみ）。
  - 「ついでに」新機能を追加したり仕様を変更したりすること。

---

## ステップ7: 実装検証（サブエージェント並列起動）

1. `tasklist.md`の全タスクが完了したことを最終確認する。
2. `Task`ツールを使用し、以下の2つのサブエージェントを**並列で**起動して検証する。

**サブエージェント1: 実装品質検証**

- `subagent_type`: "implementation-validator"
- `description`: "Refactoring quality validation"
- `prompt`: "今回のリファクタリング `[リファクタリング内容]` に関する全ての変更点の品質を検証してください。対象ファイルは `[変更したファイルのパスリスト]` です。リファクタリングとして振る舞いが保持されていること、コーディング規約の遵守、エラーハンドリングの適切さ、既存パターンとの整合性を重点的に確認してください。"

**サブエージェント2: デザインパターン準拠チェック**

- `subagent_type`: "design-pattern-checker"
- `description`: "Design pattern compliance check"
- `prompt`: "今回のリファクタリング `[リファクタリング内容]` に関する全ての変更点がプロジェクトのデザインパターンに準拠しているか検証してください。対象ファイルは `[変更したファイルのパスリスト]` です。フィーチャーベースアーキテクチャ、データフェッチング層パターン、状態管理パターン、コンポーネント構成ルール、インポートパス・モジュール境界ルール、命名規則を重点的に確認してください。"

3. 両方のサブエージェントの結果を確認する。
4. いずれかのサブエージェントで**[必須]レベルの問題**が検出された場合は、問題を修正してから再度検証を実行する。

**このステップが正常に完了したら、決して停止せず、ただちにステップ8に進むこと。**

## ステップ8: 回帰テストの実行

1. ステップ3で記録したテストベースラインと同じテスト・ビルドコマンドを実行する。

```bash
# CLAUDE.md および docs/architecture.md に記載されているコマンドを参照
# 例:
Bash('npm test')
Bash('npm run lint')
Bash('npm run typecheck')
```

2. テスト結果をステップ3のベースラインと比較する:
   - **新たに失敗したテストがある場合**: リファクタリングによる回帰のため、問題を修正して再度テストする。
   - **ベースラインで既に失敗していたテストのみ失敗**: リファクタリングとは無関係のため、問題なし。
   - **全テストパス**: 問題なし。

3. 全てのチェックがパスするまで修正とテストを繰り返す。

**このステップが正常に完了したら、決して停止せず、ただちにステップ9に進むこと。**

## ステップ9: 振り返りとドキュメント更新

1. `Skill('steering')` を**振り返りモード**で実行し、`tasklist.md` に申し送り事項を記載する。

- 実装完了日
- 計画と実績の差分
- 適用したリファクタリング手法とその効果
- コード品質の改善点（Before/Afterの定量的比較があれば記載）
- 学んだこと
- 次回への改善提案

2. 今回のリファクタリングがプロジェクトのアーキテクチャや開発ガイドラインに影響を与えるか判断する。

3. 影響がある場合、`docs/` 内の関連する永続ドキュメントを `Edit` ツールで更新する:
   - コードの構造が大きく変わった → `docs/architecture.md` や `docs/repository-structure.md` に影響
   - 新しいパターンが導入された → `docs/development-guidelines.md` に影響
   - 該当しない場合は更新不要

## 完了条件

このワークフローは、以下の全ての条件を満たした時点で完了となる。

- ステップ4: リファクタリング設計書（design.md）がユーザーに承認されている。
- ステップ6: `tasklist.md`の全てのタスクが完了状態（`[x]`または正当な理由でスキップ）になっている。
- ステップ7: `implementation-validator`および`design-pattern-checker`サブエージェントの検証をパスする（[必須]レベルの問題がないこと）。
- ステップ8: リファクタリングによる回帰テストの失敗がない（ステップ3のベースラインと比較して新たな失敗がないこと）。
- ステップ9: `tasklist.md`に申し送り事項が記載されている。

この完了条件を満たすまで、自律的に思考し、問題解決を行い、作業を継続すること。
