# 技術検証 PoC（フェーズごと承認型）

**重要:** このワークフローは `/add-feature` とは異なり、**フェーズごとにユーザーの承認を得ながら進行する対話型**です。各フェーズの承認ポイント（★マーク）で**必ず停止**し、ユーザーの承認を得てから次に進んでください。承認を得ずに次のフェーズに進むことは**絶対に禁止**です。

**引数:** 技術名（例: `/poc TanStack Query`、`/poc Hono`、`/poc Drizzle ORM`）

**`/add-feature` との使い分け:**

| 観点 | `/poc` | `/add-feature` |
|------|--------|----------------|
| 対話モデル | フェーズごとに承認 | 完全自動 |
| 出力先 | `poc/[技術名]/` | `src/` |
| コードの性質 | 使い捨て検証コード | 本番コード |
| 主な成果物 | 設計パターン文書 + docs/更新 | 動作する機能 |
| 焦点 | パターン発見 | 機能実装 |

---

## ステップ1: 準備とコンテキスト設定

1. 現在のタスクコンテキストを確立する:

- 技術名: `[引数で与えられた技術名]`
- 日時: `[現在の日時をYYYYMMDDHHmmss形式で取得]`
- PoCディレクトリパス: `poc/[技術名を英語で短縮しケバブケースにした名前]/`
  - 例: `/poc TanStack Query` → `poc/tanstack-query/`
  - 例: `/poc Drizzle ORM` → `poc/drizzle-orm/`
- ステアリングディレクトリパス: `.steering/[日時]-poc-[技術名を英語で短縮した名前]/`
  - 例: `.steering/20260222143052-poc-tanstack-query/`

2. 以下のディレクトリとファイルを作成する:

- PoCディレクトリ: `poc/[技術名]/`
- PoCソースディレクトリ: `poc/[技術名]/src/`
- ステアリングディレクトリ: `.steering/[日時]-poc-[技術名]/`
- 空ファイル:
  - `[ステアリングディレクトリパス]/requirements.md`
  - `[ステアリングディレクトリパス]/design.md`
  - `[ステアリングディレクトリパス]/tasklist.md`

## ステップ2: プロジェクト理解

1. `CLAUDE.md` を読み、プロジェクトの全体像を把握する。
2. `docs/` ディレクトリ内の永続ドキュメントを確認し、関連するアーキテクチャや技術スタックを理解する:
   - `docs/architecture.md`（存在する場合）
   - `docs/development-guidelines.md`（存在する場合）
   - `docs/repository-structure.md`（存在する場合）
   - `docs/product-requirements.md`（存在する場合）

## ステップ3: 計画フェーズ（★ ユーザー承認必須）

1. `Skill('steering')` を**計画モード**で実行し、ステップ1で作成した3つのファイル（`requirements.md`, `design.md`, `tasklist.md`）の内容を生成する。

2. **重要**: `tasklist.md` は以下の4フェーズ構成とすること。各フェーズの最後に「★ ユーザー承認を得る」タスクを含めること:

```markdown
## フェーズ1: 技術リサーチとPoC構築
- [ ] 公式ドキュメント調査
- [ ] overview.md の作成
- [ ] 最小検証コードの生成
- [ ] ★ ユーザー承認を得る

## フェーズ2: 技術検証
- [ ] 検証コードの実行と結果記録
- [ ] 評価基準に対する合否判定
- [ ] 有名OSSでの採用事例調査（最低2-3件）
- [ ] findings.md の作成
- [ ] ★ ユーザー承認を得る（採用/不採用判断）

## フェーズ3: 設計パターン探索
- [ ] 既存アーキテクチャ分析
- [ ] 有名OSSのデザインパターン深掘り調査と適用検討
- [ ] スケール時の問題洗い出し
- [ ] 候補パターンの提案（1-3個、OSS事例を踏まえて）
- [ ] design-pattern.md の作成
- [ ] ★ ユーザー承認を得る（パターン選定）

## フェーズ4: 文書化と docs/ 統合
- [ ] docs/architecture.md の更新
- [ ] ★ ユーザー承認を得る
- [ ] docs/development-guidelines.md の更新
- [ ] ★ ユーザー承認を得る
- [ ] その他 docs/ ファイルの更新（必要に応じて）
- [ ] ★ ユーザー承認を得る（該当する場合）
```

3. `requirements.md` には検証目的、評価基準（必須/加点）、検証スコープを含めること。

**ユーザー承認（必須）**:

```
「PoC計画（ステアリングファイル）の作成が完了しました。

対象技術: [技術名]
検証目的: [目的を1行で]
必須基準: [基準の数]項目
検証スコープ: [スコープを1行で]

requirements.md, design.md, tasklist.md を確認してください。
承認いただけたらフェーズ1（技術リサーチ）に進みます。修正が必要な場合はお知らせください。」
```

**承認されるまで絶対に次に進まないこと。** ユーザーからの修正指示があればステアリングファイルを修正し、再度承認を求める。

## ステップ4: フェーズ1 - 技術リサーチとPoC構築（★ ユーザー承認必須）

1. `Skill('poc-workflow')` をロードし、ガイドのフェーズ1に従って実行する。
2. `Skill('steering')` を**実装モード**で使用し、tasklist.mdの進捗を管理する。

**実行内容**:
- 公式ドキュメント調査（WebSearchを使用）
- `poc/[技術名]/overview.md` の作成（テンプレート使用）
- `poc/[技術名]/src/` に最小限の検証コード生成
- 必要な依存関係のインストール

**ユーザー承認（必須）**:

ガイドのフェーズ1の承認ポイントに従い、ユーザーの承認を得る。

**承認されるまで絶対に次に進まないこと。**

## ステップ5: フェーズ2 - 技術検証（★ ユーザー承認必須・採用/不採用判断）

1. `Skill('poc-workflow')` のガイドのフェーズ2に従って実行する。
2. `Skill('steering')` を**実装モード**で使用し、tasklist.mdの進捗を管理する。

**実行内容**:
- 検証コードの実行と結果記録
- 評価基準に対する合否判定
- `poc/[技術名]/findings.md` の作成（テンプレート使用）

**ユーザー承認（必須・採用/不採用判断）**:

ガイドのフェーズ2の承認ポイントに従い、ユーザーの承認を得る。

**承認されるまで絶対に次に進まないこと。**

**不採用の場合**: ユーザーが不採用を承認した場合、以下を実行してステップ8（振り返り）に直行する:

1. tasklist.md のフェーズ3・4のタスクを以下の形式で更新:
   ```markdown
   - [x] ~~タスク名~~（理由: フェーズ2で不採用判断のためスキップ）
   ```
2. ステップ6・7をスキップし、ステップ8に進む。

## ステップ6: フェーズ3 - 設計パターン探索（★ ユーザー承認必須・パターン選定）

1. `Skill('poc-workflow')` のガイドのフェーズ3に従って実行する。
2. `Skill('steering')` を**実装モード**で使用し、tasklist.mdの進捗を管理する。

**実行内容**:
- 既存アーキテクチャ分析
- スケール時の問題洗い出し
- 候補パターンの提案（1〜3個）
- `poc/[技術名]/design-pattern.md` の作成（テンプレート使用）

**ユーザー承認（必須・パターン選定）**:

ガイドのフェーズ3の承認ポイントに従い、ユーザーの承認を得る。
ユーザーが選定したパターンを記録する。

**承認されるまで絶対に次に進まないこと。**

## ステップ7: フェーズ4 - 文書化と docs/ 統合（★ 1ファイルずつユーザー承認）

1. `Skill('poc-workflow')` のガイドのフェーズ4に従って実行する。
2. `Skill('steering')` を**実装モード**で使用し、tasklist.mdの進捗を管理する。

**実行内容**:

各ファイルを**1ファイルずつ**更新し、**各ファイルごとにユーザーの承認を得る**:

1. `docs/architecture.md` の更新（または新規作成）
   - 採用した技術のセクションを追加
   - 選定した設計パターンを記載
   - **ファイルが存在しない場合**: `Skill('architecture-design')` のテンプレートで新規作成
   - ★ ユーザー承認を得る

2. `docs/development-guidelines.md` の更新（または新規作成）
   - 採用した技術に関するコーディング規約を追加
   - アンチパターンを記載
   - **ファイルが存在しない場合**: `Skill('development-guidelines')` のテンプレートで新規作成
   - ★ ユーザー承認を得る

3. その他の `docs/` ファイル更新（必要に応じて）
   - `docs/glossary.md`: 新しい技術用語
   - `docs/repository-structure.md`: ディレクトリ構造変更
   - ★ ユーザー承認を得る（更新がある場合）

**1ファイルずつ承認されるまで絶対に次のファイルに進まないこと。**

## ステップ8: 振り返り

1. `Skill('steering')` を**振り返りモード**で実行し、`tasklist.md` に申し送り事項を記載する:

- 実装完了日
- 計画と実績の差分
- 学んだこと（技術的な知見、プロジェクトへの影響）
- 次回への改善提案
- **不採用の場合**: 不採用の理由、代替案の検討結果

2. PoC全体のサマリをユーザーに報告する:

```
「PoC（[技術名]）が完了しました。

結果: [採用/不採用]
[採用の場合] 選定パターン: [パターン名]
[採用の場合] 更新したドキュメント: [ファイル名リスト]
[不採用の場合] 不採用理由: [理由を1行で]

振り返りを tasklist.md に記録しました。」
```

## 完了条件

このワークフローは、以下の条件を満たした時点で完了となる。

**採用の場合:**
- ステップ3: ステアリングファイルがユーザーに承認されている
- ステップ4: overview.md と検証コードがユーザーに承認されている
- ステップ5: findings.md がユーザーに承認され、採用が判断されている
- ステップ6: design-pattern.md がユーザーに承認され、パターンが選定されている
- ステップ7: docs/ の全ファイル更新がユーザーに承認されている
- ステップ8: tasklist.md に振り返りが記載されている

**不採用の場合:**
- ステップ3: ステアリングファイルがユーザーに承認されている
- ステップ4: overview.md と検証コードがユーザーに承認されている
- ステップ5: findings.md がユーザーに承認され、不採用が判断されている
- ステップ8: tasklist.md に振り返り（不採用理由含む）が記載されている

### ※ 例外処理ルール ※

- **ルールA: タスクが大きすぎる場合**
  - **対処法:** 現在のタスクをより小さな複数のサブタスクに分割する。`Edit`ツールを使い、元のタスクを削除し、その場所に新しいサブタスク（`[ ]`付き）を挿入する。その後、作業を継続する。

- **ルールB: 技術的理由でタスクが不要になった場合**
  - **条件:** 明確な技術的理由がある場合にのみ適用。
  - **対処法:** `Edit`ツールを使い、該当タスクを `[x] ~~タスク名~~ (理由: [具体的な技術的理由を簡潔に記述])` の形式で更新する。

- **❌ 絶対禁止の行為:**
  - 承認ポイント（★マーク）でユーザーの承認を得ずに次に進むこと
  - 未完了タスクを理由なく放置すること
  - フェーズの順序を入れ替えること（不採用による早期終了を除く）
