# issue作成（WF4 オーケストレーター）

ユーザーと対話的にissue種別を判定し、種別に応じた仕様ドキュメントの作成からGitHub issue登録までを順次実行する。

## 起動方法

```
/create-issue
```

## 全体フロー

```
Phase 1: 種別判定・要件整理（オーケストレーターが直接対話）
  ↓
Phase 2a（機能の追加・変更）:
  機能設計（opt）→ 画面仕様（opt）→ PRDスコープ検証
  ※ 各ドキュメント生成後にレビュー → ユーザー承認
  ↓
Phase 2b（不具合修正 / リファクタリング / パフォーマンス / 依存更新）:
  issue仕様作成 → レビュー → ユーザー承認
  ↓
Phase 3: issue登録 → ユーザー確認
```

## Phase 1: 種別判定・要件整理（オーケストレーターが直接対話）

ユーザーと対話して以下を確認する:

| 確認事項 | 内容 |
|---|---|
| issue種別 | 機能の追加・変更 / 不具合修正 / リファクタリング / パフォーマンス改善 / 依存更新 |
| 要件 | 何を変更・修正・改善するのか |
| 背景 | なぜこの変更が必要か |
| 影響範囲 | どの機能・画面に影響するか |

### 種別判定の基準

| 種別 | 判定基準 |
|---|---|
| 機能の追加・変更 | UI変更、操作挙動変更、UI+挙動の追加 |
| 不具合修正 | 既存機能が仕様通りに動作しない |
| リファクタリング | 機能変更なしのコード品質改善 |
| パフォーマンス改善 | 計測結果に基づく最適化 |
| 依存更新 | ライブラリバージョンアップ、CI/CD変更 |

種別が確定したら、関連する `docs/` 配下のドキュメントを読み込み、Phase 2 に進む。

## Phase 2a: 変更仕様作成（機能の追加・変更の場合）

### Step 1: 機能設計の変更（機能設計の変更がある場合のみ）

1. `functional-designer` エージェントを起動する
   - 動作モード: 変更仕様
   - 入力: 要件 + `docs/functional-design.md`
   - 出力: 機能変更仕様
2. `doc-reviewer` エージェントを起動する
   - 状況依存スキル: `functional-design`
   - 入力: 機能変更仕様
3. 機能変更仕様 + レビュー結果をユーザーに提示する
4. ユーザーの承認を待つ

### Step 2: 画面仕様の変更（画面仕様の変更がある場合のみ）

1. `screen-spec-writer` エージェントを起動する
   - 動作モード: 変更仕様
   - 入力: 要件 + `docs/screen-specification/`
   - 出力: 画面変更仕様
2. `doc-reviewer` エージェントを起動する
   - 状況依存スキル: `screen-specification`
   - 入力: 画面変更仕様
3. 画面変更仕様 + レビュー結果をユーザーに提示する
4. ユーザーの承認を待つ

### Step 3: PRDスコープ検証

1. `prd-scope-checker` エージェントを起動する
   - 入力: 変更仕様（Step 1, 2 の出力）+ `docs/product-requirements.md`
   - 出力: スコープ判定結果
2. スコープ判定結果をユーザーに報告する
   - **範囲内**: Phase 3 に進む
   - **逸脱あり**: PRD更新の必要性をユーザーに報告し、判断を委ねる

## Phase 2b: issue仕様作成（不具合修正 / リファクタリング / パフォーマンス / 依存更新の場合）

1. `issue-spec-writer` エージェントを起動する
   - 入力: 種別 + 要件 + 関連 `docs/`
   - 出力: 種別固有ドキュメント
2. `doc-reviewer` エージェントを起動する
   - 状況依存スキル: `issue-specification`
   - 入力: 種別固有ドキュメント
3. 種別固有ドキュメント + レビュー結果をユーザーに提示する
4. ユーザーの承認を待つ

## Phase 3: issue登録

1. `issue-decomposer` エージェントを起動する
   - 動作モード: 登録のみ（WF4モード）
   - 入力: 全仕様ドキュメント（Phase 2a または 2b の成果物）
   - 出力: GitHub issue + `.issue/{N}/`
2. 作成されたissueをユーザーに提示する
3. ユーザーの承認を待つ

issue登録はドキュメントレビューの対象外（構造的な処理のため）。

## 完了条件

- [ ] issue種別が判定済み
- [ ] 種別に応じた仕様ドキュメントが承認済み
- [ ] PRDスコープ検証が完了（機能の追加・変更の場合のみ）
- [ ] GitHub issueが作成済み
- [ ] `.issue/{N}/` に詳細ドキュメントが配置済み

## エラー時の動作

| ケース | 動作 |
|---|---|
| `docs/product-requirements.md` が存在しない | PRDスコープ検証（Step 3）をスキップし、ユーザーに警告する |
| `docs/functional-design.md` が存在しない | 機能変更仕様の作成をスキップし、ユーザーに警告する |
| `docs/screen-specification/` が存在しない | 画面変更仕様の作成をスキップし、ユーザーに警告する |
| サブエージェントがエラーで終了 | エラー内容をユーザーに報告し、再実行するか確認する |
| ユーザーがフェーズをスキップしたい | 該当フェーズをスキップし、次のフェーズに進む。依存するフェーズがある場合は警告する |
