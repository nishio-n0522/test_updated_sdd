# 実装計画スキル

autopilot管理エージェントがissueの実装計画を策定する際に参照するドメイン知識。テンプレートと品質基準を定義する。

## スコープ

- **提供するもの**: 実装計画ドキュメントのテンプレート、品質基準、ドキュメント構造定義
- **提供しないもの**: 計画策定の手順（Agent が定義）、プロジェクト固有の技術判断

## ドキュメント構造

`.issue/{N}/` ディレクトリに以下のドキュメントを配置する:

| ファイル | 目的 | 作成タイミング |
|---|---|---|
| `requirements.md` | 要求の構造化 | Phase A（計画） |
| `design.md` | 実装設計 | Phase A（計画） |
| `test-spec.md` | テスト仕様 | Phase A（計画） |
| `tasklist.md` | タスク管理・進捗追跡 | Phase A（計画） |
| `reviews/` | レビュー結果の格納先 | Phase C（レビュー） |
| `fixes/` | 修正指示書の格納先 | Phase C（レビュー） |

## テンプレート

### requirements.md

```markdown
# 要求内容

## 概要

{機能の概要を1-2文で記述}

## 背景

{なぜこの機能が必要か、どんな問題を解決するか}

## 実装対象の機能

### 1. {機能名1}
- {具体的な機能説明}
- {ユーザーができるようになること}

## 受け入れ条件

### {機能名1}
- [ ] {検証可能な条件1}
- [ ] {検証可能な条件2}

## スコープ外

- {実装しない機能1}

## 参照ドキュメント

- `docs/product-requirements.md`
- `docs/functional-design.md`
- `docs/architecture.md`
```

### design.md

```markdown
# 設計書

## アーキテクチャ概要

{採用するアーキテクチャパターンを記述}

## コンポーネント設計

### 1. {コンポーネント名}

**責務**:
- {責務1}

**実装の要点**:
- {実装時の注意点}

## データフロー

### {ユースケース名}

1. {ステップ1}
2. {ステップ2}

## エラーハンドリング戦略

{エラーをどのように処理するか}

## ディレクトリ構造

{追加・変更されるファイル構造}

## 実装の順序

1. {実装ステップ1}
2. {実装ステップ2}
```

### test-spec.md

```markdown
# テスト仕様書

## 概要

{テストの目的と対象範囲}

## テスト対象

- {対象モジュール/コンポーネント}

## テスト環境

- テストフレームワーク: {Jest / Vitest / pytest 等}
- テスト実行コマンド: {npm test / pytest 等}
- モック戦略: {何をモックするか}

## ユニットテスト

### {テスト対象}

| # | テストケース | 入力/条件 | 期待結果 | 受け入れ条件 |
|---|---|---|---|---|
| 1 | {ケース名} | {入力値} | {期待出力} | requirements.md #{番号} |

## 統合テスト

### {テストシナリオ}

- 前提条件: {テスト開始時の状態}
- 手順: {ステップ}
- 期待結果: {最終的な期待状態}

## エッジケース

- {考慮すべきエッジケース}

## テストファイル配置

- {テストファイルのパスと対応する実装ファイル}
```

### tasklist.md

```markdown
# タスクリスト

## タスク完全完了の原則

- 全てのタスクを `[x]` にすること
- 未完了タスク（`[ ]`）を残したまま作業を終了しない
- タスクスキップは技術的理由がある場合のみ許可

スキップ時の記法:
- [x] ~~タスク名~~（実装方針変更により不要: 具体的な技術的理由）

## フェーズ1: 実装（実装エージェント担当）

- [ ] {タスク1}
  - [ ] {サブタスク1-1}

## フェーズ2: テスト（テスト作成エージェント担当）

- [ ] {タスク1}
  - [ ] {サブタスク1-1}

---

## 実装後の振り返り

### 実装完了日
{YYYY-MM-DD}

### 計画と実績の差分
- {計画と異なった点}

### 学んだこと
- {技術的な学び}

### 次回への改善提案
- {改善点}
```

### fix-order.md（修正指示書）

```markdown
# 修正指示書（ラウンド{N}）

## 参照したレビュー結果

- reviews/review-coding-r{N}.md: PASS / FAIL（MUST: {件数}）
- reviews/review-architecture-r{N}.md: PASS / FAIL（MUST: {件数}）
- reviews/review-i18n-r{N}.md: PASS / FAIL（MUST: {件数}）
- reviews/test-results-r{N}.md: PASS / FAIL

## 修正方針の概要

{レビュー指摘を踏まえた修正の全体方針}

## 設計変更（ある場合）

{design.mdの変更点。なければ「設計変更なし」}

## 実装エージェントへの修正指示

### 修正1
- 対象ファイル: {パス}
- 指摘元: {レビュー種別}
- 指摘内容: {問題の要約}
- 修正内容: {具体的に何をどう修正するか}

## テスト作成エージェントへの修正指示

### 修正1
- 対象ファイル: {パス}
- 指摘元: {レビュー種別}
- 指摘内容: {問題の要約}
- 修正内容: {具体的に何をどう修正するか}
```

## 品質基準

### requirements.md

- [ ] 概要が1-2文で簡潔に記述されているか
- [ ] 受け入れ条件が検証可能な形式（チェックリスト）か
- [ ] スコープ外が明記されているか
- [ ] 参照ドキュメントへのリンクがあるか

### design.md

- [ ] コンポーネントの責務が明確に分離されているか
- [ ] データフローが順序付きで記述されているか
- [ ] 実装の順序が明記されているか
- [ ] ディレクトリ構造が示されているか
- [ ] エラーハンドリング戦略が定義されているか

### test-spec.md

- [ ] テスト環境（フレームワーク、実行コマンド）が明記されているか
- [ ] テストケースが受け入れ条件と紐付いているか
- [ ] エッジケースが列挙されているか
- [ ] テストファイルの配置先が指定されているか

### tasklist.md

- [ ] フェーズ1（実装）とフェーズ2（テスト）が分離されているか
- [ ] タスクが具体的で実行可能な粒度か
- [ ] 「将来やるかもしれないタスク」が含まれていないか
- [ ] 振り返りセクションが用意されているか

### fix-order.md

- [ ] 参照したレビュー結果の判定が記載されているか
- [ ] 実装エージェントとテスト作成エージェントへの指示が分離されているか
- [ ] 各修正指示に対象ファイル・指摘元・修正内容が含まれるか
