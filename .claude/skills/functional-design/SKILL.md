# 機能設計書作成スキル

機能設計書を作成・レビューする際に参照するドメイン知識。テンプレートと品質基準を定義する。

## スコープ

- **提供するもの**: 機能設計書のテンプレート、品質基準、記述ガイドライン
- **提供しないもの**: 作成手順（Agent が定義）、プロジェクト固有の内容

## 入出力

| 項目 | WF2: 新規作成 | WF4: 既存変更 |
|---|---|---|
| 入力 | PRD + アイデアメモ | 要件 + 既存 `docs/functional-design.md` |
| 出力 | `docs/functional-design.md` | 機能変更仕様 |

> WF2（新規作成）: プロジェクト立ち上げ時にドキュメントを新規作成するワークフロー
> WF4（既存変更）: 既存ドキュメントに対して変更・追記するワークフロー

## 責務の定義

機能設計書は「PRDの要求を具体的な機能に分解・構造化する」ドキュメントである。

**答える問い**: 要求をどんな機能・画面に分割するか？

### 記載する内容

- 機能一覧とその関係（PRDの要求をどう機能分解したか）
- 各機能の詳細（目的、サブ機能、入出力、ビジネスルール）
- 画面構成（どんな画面が必要か、画面遷移の概要）
- ユーザーフロー（主要な機能の操作フロー、概要レベル）
- ドメインモデル（概念レベル: エンティティの存在と関係性のみ）
- 機能カタログ（該当する場合: ドメイン固有の機能バリエーション一覧。例: ノード種別一覧、レポート種類一覧）
- PRD機能要件との対応確認

### 記載しない内容

- 技術スタック・システム構成図（architecture.md の責務）
- コンポーネント設計・実装インターフェース（architecture.md の責務）
- データモデルの詳細: フィールド定義・型・制約・ER図の物理設計（`data-model-design` スキルの責務）
- 各画面の詳細なUI仕様・操作挙動（screen-specification の責務）
- エラーハンドリングの詳細: 具体的なエラーメッセージ・UI表示方法（screen-specification の責務）
- パフォーマンス最適化・セキュリティ考慮事項（architecture.md の責務）
- テスト戦略・テストケース（test-specification の責務）
- ファイル構造・データ保存形式の詳細（architecture.md or repository-structure の責務）

## 他ドキュメントとの関係

| ドキュメント | 関係 |
|---|---|
| PRD | 上流。PRDの要求を機能に分解したものが本ドキュメント |
| 画面仕様書 | 下流。本ドキュメントの画面構成を詳細化したもの |
| データモデル設計書 | 下流。本ドキュメントのドメインモデルを詳細化したもの |
| アーキテクチャ設計書 | 並行。技術的な実現方法は architecture.md が担う |

## 既存ドキュメントの優先順位

| 優先度 | 対象 | 説明 |
|---|---|---|
| 1（最優先） | `docs/functional-design.md` | プロジェクト固有の設計。このスキルのガイドより優先 |
| 2（参考） | このスキル | 汎用テンプレート。既存設計書がない場合に使用 |

## 品質基準

### 必須チェック項目

- [ ] 機能一覧がPRDの全機能要件をカバーしているか
- [ ] 各機能の目的・入出力・ビジネスルールが明確か
- [ ] 機能間の関係が図示されているか
- [ ] 画面一覧と画面遷移の概要が示されているか（UIありの場合）
- [ ] 主要ユーザーフローがシーケンス図で示されているか
- [ ] ドメインモデルの概念（エンティティと関係性）が示されているか
- [ ] PRDの機能要件との対応が確認できるか
- [ ] 機能カタログが必要な場合、種別・分類の一覧が網羅されているか
- [ ] 責務外の内容（技術スタック、実装詳細、テスト戦略等）が混入していないか

### 記述ガイドライン

- 機能の「何をするか」に集中し、「どう実現するか」は書かない
- UIは概要レベルで十分。「だいたいこんなアプリ」が伝わればよい
- ドメインモデルは概念レベル（エンティティ名と関係性）にとどめる。フィールド定義は書かない
  - OK: 「Project は複数の Image を持つ」（概念レベル）
  - NG: 「Project は name: string, createdAt: Date を持つ」（フィールド定義 → data-model の責務）
- ユーザーフローは主要フローをシーケンス図で表現する
- 機能カタログは、ドメイン固有のバリエーションが多い場合に作成する。個々の詳細仕様ではなく種別・分類の一覧にとどめる
- 将来フェーズへの備考は、現フェーズの設計判断に影響する事項に限定する（将来の詳細仕様は書かない）
- 技術用語よりドメイン用語を優先する
