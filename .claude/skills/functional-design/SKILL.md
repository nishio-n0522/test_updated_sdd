# 機能設計書作成スキル

機能設計書を作成・レビューする際に参照するドメイン知識。テンプレートと品質基準を定義する。

## スコープ

- **提供するもの**: 機能設計書のテンプレート、品質基準、記述ガイドライン
- **提供しないもの**: 作成手順（Agent が定義）、プロジェクト固有の内容

## 入出力

| 項目 | WF2（新規） | WF4（変更） |
|---|---|---|
| 入力 | PRD + アイデアメモ | 要件 + 既存 `docs/functional-design.md` |
| 出力 | `docs/functional-design.md` | 機能変更仕様 |

## 画面仕様書（screen-specification）との関係

| 項目 | 本スキル（機能設計書） | screen-specification（画面仕様書） |
|---|---|---|
| 責務 | What の構造化 | How / UI詳細 |
| UI粒度 | 概要レベル（画面一覧、大まかなレイアウト、画面遷移） | 詳細レベル（全UI要素定義、状態別ワイヤーフレーム、操作挙動） |
| 主な内容 | データモデル、コンポーネント設計、ユースケース + 概要UI | 画面ごとの詳細仕様のみ |

## 既存ドキュメントの優先順位

| 優先度 | 対象 | 説明 |
|---|---|---|
| 1（最優先） | `docs/functional-design.md` | プロジェクト固有の設計。このスキルのガイドより優先 |
| 2（参考） | このスキル | 汎用テンプレート。既存設計書がない場合に使用 |

## テンプレート

機能設計書を作成する際は次のテンプレートを使用する: ./template.md

## 品質基準

### 必須チェック項目

- [ ] システム構成図があるか
- [ ] 技術スタックが選定理由付きで定義されているか
- [ ] データモデルがER図付きで定義されているか
- [ ] 主要ユースケースがシーケンス図で示されているか
- [ ] コンポーネントの責務と依存関係が明確か
- [ ] エラーハンドリングが考慮されているか
- [ ] 画面遷移の全体像が示されているか（UIありの場合）
- [ ] PRDの機能要件との対応が確認できるか

### 記述ガイドライン

- UIは概要レベルで十分。「だいたいこんなアプリ」が伝わればよい
- 技術スタックの選定には必ず理由を添える
- データモデルはTypeScriptインターフェースとER図の両方で示す
- ユースケースは主要フローをシーケンス図で表現する
