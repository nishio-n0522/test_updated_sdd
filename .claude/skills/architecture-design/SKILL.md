# アーキテクチャ設計スキル

アーキテクチャ設計書を作成・レビューする際に参照するドメイン知識。テンプレートと品質基準を定義する。

## スコープ

- **提供するもの**: アーキテクチャ設計書のテンプレート、品質基準
- **提供しないもの**: 作成手順（Agent が定義）、プロジェクト固有の内容

## 入出力

| 項目 | 内容 |
|---|---|
| 入力 | PRD + 機能設計書 + 画面仕様書 |
| 出力 | `docs/architecture/` |

## ファイル構成

```
docs/architecture/
├── index.md              # 全体概要（システム構成図・スタック間連携・共通方針）
└── {stack}.md            # スタック固有の詳細（1スタック = 1ファイル）
```

### 概要（index.md）の役割

- システム全体のアーキテクチャ概要・構成図を記載
- 技術スタック一覧と各スタックの役割を記載
- スタック間の連携方法（API、メッセージング等）を記載
- Agentはindex.mdを参照し、タスクに該当するスタックのみを読み込む

### 個別スタックの命名規則

```
{stack}.md
```

`stack` はスタックを表す kebab-case の名前（例: `frontend.md`, `backend.md`, `infrastructure.md`）

## 読み込み戦略

アーキテクチャ設計書は**選択的読み込み**で使用する。

```
実装タスク開始
  └─ architecture/index.md を参照
       └─ タスクに該当するスタックのみ読み込み
```

## 既存ドキュメントの優先順位

| 優先度 | 対象 | 説明 |
|---|---|---|
| 1（最優先） | `docs/architecture/` | プロジェクト固有の技術選定と設計。このスキルのガイドより優先 |
| 2（参考） | このスキル | 汎用テンプレート。既存設計書がない場合に使用 |

## テンプレート

アーキテクチャ設計書を作成する際は次のテンプレートを使用する: ./template.md

テンプレートには index.md 用と個別スタック用の2種類が含まれる。

## 品質基準

### 概要（index.md）のチェック項目

- [ ] システム全体の構成図が含まれているか
- [ ] 全スタックが一覧に含まれているか
- [ ] 各スタックの役割と責務が明確か
- [ ] スタック間の連携方法が定義されているか
- [ ] 共通の技術方針（セキュリティ、パフォーマンス等）が記載されているか

### 個別スタックのチェック項目

- [ ] 技術スタックが選定理由付きで定義されているか
- [ ] テクノロジースタック表のバージョンが最新安定版であるか（WebSearch で確認済みであること。AIモデルの学習データに基づくバージョンは古い可能性がある）
- [ ] レイヤードアーキテクチャが図解付きで明確か
- [ ] 各レイヤーの責務・許可/禁止操作が定義されているか
- [ ] データ永続化戦略が定義されているか
- [ ] パフォーマンス要件が測定可能な数値で定義されているか
- [ ] セキュリティ考慮事項が記載されているか
- [ ] スケーラビリティが考慮されているか
- [ ] テスト戦略（フレームワーク、カバレッジ目標）が定義されているか
- [ ] 技術的制約（環境要件、リソース上限）が明記されているか
- [ ] 依存関係管理方針が定義されているか

### 整合性チェック（全ファイル横断・必須）

以下はドキュメント全体の品質に直結する。生成後に必ず確認すること。

#### 名称一貫性

- [ ] テクノロジースタック表に記載した技術名（公式の現行名称）が、アーキテクチャ図・レイヤー説明・依存関係表など、ドキュメント全体で統一されているか
- [ ] 旧名称・通称は使用していないか（例: ✕ React Query → ○ TanStack Query）
- [ ] npmパッケージ名とライブラリの表示名が混在していないか（例: `reactflow` vs `React Flow`。テクノロジースタック表ではライブラリ名、依存関係管理表ではパッケージ名を使用する）

#### 依存関係の完全性

- [ ] 本文中（レイヤー説明・セキュリティ・入力検証等）で言及されるすべてのライブラリが、テクノロジースタック表に記載されているか
- [ ] テクノロジースタック表のすべてのライブラリが、依存関係管理表にも記載されているか（1:1対応）
- [ ] バージョン表記がテクノロジースタック表と依存関係管理表で一致しているか

#### 状態管理の網羅性

- [ ] 状態管理の責務を持つライブラリが複数ある場合（例: Zustand=クライアント状態、TanStack Query=サーバー/非同期状態）、各ライブラリの管理対象が明確に区別されているか
- [ ] レイヤードアーキテクチャ図のレイヤー名と、本文のレイヤー説明が完全に一致しているか
- [ ] レイヤー図内で使用する技術名が、テクノロジースタック表の名称と一致しているか

#### データ永続化のスタック間整合性

- [ ] 各スタックのデータ永続化表の「保存先」が、infrastructureスタックのストレージ定義と矛盾しないか
- [ ] 同一データに複数のストレージが使用される場合（例: JSONファイル + SQLiteインデックス）、ソースオブトゥルースと同期方法がinfrastructureスタックに明示されているか
- [ ] 各スタックは自身の責務範囲のデータのみ記載し、他スタックのストレージ詳細には「infrastructureスタック参照」と委譲しているか
