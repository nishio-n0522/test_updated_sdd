# リポジトリ構造定義スキル

リポジトリ構造定義書を作成・レビューする際のドメイン知識。テンプレートと品質基準を定義する。

## スコープ

- **提供するもの**: リポジトリ構造のテンプレート、品質基準、依存ルールの定義方法
- **提供しないもの**: 作成手順（Agent が定義）、プロジェクト固有のディレクトリ構造

## 記述レベル

リポジトリ構造定義書は**配置ルール**のレベルで記述する。

| 書くこと | 書かないこと（責務の所在） |
|---|---|
| ディレクトリツリー（tree形式） | コード例（design-patterns） |
| 各ディレクトリの役割・責務 | 実装パターン（design-patterns） |
| ファイル命名規則・配置規則 | コーディング規約（development-guidelines） |
| 依存ルール（インポート制約） | 技術選定理由（architecture） |

**判断基準**: 「何をどこに置くか」を書く。「どう書くか」は development-guidelines または design-patterns の責務。

## 入出力

| 項目 | 内容 |
|---|---|
| 入力 | PRD + 機能設計書 + アーキテクチャ設計書 |
| 出力 | `docs/repository-structure/` |

## ファイル構成

```
docs/repository-structure/
├── index.md              # トップレベル構造・共通ルール・特殊ディレクトリ
└── {stack}.md            # スタック固有のディレクトリ詳細（1スタック = 1ファイル）
```

### 概要（index.md）の役割

- プロジェクト全体のトップレベルディレクトリ構造を記載
- 共通の命名規則・除外設定を記載
- 特殊ディレクトリ（`.claude/`, `docs/`, `.issue/`）の説明を記載
- Agentはindex.mdを参照し、タスクに該当するスタックのみを読み込む

### 個別スタックの命名規則

```
{stack}.md
```

`stack` はスタックを表す kebab-case の名前（例: `frontend.md`, `backend.md`, `infrastructure.md`）

## 読み込み戦略

リポジトリ構造定義書は**選択的読み込み**で使用する。

```
実装タスク開始
  └─ repository-structure/index.md を参照
       └─ タスクに該当するスタックのみ読み込み
```

## 既存ドキュメントの優先順位

| 優先度 | 対象 | 説明 |
|---|---|---|
| 1（最優先） | `docs/repository-structure/` | プロジェクト固有の構造。このスキルのガイドより優先 |
| 2（参考） | このスキル | 汎用テンプレート。既存定義書がない場合に使用 |

## テンプレート

リポジトリ構造定義書を作成する際は次のテンプレートを使用する: ./template.md

テンプレートには index.md 用と個別スタック用の2種類が含まれる。

## 品質基準

### 概要（index.md）のチェック項目

- [ ] プロジェクト全体のトップレベルディレクトリ構造がツリー形式で視覚化されているか
- [ ] 各トップレベルディレクトリの役割・責務が説明されているか
- [ ] 全スタックが一覧に含まれ、詳細ファイルへの参照があるか
- [ ] 特殊ディレクトリ（`.claude/`, `docs/`, `.issue/` 等）の役割が説明されているか
- [ ] 共通の除外設定（.gitignore等）が定義されているか

### 個別スタックのチェック項目

- [ ] スタック固有のディレクトリ構造がツリー形式で視覚化されているか
- [ ] 各ディレクトリの役割・責務が説明されているか
- [ ] レイヤー間の依存ルール（許可/禁止）が表形式で定義されているか
- [ ] ファイル命名規則が種別ごとに明確か
- [ ] テストファイルの配置規則が定義されているか
- [ ] スケーリング戦略（機能追加時のルール）が記載されているか

### 整合性チェック（アーキテクチャ設計書との横断・必須）

- [ ] ディレクトリ構造のレイヤー分割が `docs/architecture/` のレイヤードアーキテクチャ定義と一致しているか
- [ ] ディレクトリ名に使用される技術名が `docs/architecture/` のテクノロジースタック表の名称と一致しているか
- [ ] テストファイルの配置規則が `docs/architecture/` のテスト戦略と整合しているか（E2Eテストのディレクトリ等）
