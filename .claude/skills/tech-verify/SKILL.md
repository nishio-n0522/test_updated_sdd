# 技術検証（WF3 オーケストレーター）

ユーザーの技術要件を対話的に整理し、技術候補の調査→PoC実装→技術選定→デザインパターン設計を順次実行する。

## 起動方法

```
/tech-verify
```

## 全体フロー

```
要件整理（オーケストレーターが直接対話）
  ↓
Phase 1: 技術候補の調査 → レビュー → ユーザー承認（PoC対象選択）
  ↓
Phase 2: PoC実装（候補ごとにループ）→ ユーザー確認
  ↓
Phase 3: 技術選定 → レビュー → ユーザー承認（採用技術決定）
  ↓
Phase 4: デザインパターン設計 → レビュー → ユーザー承認
  ※ Phase 3 で不採用の場合、Phase 4 はスキップ
```

## 要件整理（オーケストレーターが直接対話）

ユーザーと対話して以下を確認する:

| 確認事項 | 内容 |
|---|---|
| 用途 | 何のために技術を導入するのか |
| 制約 | 既存の技術スタック、ライセンス、パフォーマンス要件 |
| 重視する観点 | DX、パフォーマンス、エコシステム、型安全性等 |
| 評価基準 | 必須基準（1つでも不合格なら不採用）と加点基準 |

要件が整理できたら Phase 1 に進む。

## Phase 1: 技術候補の調査

1. `tech-evaluator` エージェントを起動する
   - 入力: 要件 + `docs/architecture.md`
   - 出力: 技術候補一覧（比較表 + 各候補の概要）
2. `doc-reviewer` エージェントを起動する
   - 状況依存スキル: `tech-evaluation`
   - 入力: 技術候補一覧
3. 技術候補一覧 + レビュー結果をユーザーに提示する
4. ユーザーの承認を待つ（PoC対象の候補を選択してもらう）

## Phase 2: PoC実装

選択された各候補技術に対してループ実行する:

1. `poc-implementer` エージェントを起動する
   - 入力: 候補技術 + 要件
   - 出力: `poc/{技術名}/`（`overview.md` + `src/` + `findings.md`）
2. 検証コードをユーザーに提示し、動作確認を依頼する
3. ユーザーの承認を待つ

Phase 2 はドキュメントレビューの対象外（PoCコードは使い捨ての検証コード）。

## Phase 3: 技術選定

1. `tech-evaluator` エージェントを起動する（Phase 1 と同一エージェント、入力が異なる）
   - 入力: 全PoC結果 + 候補一覧 + 要件
   - 出力: `docs/tech-decisions/YYYYMMDD-{topic}.md`
2. `doc-reviewer` エージェントを起動する
   - 状況依存スキル: `tech-evaluation`
   - 入力: 技術選定結果
3. 技術選定結果 + レビュー結果をユーザーに提示する
4. ユーザーの承認を待つ（採用技術を決定してもらう）

### 不採用の場合

ユーザーが全候補を不採用とした場合、Phase 4 をスキップして技術検証を終了する。

## Phase 4: デザインパターン設計

技術が採用された場合のみ実行する:

1. `pattern-designer` エージェントを起動する
   - 入力: 技術選定結果 + `poc/{技術名}/` + `docs/architecture.md` + `docs/design-patterns/`
   - 出力: `docs/design-patterns/{concern}.md`（新規）+ `docs/design-patterns/index.md`（更新）+ `poc/{技術名}/src/`（パターン検証コード）
2. `doc-reviewer` エージェントを起動する
   - 状況依存スキル: `design-patterns`
   - 入力: 新規デザインパターン
3. デザインパターン + 検証コード + レビュー結果をユーザーに提示する
4. ユーザーの承認を待つ

## 完了条件

以下のいずれかを満たした時点で技術検証完了とする:

### 技術採用の場合

- [ ] 技術候補一覧が承認済み
- [ ] 全候補のPoC実装が完了し承認済み
- [ ] `docs/tech-decisions/YYYYMMDD-{topic}.md` が承認済み
- [ ] `docs/design-patterns/{concern}.md` が承認済み
- [ ] `docs/design-patterns/index.md` が更新済み

### 技術不採用の場合

- [ ] 技術候補一覧が承認済み
- [ ] 全候補のPoC実装が完了し承認済み
- [ ] `docs/tech-decisions/YYYYMMDD-{topic}.md` が承認済み（不採用理由を記録）

## エラー時の動作

| ケース | 動作 |
|---|---|
| `docs/architecture.md` が存在しない | 警告を表示し、既存スタック情報なしで Phase 1 を実行する |
| サブエージェントがエラーで終了 | エラー内容をユーザーに報告し、再実行するか確認する |
| PoC実装中にビルドエラー | エラー内容を findings.md に記録し、ユーザーに報告する |
| ユーザーがフェーズをスキップしたい | 該当フェーズをスキップし、次のフェーズに進む。依存するフェーズがある場合は警告する |
