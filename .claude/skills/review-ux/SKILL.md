# UI/UXレビュースキル

画面仕様書・ユーザー体験設計をUI/UXの専門的な観点からレビューするためのヒューリスティクス、チェックリスト、基準を定義する。

## スコープ

- **提供するもの**: UI/UXレビューのヒューリスティクス、チェックリスト、分類基準、出力フォーマット
- **提供しないもの**: レビュー手順（Agent が定義）、プロジェクト固有のUI設計（`docs/screen-specification/` が定義）

## 参照ドキュメント

レビュー前に必ず以下を読み込むこと:

- `docs/product-requirements.md` — ユーザーペルソナ・ユースケースの理解
- `docs/functional-design.md` — 機能一覧・ユーザーフロー
- `docs/screen-specification/index.md` — 画面一覧・画面遷移図・共通挙動仕様
- 対象画面の `user-experience.md`（存在する場合）
- 対象画面の `wireframes.md` / `behaviors.md`（または単一ファイル）

## レビュー対象

- `docs/screen-specification/` 内の指定されたファイル
- 新規作成・変更された画面仕様書およびユーザー体験設計ドキュメント

## ヒューリスティクス（評価原則）

### H1: 操作の一貫性

- 同じ操作が異なる画面で異なる挙動をしていないか
- 用語・ラベル・アイコンの使い方が統一されているか
- 共通コンポーネント（サイドバー、ダイアログ等）の挙動が全画面で統一されているか

### H2: ユーザーの認知負荷

- 1つの画面・ビュー・ダイアログに過剰な情報量を詰め込んでいないか
- 操作ステップ数が必要最小限か（不要な確認画面・モーダルがないか）
- 状態の変化がユーザーに明確にフィードバックされるか

### H3: エラー防止と回復

- 破壊的操作（削除等）に確認ダイアログがあるか
- エラー発生時にユーザーが何をすべきか明確か
- 元に戻す（Undo）操作が提供されているか（特に編集系操作）
- 入力値のバリデーションが適切なタイミングで行われるか

### H4: ナビゲーションと現在地の認識

- ユーザーが「今どこにいるか」を常に把握できるか（パンくず、アクティブ状態等）
- 前の画面に戻る手段が明確か
- 画面遷移に抜け漏れがないか（遷移先から戻れない状態がないか）

### H5: 操作効率

- 頻繁に行う操作に過剰なステップが必要でないか
- キーボードショートカット・バッチ操作など効率化手段が検討されているか
- 大量データ処理時のパフォーマンス考慮があるか（ページネーション、仮想スクロール等）

### H6: ユーザーフローの完結性

- ユーザーの目的（タスク）が画面仕様の操作フローで達成可能か
- フローの途中で中断・再開する場合の挙動が定義されているか
- エッジケース（データ0件、大量データ、並行操作等）の挙動が定義されているか

### H7: 情報の視認性と階層

- 重要な情報が目立つ位置に配置されているか
- 関連する情報がグルーピングされているか
- 読み取り専用と編集可能の区別が視覚的に明確か

### H8: 対象ユーザーへの適合性

- PRDで定義されたペルソナ（製造技術担当者、オペレーター等）のスキルレベルに適合した複雑さか
- 専門用語の使用が対象ユーザーにとって適切か
- 非機能要件（操作習得時間等）を満たせる設計か

## チェックリスト

### 1. ユーザー体験設計（user-experience.md）

- [ ] 主要なユーザーシナリオが網羅されているか
- [ ] シナリオが具体的なステップで記述されているか（抽象的でないか）
- [ ] 設計判断の理由が記録されているか
- [ ] シナリオ間の矛盾がないか
- [ ] 中断・再開・エラー時のシナリオが考慮されているか

### 2. ワイヤーフレーム（wireframes.md）

- [ ] 全状態（初期・データあり・空・エラー・ローディング等）のワイヤーフレームがあるか
- [ ] レイアウトがユーザーの操作フローに適したものか
- [ ] 情報の優先順位がレイアウトに反映されているか（重要な情報が目立つ位置にあるか）
- [ ] 画面間の視覚的な一貫性があるか

### 3. 操作挙動（behaviors.md）

- [ ] 全てのインタラクティブ要素に操作挙動が定義されているか
- [ ] 正常系・異常系の両方が定義されているか
- [ ] 操作のフィードバック（トースト、インライン表示等）が適切か
- [ ] 画面遷移のトリガーと遷移先が明確か

### 4. 画面間の整合性

- [ ] 画面遷移図と各画面の遷移定義が一致しているか
- [ ] 共通コンポーネントの使用が一貫しているか
- [ ] データの受け渡し（画面A→画面Bの遷移時に何を引き継ぐか）が定義されているか

## 指摘の分類基準

### MUST（修正必須）

- **ユーザーフロー断絶**: タスクを完了できない操作フローの欠陥
- **一貫性の重大な違反**: 同じ操作が画面によって異なる挙動
- **エラーハンドリング欠如**: 破壊的操作に確認なし、エラー時の挙動未定義
- **PRD要件との不整合**: PRDのユースケース・ペルソナ要件を満たさない設計

### SHOULD（修正推奨）

- 認知負荷が高い設計（改善により大幅にUX向上が見込まれるもの）
- 操作効率の改善提案（ショートカット追加、ステップ削減等）
- 情報設計の改善（グルーピング、視認性向上）
- エッジケースの挙動未定義

### MAY（検討事項）

- より良いUXパターンの提案
- 将来的なUX改善の方向性
- アクセシビリティの改善提案

## 出力フォーマット

```markdown
# UI/UXレビュー結果

## 対象
- レビュー対象: {ファイルパス一覧}
- 対象画面: {SCR-XXX: 画面名}

## サマリ
- 総合判定: PASS / FAIL
- MUST指摘: {件数}件
- SHOULD指摘: {件数}件
- MAY指摘: {件数}件

## MUST（修正必須）

### 指摘1
- ファイル: {パス}
- ヒューリスティクス: {H1〜H8のいずれか}
- 問題: {問題の説明}
- 影響: {ユーザーにどのような悪影響があるか}
- 修正案: {具体的な修正提案}

## SHOULD（修正推奨）

### 指摘1
- ファイル: {パス}
- ヒューリスティクス: {H1〜H8のいずれか}
- 問題: {問題の説明}
- 改善案: {具体的な改善提案}

## MAY（検討事項）

### 指摘1
- ヒューリスティクス: {H1〜H8のいずれか}
- 提案: {提案内容}
```

**総合判定ルール**: MUST指摘が1件でもあれば FAIL、0件なら PASS
