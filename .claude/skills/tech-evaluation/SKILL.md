# 技術評価スキル

技術候補の調査・比較・選定に使用する方法論とフレームワーク。

## 評価フレームワーク

### Phase 1: 技術候補の調査・比較

技術要件とアーキテクチャ制約に基づき、候補技術を調査・比較する。

#### 調査項目

各候補技術について以下を調査する:

| 項目 | 内容 |
|---|---|
| 基本情報 | バージョン、ライセンス、メンテナンス状況、最終リリース日 |
| 技術概要 | 主要な特徴、設計思想 |
| エコシステム | 関連ライブラリ、ツール、コミュニティ規模 |
| 既存スタックとの互換性 | `docs/architecture.md` との整合性 |
| 導入コスト | 学習コスト、既存コード改修の影響範囲 |

#### 比較表フォーマット

```markdown
| 観点 | {候補A} | {候補B} | {候補C} |
|---|---|---|---|
| ライセンス | {内容} | {内容} | {内容} |
| メンテナンス状況 | {Active/Maintenance/Deprecated} | ... | ... |
| 既存スタックとの互換性 | {高/中/低} | ... | ... |
| 学習コスト | {低/中/高} | ... | ... |
| コミュニティ規模 | {大/中/小} | ... | ... |
| パフォーマンス | {概要} | ... | ... |
```

### Phase 3: PoC結果を踏まえた評価・推薦

全PoC結果を統合し、メリット・デメリットを比較して推薦を行う。

#### 評価基準

| 基準 | 判定 | 説明 |
|---|---|---|
| 必須基準 | 合格/不合格 | 1つでも不合格なら不採用 |
| 加点基準 | 合格/不合格 | 合格数が多いほど好ましい |

#### 出力フォーマット（Phase 3: 技術選定結果）

```markdown
# 技術選定結果: {トピック}

## 結論
**推薦: {技術名}**
{推薦理由を2-3行で記述}
```

## 品質基準チェックリスト

### Phase 1 出力

- [ ] 全候補について調査項目が網羅されている
- [ ] 比較表が同一の観点で統一されている
- [ ] `docs/architecture.md` との互換性が分析されている
- [ ] PoC推奨候補の選定理由が明記されている
- [ ] 候補数が2〜4件の範囲である

### Phase 3 出力

- [ ] 全PoC結果が評価に反映されている
- [ ] 必須基準・加点基準の判定にエビデンス（PoC結果）が紐付いている
- [ ] メリット・デメリットが具体的である（抽象的な記述ではない）
- [ ] 推薦理由が比較に基づいている（単独評価ではない）
- [ ] 不採用候補の理由が記録されている
- [ ] 導入時の注意事項が記載されている
