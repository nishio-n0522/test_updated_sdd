# i18nレビュースキル

差分コードが多言語対応（日本語・英語）のルールに準拠しているかをレビューするためのチェックリストと基準。

## スコープ

- **提供するもの**: i18nレビューのチェックリスト、分類基準、出力フォーマット
- **提供しないもの**: レビュー手順（Agent が定義）、プロジェクト固有のi18n設定（`docs/development-guidelines.md` が定義）

## 参照ドキュメント

レビュー前に必ず以下を読み込むこと:

- `docs/development-guidelines.md` — 国際化（i18n）ガイドラインセクション
- `docs/glossary.md` — ドメイン用語の日英表記
- 翻訳ファイル（プロジェクトのi18n設定に応じたパス）

## レビュー対象範囲

`git diff main...HEAD` で取得される差分ファイルのみ。差分に含まれないファイルはレビュー対象外。

## チェックリスト

### 1. ハードコードテキストの検出

- [ ] UIに表示される日本語テキストがコンポーネント内にハードコードされていないか
- [ ] UIに表示される英語テキストがコンポーネント内にハードコードされていないか
- [ ] `aria-label`、`placeholder`、`title`属性にハードコードされたテキストがないか
- [ ] エラーメッセージ、トースト通知、確認ダイアログのテキストがハードコードされていないか
- [ ] 定数ファイルにUIテキストが直接記述されていないか

例外（指摘対象外）: コード内コメント、ログ出力、テスト説明文、技術的な文字列（URL、CSSクラス名等）、ブランド名・固有名詞

### 2. 翻訳ファイルの整合性

- [ ] 差分で追加・変更された翻訳キーが全言語ファイルに存在するか
- [ ] 翻訳キーの値が空文字列 `""` になっていないか
- [ ] プレースホルダー変数（`{name}`, `{count}` 等）が全言語で一致しているか

### 3. 翻訳の品質

- [ ] 翻訳が自然な表現として適切か（機械翻訳的な不自然さがないか）
- [ ] `docs/glossary.md` で定義されたドメイン用語の表記と一致しているか
- [ ] UIコンテキストに適した訳語が使われているか

### 4. 用語の一貫性

- [ ] 同じ概念・操作に対して同じ訳語が使われているか
- [ ] 既存の翻訳キーと矛盾する新しい訳語が導入されていないか
- [ ] ボタンラベル、メニュー項目、見出しなどで同じ機能に異なる表現が使われていないか

### 5. 翻訳関数の使用パターン

- [ ] 翻訳関数のインポートと使用がプロジェクトのパターンに従っているか
- [ ] 名前空間がガイドラインに沿っているか
- [ ] 新しい名前空間を導入した場合、全言語ファイルにセクションが追加されているか

### 6. コンテキスト依存の翻訳

- [ ] 単数形/複数形の区別が必要な箇所で適切に処理されているか
- [ ] 日付・時刻・数値のフォーマットがロケールに依存する場合、適切に処理されているか
- [ ] テキストの連結で語順の問題が発生しないか

## 指摘の分類基準

### MUST（修正必須）

- UIに表示されるテキストのハードコード（翻訳関数未使用）
- 翻訳キーの言語間不整合（一方にのみキーが存在）
- 翻訳キーの値が空文字列
- `glossary.md` の定義と明確に矛盾する訳語の使用

### SHOULD（修正推奨）

- 翻訳の品質に改善の余地がある（不自然な表現、より適切な訳語がある等）
- 同じ概念に対する訳語の軽微な不統一
- 名前空間の選択がガイドラインと異なる

### MAY（検討事項）

- 翻訳キー名の改善提案
- 既存の翻訳キーの再利用提案
- 将来的なi18n対応の改善提案

```markdown
## サマリ
- 総合判定: PASS / FAIL
- MUST指摘: {件数}件
- SHOULD指摘: {件数}件
- MAY指摘: {件数}件

## 翻訳キー整合性チェック
- 一方の言語のみに存在するキー: {件数}件
- 空文字列のキー: {件数}件

## MUST（修正必須）

### 指摘1
- ファイル: {パス}
- 行: {行番号}
- カテゴリ: {ハードコードテキスト / 翻訳キー不整合 / 空文字列 / 用語不一致}
- 問題: {問題の説明}
- 修正案: {具体的な修正提案}
- 根拠: {docs/内のどのドキュメントのどの記述に基づくか}

## SHOULD（修正推奨）

### 指摘1
- ファイル: {パス}
- 行: {行番号}
- カテゴリ: {翻訳品質 / 用語の軽微な不統一 / 名前空間}
- 問題: {問題の説明}
- 修正案: {具体的な修正提案}
- 根拠: {根拠}

## MAY（検討事項）

### 指摘1
- ファイル: {パス}
- カテゴリ: {翻訳キー名の改善 / 既存キーの再利用 / 将来的な改善}
- 提案: {提案内容}
- 根拠: {根拠}
```

**出力先**: `.issue/{N}/reviews/review-i18n-r{ラウンド番号}.md`

**総合判定ルール**: MUST指摘が1件でもあれば FAIL、0件なら PASS
