# issue仕様スキル

issue種別に応じた固有ドキュメントのテンプレートと品質基準を提供する。

## スコープ

- **提供するもの**: 種別ごとのドキュメントテンプレート、品質基準、記述ガイドライン
- **提供しないもの**: ドキュメント作成手順（Agent が定義）、プロジェクト固有の内容

## issue種別と固有ドキュメント

| issue種別 | 固有ドキュメント | 配置先 |
|---|---|---|
| 不具合修正 | 不具合調査レポート | `.issue/{N}/bug-report.md` |
| リファクタリング | リファクタリング方針書 | `.issue/{N}/refactoring-plan.md` |
| パフォーマンス改善 | パフォーマンス改善計画書 | `.issue/{N}/performance-plan.md` |
| 依存ライブラリ・基盤更新 | 更新計画書 | `.issue/{N}/update-plan.md` |

※ 「機能の追加・変更」は WF4 の Phase 2a で機能設計・画面仕様エージェントが仕様を作成するため、本スキルの対象外。

## テンプレート

### 不具合調査レポート（bug-report.md）

```markdown
# 不具合調査レポート

## 事象

[不具合の内容を1-2文で記述]

## 再現手順

1. [手順1]
2. [手順2]
3. [手順3]

## 期待される動作

[本来の正しい動作]

## 実際の動作

[現在発生している動作]

## 影響範囲

- 影響を受ける機能: [機能名]
- 影響を受けるユーザー: [対象ユーザー層]
- 発生頻度: [常時 / 特定条件下 / 不定期]

## 原因分析

### 推定原因

[原因の仮説]

### 関連コード

- `[ファイルパス]`: [該当箇所の説明]

## 修正方針

[修正のアプローチ]

## テスト観点

- [ ] [再現テスト: 修正前に再現することを確認]
- [ ] [修正確認テスト: 修正後に事象が解消されることを確認]
- [ ] [回帰テスト: 修正による副作用がないことを確認]
```

### リファクタリング方針書（refactoring-plan.md）

```markdown
# リファクタリング方針書

## 目的

[リファクタリングの動機と期待される効果]

## 対象範囲

| 対象 | パス | 変更内容 |
|---|---|---|
| [対象1] | `[パス]` | [変更概要] |

## 方針

### 適用するパターン・原則

- [パターン/原則名]: [適用方法]

### 変更前後の構造

#### Before

[現在の構造の概要]

#### After

[リファクタリング後の構造]

## 制約

- 機能変更を含まないこと
- 既存テストが全て通ること

## テスト観点

- [ ] [既存テストの全パス]
- [ ] [変更対象のコードパスが既存テストでカバーされていることの確認]
```

### パフォーマンス改善計画書（performance-plan.md）

```markdown
# パフォーマンス改善計画書

## 改善対象

[改善対象の機能・処理]

## 現状の計測結果

| 指標 | 現在値 | 目標値 | 計測方法 |
|---|---|---|---|
| [指標1] | [値] | [値] | [方法] |

## 原因分析

[ボトルネックの特定結果]

## 改善方針

### アプローチ

[改善のアプローチ]

### 期待される効果

[定量的な改善見込み]

## 影響範囲

- [影響を受ける機能・コンポーネント]

## テスト観点

- [ ] [改善後の計測: 目標値を達成していることの確認]
- [ ] [機能テスト: 既存機能に影響がないことの確認]
```

### 更新計画書（update-plan.md）

```markdown
# 更新計画書

## 更新対象

| パッケージ | 現バージョン | 更新後バージョン | 変更種別 |
|---|---|---|---|
| [パッケージ名] | [現在] | [更新後] | major/minor/patch |

## 更新理由

[更新の動機: セキュリティ修正、EOL対応、新機能利用等]

## Breaking Changes

| 変更内容 | 影響範囲 | 対応方針 |
|---|---|---|
| [変更1] | [影響を受けるコード] | [対応方法] |

## 移行手順

1. [手順1]
2. [手順2]

## テスト観点

- [ ] [ビルド成功の確認]
- [ ] [既存テストの全パス]
- [ ] [Breaking Changesの対応箇所が正しく動作することの確認]
```

## 品質基準

### 全種別共通

- [ ] issue種別に対応する正しいテンプレートが使用されているか
- [ ] 全セクションが記入されているか（該当なしの場合は「該当なし」と明記）
- [ ] テスト観点が具体的で測定可能か
- [ ] 関連する既存ドキュメント（`docs/`）との整合性があるか

### 不具合修正

- [ ] 再現手順が第三者にも再現可能な具体性を持つか
- [ ] 原因分析に関連コードへの参照が含まれるか
- [ ] 修正方針が原因に対応しているか

### リファクタリング

- [ ] 「機能変更を含まない」制約が明記されているか
- [ ] 変更前後の構造が比較可能な形で記述されているか
- [ ] 適用するパターン・原則の根拠があるか

### パフォーマンス改善

- [ ] 現状の計測結果に具体的な数値があるか
- [ ] 目標値が設定されているか
- [ ] 計測方法が再現可能か

### 依存ライブラリ・基盤更新

- [ ] 更新対象のバージョン情報が正確か
- [ ] Breaking Changesの有無が確認されているか
- [ ] 移行手順が実行可能な粒度で記述されているか
