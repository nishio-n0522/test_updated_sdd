# issue分解・登録スキル

仕様ドキュメントをGitHub issueに分解し、`.issue/` ディレクトリに詳細ドキュメントを配置する際のドメイン知識。

## スコープ

- **提供するもの**: issue分解の方法論、GitHub issueフォーマット、`.issue/` 構造規約
- **提供しないもの**: 分解手順（Agent が定義）、プロジェクト固有のissue内容

## 入出力

| 項目 | WF2（分解+登録） | WF4（登録のみ） |
|---|---|---|
| 入力 | 全仕様ドキュメント | 単一issueの仕様ドキュメント |
| 出力 | GitHub issues + `.issue/{N}/` | GitHub issue + `.issue/{N}/` |

## issueの二層構造

| 層 | 配置先 | 詳細度 | 役割 |
|---|---|---|---|
| GitHub issue | GitHub | 低（概要） | 人間が管理・確認するインデックス |
| issueディレクトリ | `.issue/{issue番号}/` | 高（詳細） | autopilotが参照する主な情報源 |

## issue分解の方法論

### 分解の粒度

1つのissueは以下の条件を満たすサイズにする:

- 1つの明確な目的を持つ
- 独立して実装・テスト可能
- 他のissueへの依存が最小限

### 依存関係の管理

- issueは依存関係を考慮した適切な順序で登録する
- issue #N を実行するために先に実施すべきissueがある場合、それは #N より小さい番号で登録する
- 依存関係はGitHub issueの本文に明記する

### 分解の順序

1. **基盤・共通部分**: 他の機能が依存する共通コンポーネント・設定
2. **データモデル**: エンティティ定義、DB設計
3. **コア機能**: MVP機能を優先度順に
4. **UI/画面**: 画面実装（コア機能と並行可能な場合は並行）
5. **統合・調整**: 機能間の連携、E2Eテスト

## GitHub issueフォーマット

```markdown
## 概要

[issueの目的を1-2文で説明]

## 背景

[なぜこのissueが必要か]

## スコープ

### 含めるもの
- [実装対象1]
- [実装対象2]

### 含めないもの
- [除外対象1]

## 受け入れ条件

- [ ] [条件1]
- [ ] [条件2]

## 依存関係

- 前提: #[先行issue番号]（[簡潔な説明]）

## 参照ドキュメント

- `.issue/{N}/` 配下の詳細ドキュメント
```

## .issue/ ディレクトリ構造

```
.issue/{issue番号}/
├── spec.md              # issueの詳細仕様（実装に必要な情報を集約）
└── [種別固有ドキュメント]  # issue種別に応じた追加ドキュメント
```

### spec.md の内容

- issueの詳細な要件と仕様
- 関連する永続ドキュメント（PRD, 機能設計書等）からの抜粋・参照
- 実装に必要な技術的な詳細
- テスト観点

## 品質基準

### 必須チェック項目

- [ ] 各issueが1つの明確な目的を持つか
- [ ] issueが独立して実装・テスト可能か
- [ ] 依存関係が正しい順序で番号付けされているか
- [ ] 受け入れ条件が測定可能か
- [ ] `.issue/{N}/spec.md` に実装に十分な情報が含まれているか
- [ ] PRDの全機能要件がissueでカバーされているか（WF2の場合）
