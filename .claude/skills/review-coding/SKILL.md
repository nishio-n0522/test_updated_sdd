# コーディング規約レビュースキル

差分コードがプロジェクトのコーディング規約に準拠しているかをレビューするためのチェックリストと基準です。

## 参照ドキュメント

レビュー前に必ず以下を読み込むこと:

- `docs/development-guidelines.md` — コーディング規約の定義
- `docs/design-patterns/` — 技術スタック固有のパターン（存在する場合）

## レビュー対象範囲

レビュー対象は `git diff main...HEAD` で取得される差分ファイルのみです。差分に含まれないファイルはレビュー対象外です。

## チェックリスト

### 1. 命名規則

- [ ] 変数名がプロジェクトの命名規約に従っているか（camelCase / snake_case 等）
- [ ] 関数名が動詞で始まり、処理内容を適切に表現しているか
- [ ] クラス名・型名がPascalCaseで命名されているか
- [ ] 定数名がプロジェクトの規約に従っているか（UPPER_SNAKE_CASE 等）
- [ ] ファイル名がプロジェクトの規約に従っているか
- [ ] 略語の使用がプロジェクトの慣習に沿っているか

### 2. コードスタイル

- [ ] インデントがプロジェクト規約通りか（タブ / スペース数）
- [ ] セミコロン、クォートのスタイルが統一されているか
- [ ] import文の順序がプロジェクト規約に従っているか
- [ ] 不要なimportが残っていないか
- [ ] コンソール出力（console.log, print等）がデバッグ目的で残っていないか

### 3. 関数・メソッド設計

- [ ] 1つの関数が単一の責務を持っているか
- [ ] 関数の引数が多すぎないか（目安: 3個以内、多くても5個）
- [ ] 戻り値の型が明確か（型付き言語の場合）
- [ ] 副作用が最小限か
- [ ] ネストが深すぎないか（目安: 3段以内）

### 4. エラーハンドリング

- [ ] try-catchが適切に使用されているか
- [ ] エラーが無視（catch(e) {}）されていないか
- [ ] エラーメッセージが具体的で、デバッグに役立つか
- [ ] エラーの伝播が適切か（握り潰し・過剰な再スローがないか）
- [ ] ユーザー向けのエラーメッセージに内部情報が含まれていないか

### 5. 型安全性（型付き言語の場合）

- [ ] `any` 型の使用が最小限か
- [ ] 型アサーション（as）が不必要に使われていないか
- [ ] nullableな値の扱いが適切か（オプショナルチェーン、null合体演算子等）
- [ ] ジェネリクスが適切に活用されているか

### 6. テストコードの品質

- [ ] テスト名が「何をテストしているか」を明確に表現しているか
- [ ] テストが独立して実行可能か（他のテストに依存していないか）
- [ ] アサーションが具体的で意味のあるものか
- [ ] エッジケースがカバーされているか
- [ ] モック・スタブが適切に使用されているか

### 7. セキュリティ

- [ ] ユーザー入力のバリデーションが適切か
- [ ] SQLインジェクション・XSSなどの脆弱性がないか
- [ ] 機密情報がハードコードされていないか
- [ ] 認証・認可のチェックが適切か（該当する場合）

### 8. パフォーマンス

- [ ] 不要なループや再計算がないか
- [ ] メモリリークの可能性がないか
- [ ] 非同期処理が適切に扱われているか
- [ ] N+1問題がないか（データベースアクセスがある場合）

## 指摘の分類基準

### MUST（修正必須）

以下に該当する場合はMUSTとして指摘する:

- コーディング規約への**明確な違反**
- バグの可能性がある問題
- セキュリティ上の脆弱性
- エラーの握り潰し
- 型安全性の重大な問題（any の乱用等）

### SHOULD（修正推奨）

以下に該当する場合はSHOULDとして指摘する:

- 規約からの**軽微な逸脱**
- 可読性の改善
- 命名の改善提案
- テストカバレッジの不足
- パフォーマンスの軽微な改善

### MAY（検討事項）

以下に該当する場合はMAYとして指摘する:

- より良い実装パターンの提案
- リファクタリングの提案
- 将来的な拡張性の改善

## 出力フォーマット

レビュー結果は `.claude/skills/steering/templates/review-coding.md` のテンプレートに従って出力すること。

**総合判定ルール**: MUST指摘が1件でもあれば FAIL、0件なら PASS
