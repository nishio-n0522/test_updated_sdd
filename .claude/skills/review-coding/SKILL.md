# コーディング規約レビュースキル

差分コードがプロジェクトのコーディング規約に準拠しているかをレビューするためのチェックリストと基準。

## スコープ

- **提供するもの**: コーディング規約レビューのチェックリスト、分類基準、出力フォーマット
- **提供しないもの**: レビュー手順（Agent が定義）、プロジェクト固有の規約（`docs/development-guidelines.md` が定義）

## 参照ドキュメント

レビュー前に必ず以下を読み込むこと:

- `docs/development-guidelines.md` — コーディング規約の定義
- `docs/design-patterns/` — 技術スタック固有のパターン（存在する場合）

## レビュー対象範囲

`git diff main...HEAD` で取得される差分ファイルのみ。差分に含まれないファイルはレビュー対象外。

## チェックリスト

### 1. 命名規則

- [ ] 変数名がプロジェクトの命名規約に従っているか（camelCase / snake_case 等）
- [ ] 関数名が動詞で始まり、処理内容を適切に表現しているか
- [ ] クラス名・型名がPascalCaseで命名されているか
- [ ] 定数名がプロジェクトの規約に従っているか
- [ ] ファイル名がプロジェクトの規約に従っているか

### 2. コードスタイル

- [ ] import文の順序がプロジェクト規約に従っているか
- [ ] 不要なimportが残っていないか
- [ ] コンソール出力（console.log等）がデバッグ目的で残っていないか

### 3. 関数・メソッド設計

- [ ] 1つの関数が単一の責務を持っているか
- [ ] 関数の引数が多すぎないか（目安: 3個以内、多くても5個）
- [ ] 戻り値の型が明確か（型付き言語の場合）
- [ ] ネストが深すぎないか（目安: 3段以内）

### 4. エラーハンドリング

- [ ] try-catchが適切に使用されているか
- [ ] エラーが無視（catch(e) {}）されていないか
- [ ] エラーメッセージが具体的で、デバッグに役立つか
- [ ] ユーザー向けのエラーメッセージに内部情報が含まれていないか

### 5. 型安全性（型付き言語の場合）

- [ ] `any` 型の使用が最小限か
- [ ] 型アサーション（as）が不必要に使われていないか
- [ ] nullableな値の扱いが適切か

### 6. テストコードの品質

- [ ] テスト名が「何をテストしているか」を明確に表現しているか
- [ ] テストが独立して実行可能か
- [ ] アサーションが具体的で意味のあるものか
- [ ] エッジケースがカバーされているか

### 7. 配色ルール

- [ ] Tailwind の直接色指定クラス（`text-gray-*`, `bg-blue-*` 等）が使われていないか
- [ ] ハードコードされた hex/rgb/hsl 値がスタイルに使われていないか
- [ ] `dark:` プレフィックスによる手動色分岐が不必要に使われていないか
- [ ] セマンティックトークン（`text-foreground`, `bg-primary` 等）が使用されているか

例外（指摘対象外）: カラーピッカーUI、オーバーレイ `bg-black/50`、テストデータ

### 8. セキュリティ

- [ ] ユーザー入力のバリデーションが適切か
- [ ] SQLインジェクション・XSSなどの脆弱性がないか
- [ ] 機密情報がハードコードされていないか

### 9. パフォーマンス

- [ ] 不要なループや再計算がないか
- [ ] メモリリークの可能性がないか
- [ ] 非同期処理が適切に扱われているか

## 指摘の分類基準

### MUST（修正必須）

- コーディング規約への**明確な違反**
- バグの可能性がある問題
- セキュリティ上の脆弱性
- エラーの握り潰し
- 型安全性の重大な問題（any の乱用等）
- ハードコード配色の使用（セマンティックトークン未使用）

### SHOULD（修正推奨）

- 規約からの**軽微な逸脱**
- 可読性の改善
- 命名の改善提案
- テストカバレッジの不足

### MAY（検討事項）

- より良い実装パターンの提案
- リファクタリングの提案
- 将来的な拡張性の改善

## 出力フォーマット

```markdown
# コーディング規約レビュー結果（ラウンド{N}）

## サマリ
- 総合判定: PASS / FAIL
- MUST指摘: {件数}件
- SHOULD指摘: {件数}件
- MAY指摘: {件数}件

## MUST（修正必須）

### 指摘1
- ファイル: {パス}
- 行: {行番号}
- 規約: {違反している規約名}
- 問題: {問題の説明}
- 修正案: {具体的な修正提案}

## SHOULD（修正推奨）

### 指摘1
- ファイル: {パス}
- 行: {行番号}
- 規約: {関連する規約名}
- 問題: {問題の説明}
- 修正案: {具体的な修正提案}

## MAY（検討事項）

### 指摘1
- ファイル: {パス}
- 提案: {改善提案の説明}
- メリット: {この改善のメリット}
```

**出力先**: `.issue/{N}/reviews/review-coding-r{ラウンド番号}.md`

**総合判定ルール**: MUST指摘が1件でもあれば FAIL、0件なら PASS
