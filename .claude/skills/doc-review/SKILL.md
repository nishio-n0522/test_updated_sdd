# ドキュメントレビュースキル

仕様ドキュメント（PRD、機能設計書、画面仕様書等）をレビューする際のドメイン知識。レビュー観点、重要度分類、出力フォーマットを定義する。

## スコープ

- **レビュー対象**: 仕様ドキュメント（PRD、機能設計書、画面仕様書、アーキテクチャ設計書、開発ガイドライン、リポジトリ構造定義書、用語集、技術選定結果、デザインパターン、issue仕様）
- **レビュー対象外**: アイデアメモ（非公式な対話成果物）、issue登録（構造的な処理）

## 重要度分類

| 分類 | 意味 | 対応 |
|---|---|---|
| **MUST** | 修正必須。品質基準を満たしていない | レビュー後に必ず修正する |
| **SHOULD** | 修正推奨。品質向上に寄与する | 可能な限り修正する |
| **MAY** | 任意。将来的な改善候補 | 時間があれば対応 |

## レビュー観点

### 1. 完全性

- 必要なセクションが全て含まれているか
- 各セクションに十分な情報が記載されているか
- 前提条件が明記されているか

### 2. 明確性

- 曖昧な表現（「適切に」「必要に応じて」等）がないか
- 用語が一貫して使われているか
- 具体例が含まれているか

### 3. 一貫性

- 他のドキュメントと矛盾がないか
- 用語・フォーマットが統一されているか
- 数値や日付が整合しているか

### 4. 実装可能性

- 開発者が実装に必要な情報が揃っているか
- 技術的に実現可能か
- 依存関係が明確か

### 5. 測定可能性

- 成功基準が測定可能か
- パフォーマンス要件に具体的な数値があるか
- 受け入れ基準が定義されているか

### 6. 責務境界

- ドキュメントが自身の責務範囲内の内容のみを記載しているか
- 他のドキュメントの責務に踏み込んでいないか（対応するスキルの「記載しない内容」を参照）
- 適切な抽象度で記述されているか（上流ドキュメントほど抽象的、下流ほど具体的）

## ドキュメント種別ごとの追加観点

| 種別 | 追加チェック項目 |
|---|---|
| PRD | ペルソナの具体性、KPI定義、優先度設定、スコープ外の明示 |
| 機能設計書 | システム構成図、データモデル、シーケンス図、エラーハンドリング |
| 画面仕様書 | PRD/機能設計書との整合性、全UI要素の挙動定義、画面遷移の網羅性 |
| アーキテクチャ設計書 | 技術選定の理由、レイヤー構造、パフォーマンス/セキュリティ要件 |
| 開発ガイドライン | コード例の有無、ブランチ戦略、テスト戦略、レビュープロセス |
| リポジトリ構造定義書 | ディレクトリ構造の視覚化、依存ルール、命名規則、スケーリング戦略 |
| 用語集 | 分類の適切性、定義の明確性、使用例、相互参照 |

## 出力フォーマット

```markdown
## レビュー結果: [ドキュメント名]

### 総合評価

| 観点 | 判定 |
|---|---|
| 完全性 | MUST / SHOULD / MAY / OK |
| 明確性 | MUST / SHOULD / MAY / OK |
| 一貫性 | MUST / SHOULD / MAY / OK |
| 実装可能性 | MUST / SHOULD / MAY / OK |
| 測定可能性 | MUST / SHOULD / MAY / OK |
| 責務境界 | MUST / SHOULD / MAY / OK |

### 良い点

- [具体的な良い点1]
- [具体的な良い点2]

### 指摘事項

#### [MUST] [問題の概要]

- **箇所**: [セクション名または行番号]
- **問題**: [何が問題か]
- **改善案**: [具体的な修正提案]

#### [SHOULD] [問題の概要]

- **箇所**: [セクション名]
- **問題**: [何が問題か]
- **改善案**: [具体的な修正提案]
```

## レビューの姿勢

- **建設的**: 批判ではなく改善のための提案を行う
- **具体的**: 「わかりにくい」ではなく、「どこが」「なぜ」「どう改善するか」を明示
- **バランス**: 良い点も必ず指摘する
- **根拠**: 改善提案には必ず理由を添える
