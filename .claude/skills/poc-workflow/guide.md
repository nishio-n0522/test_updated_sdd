# PoC ワークフロー詳細ガイド

## 概要

このガイドは、`/poc [技術名]` コマンドで起動されるPoCワークフローの各フェーズの詳細手順を定義します。

---

## フェーズ1: 技術リサーチとPoC構築

### 目的

対象技術の基本を理解し、最小限の動作する検証コードを構築する。

### 手順

1. **公式ドキュメント調査**
   - WebSearchを使って対象技術の公式ドキュメント、Getting Startedを調査
   - バージョン情報、ライセンス、メンテナンス状況を確認
   - プロジェクトの技術スタック（`docs/architecture.md`）との互換性を確認

2. **overview.md の作成**
   - テンプレート（`./templates/poc-overview.md`）を使用
   - `poc/[技術名]/overview.md` に出力
   - 技術概要、目的、評価基準、検証スコープを記載

3. **最小検証コードの生成**
   - `poc/[技術名]/src/` に最小限の動作する検証コードを生成
   - **検証に必要な最小限のコードのみ**を書く
   - 本番品質は不要、動作確認が目的
   - 必要な依存関係のインストール手順を `overview.md` に追記

### 成果物

- `poc/[技術名]/overview.md`（完成）
- `poc/[技術名]/src/`（最小限の検証コード）

### ユーザー承認ポイント

```
「フェーズ1（技術リサーチとPoC構築）が完了しました。

[技術の概要を2-3行で簡潔に記述]

overview.md と検証コードを確認してください。
承認いただけたらフェーズ2（技術検証）に進みます。修正が必要な場合はお知らせください。」
```

---

## フェーズ2: 技術検証

### 目的

PoCコードを実行し、評価基準に対する合否を判定する。

### 手順

1. **検証コードの実行**
   - `poc/[技術名]/src/` のコードを実行
   - 動作結果を記録
   - エラーが発生した場合はデバッグし、原因を記録

2. **評価基準に対する検証**
   - `overview.md` の評価基準（必須/加点）に対して、各項目の合否を判定
   - 必須基準が1つでも不合格の場合、「不採用」候補としてマーク
   - 具体的な数値やエビデンスを記録

3. **有名OSSでの採用事例調査**
   - WebSearchを使い、対象技術を採用している有名OSS・実績のあるプロダクトを調査する
   - 各OSSについて以下を記録:
     - プロジェクト名とリポジトリURL
     - 採用している設計パターン（ディレクトリ構造、レイヤー分離、状態管理の方法など）
     - なぜそのパターンを採用しているか（READMEやドキュメントから読み取れる設計思想）
     - プロジェクト規模（スター数、コントリビューター数など、技術的信頼性の参考情報）
   - 最低2〜3件の事例を収集する
   - 調査結果を `findings.md` の「OSSでの採用事例」セクションに記載する

4. **問題点の記録**
   - 技術的な制約・制限事項
   - 既存アーキテクチャとの衝突点
   - パフォーマンス上の懸念
   - 学習コスト・導入コスト

5. **findings.md の作成**
   - テンプレート（`./templates/poc-findings.md`）を使用
   - `poc/[技術名]/findings.md` に出力

### 成果物

- `poc/[技術名]/findings.md`（完成）

### ユーザー承認ポイント（採用/不採用判断）

**採用の場合:**
```
「フェーズ2（技術検証）が完了しました。

結論: [技術名] は採用を推奨します。
- 必須基準: 全項目合格
- 加点基準: [N]/[M] 項目合格

[主要な発見を2-3点記述]

findings.md を確認してください。
承認いただけたらフェーズ3（設計パターン探索）に進みます。」
```

**不採用の場合:**
```
「フェーズ2（技術検証）が完了しました。

結論: [技術名] は不採用を推奨します。
- 不合格の必須基準: [項目名]
- 理由: [具体的な理由]

findings.md を確認してください。
承認いただけたら振り返り（ステップ8）に進みます。
それでもパターン探索に進みたい場合はお知らせください。」
```

**不採用承認の場合**: フェーズ3・4をスキップし、振り返り（ステップ8）に直行する。

## フェーズ3: 設計パターン探索

### 目的

採用する技術をプロジェクトに統合する際の、最適な設計パターンを発見する。

### 手順

1. **既存アーキテクチャ分析**
   - `docs/architecture.md` のアーキテクチャパターンを確認
   - `docs/repository-structure.md` のディレクトリ構造を確認
   - `docs/development-guidelines.md` のコーディング規約を確認
   - 既存コードベースの実装パターンをGrepで調査

2. **有名OSSのデザインパターン深掘り調査**
   - フェーズ2で収集したOSS採用事例をもとに、設計パターンを深掘りする
   - WebSearchで追加の事例・ベストプラクティス記事・公式推奨パターンを調査する
   - 調査観点:
     - **ディレクトリ構成**: OSSがどのようにファイル・モジュールを配置しているか
     - **責務分離**: レイヤー構造、関心の分離をどう実現しているか
     - **テスト戦略**: そのパターンがテスタビリティにどう寄与しているか
     - **スケーラビリティ**: 機能追加時にどう拡張しているか
   - 各パターンについて「このプロジェクトで採用できるか」を以下の基準で評価:
     - 既存アーキテクチャ（`docs/architecture.md`）との整合性
     - プロジェクト規模・チーム構成への適合性
     - 学習コストと導入の段階性
   - 調査結果を `design-pattern.md` の「OSSデザインパターン調査」セクションに記載する

3. **スケール時の問題洗い出し**
   - 「AI生成コードが陥りやすい問題」を先回りして検討:
     - 巨大コンポーネント化のリスク
     - 責務の曖昧な境界
     - テスト困難な密結合
     - 状態管理の複雑化
   - 既存アーキテクチャとの整合性を検証

4. **候補パターンの提案（1〜3個）**
   - ステップ2のOSS調査結果を踏まえ、このプロジェクトに適した候補パターンを提案する
   - OSSで実績のあるパターンを優先的に候補に含める（ただしプロジェクト規模に合わせて簡略化してもよい）
   - 各パターンに対して:
     - ディレクトリ構造の例
     - 責務分離表
     - 簡易コード例（`poc/[技術名]/src/` に追加）
     - メリット・デメリット
     - 参考にしたOSSプロジェクト名（該当する場合）
   - パターン間の比較表を作成

5. **design-pattern.md の作成**
   - テンプレート（`./templates/design-pattern.md`）を使用
   - `poc/[技術名]/design-pattern.md` に出力

### 成果物

- `poc/[技術名]/design-pattern.md`（完成）
- `poc/[技術名]/src/`（パターン検証コード追加）

### ユーザー承認ポイント（パターン選定）

```
「フェーズ3（設計パターン探索）が完了しました。

候補パターン:
1. [パターンA名] - [特徴を1行で]
2. [パターンB名] - [特徴を1行で]
3. [パターンC名] - [特徴を1行で]（該当する場合）

推奨: パターン[X]（[推奨理由を1行で]）

design-pattern.md を確認してください。
採用するパターンを選択いただけたら、フェーズ4（文書化）に進みます。」
```

## フェーズ4: 文書化と docs/ 統合

### 目的

PoC の結果と選定した設計パターンを、プロジェクトの永続ドキュメントに統合する。

### 手順

1. **findings.md の最終化**
   - 採用パターンの決定を反映
   - 最終的な推奨事項を記載

2. **docs/architecture.md の更新**
   - 採用した技術のセクションを追加
   - 選定した設計パターンを記載
   - 既存のアーキテクチャ記述との整合性を確認
   - **ファイルが存在しない場合**: `architecture-design` スキルのテンプレートを使用して新規作成

3. **docs/development-guidelines.md の更新**
   - 採用した技術に関するコーディング規約を追加
   - アンチパターン（やってはいけない実装）を記載
   - **ファイルが存在しない場合**: `development-guidelines` スキルのテンプレートを使用して新規作成

4. **その他の docs/ ファイル更新**（必要に応じて）
   - `docs/glossary.md`: 新しい技術用語の追加
   - `docs/repository-structure.md`: ディレクトリ構造の変更

### 成果物

- `docs/architecture.md`（更新）
- `docs/development-guidelines.md`（更新）
- その他必要な `docs/` ファイル

### ユーザー承認ポイント（1ファイルずつ）

各ファイルの更新後:

```
「[ファイル名] を更新しました。変更点を確認してください。

[変更内容のサマリを2-3行で記述]

承認いただけたら次のファイルに進みます。修正が必要な場合はお知らせください。」
```

**承認されるまで絶対に次のファイルに進まないこと。**

## steering 統合

### ステアリングディレクトリ

命名規則: `.steering/[YYYYMMDDHHmmss]-poc-[技術名]/`

例: `.steering/20260222143052-poc-tanstack-query/`

### tasklist.md の構成

tasklist.md は4フェーズ構成とし、各フェーズの最後に「ユーザー承認を得る」タスクを含める:

```markdown
## フェーズ1: 技術リサーチとPoC構築
- [ ] 公式ドキュメント調査
- [ ] overview.md の作成
- [ ] 最小検証コードの生成
- [ ] ★ ユーザー承認を得る

## フェーズ2: 技術検証
- [ ] 検証コードの実行と結果記録
- [ ] 評価基準に対する合否判定
- [ ] 有名OSSでの採用事例調査（最低2-3件）
- [ ] findings.md の作成
- [ ] ★ ユーザー承認を得る（採用/不採用判断）

## フェーズ3: 設計パターン探索
- [ ] 既存アーキテクチャ分析
- [ ] 有名OSSのデザインパターン深掘り調査と適用検討
- [ ] スケール時の問題洗い出し
- [ ] 候補パターンの提案（1-3個、OSS事例を踏まえて）
- [ ] design-pattern.md の作成
- [ ] ★ ユーザー承認を得る（パターン選定）

## フェーズ4: 文書化と docs/ 統合
- [ ] docs/architecture.md の更新
- [ ] ★ ユーザー承認を得る
- [ ] docs/development-guidelines.md の更新
- [ ] ★ ユーザー承認を得る
- [ ] その他 docs/ ファイルの更新（必要に応じて）
- [ ] ★ ユーザー承認を得る（該当する場合）
```

### steering スキルとの連携

- **計画時**: `Skill('steering')` モード1でステアリングファイルを作成
- **実装時**: `Skill('steering')` モード2でtasklist.mdの進捗管理
- **振り返り時**: `Skill('steering')` モード3で振り返り記録

---

## 不採用時の早期終了

フェーズ2でユーザーが「不採用」を承認した場合:

1. フェーズ3・4をスキップ
2. tasklist.md のフェーズ3・4のタスクを以下の形式で更新:
   ```markdown
   - [x] ~~タスク名~~（理由: フェーズ2で不採用判断のためスキップ）
   ```
3. 振り返り（ステップ8）に直行
4. 振り返りに不採用の理由と学びを記録
