# PoC ワークフロースキル

技術検証のためのPoC（概念実証）実装に関するガイドラインとテンプレート。

## ディレクトリ構造規約

PoCコードは `poc/{技術名}/` に格納する。本番コード（`src/`）とは完全に分離する。

```
poc/{技術名}/
├── overview.md          # PoC概要（テンプレートから生成）
├── src/                 # 検証コード
├── findings.md          # 検証結果
└── design-pattern.md    # 設計パターン（パターン設計Agent が作成）
```

### PoCコードの原則

| 原則 | 説明 |
|---|---|
| 使い捨て | 本番コードには含めない。検証完了後も参照用として保持する |
| 最小限 | 検証に必要な最小限のコードのみ記述する |
| 本番品質不要 | エラーハンドリング、テスト、リファクタリングは不要 |
| 動作優先 | コードの美しさより動作確認を優先する |

## テンプレート

### PoC概要（overview.md）

テンプレート: `./templates/poc-overview.md`

以下の情報を記載する:

- 基本情報（技術名、バージョン、ライセンス、メンテナンス状況）
- 検証目的（背景、解決したい課題）
- 評価基準（必須基準、加点基準）
- 検証スコープ（検証すること / しないこと）
- 技術概要と既存技術との比較
- セットアップ手順と実行方法

### 検証結果（findings.md）

テンプレート: `./templates/poc-findings.md`

以下の情報を記載する:

- 結論（採用推奨 / 不採用推奨 / 条件付き採用）
- 評価結果（必須基準・加点基準の合否とエビデンス）
- OSSでの採用事例（2〜3件。採用パターン、設計思想）
- 技術的発見（利点、制約、予想外の発見）
- 既存アーキテクチャとの親和性
- 導入コスト見積もり
- 推奨事項

### 設計パターン（design-pattern.md）

テンプレート: `./templates/design-pattern.md`

パターン設計Agentが使用する。以下の情報を記載する:

- 既存パターンとの整合性分析
- OSSデザインパターン調査
- 候補パターン（1〜3個。ディレクトリ構造、責務分離、コード例、メリット・デメリット）
- 比較表と推薦
- アンチパターン
- docs/ への反映案

## 評価基準の設計ガイド

### 必須基準

1つでも不合格なら技術を不採用とする基準。以下の観点で設定する:

- 既存技術スタックとの互換性
- ライセンスの制約
- パフォーマンス要件
- セキュリティ要件

### 加点基準

合格数が多いほど好ましい基準。以下の観点で設定する:

- 開発体験（DX）
- エコシステムの充実度
- 型安全性
- テスタビリティ
- ドキュメントの充実度

## 品質基準チェックリスト

- [ ] `poc/{技術名}/` ディレクトリ構造が規約に従っている
- [ ] overview.md が検証開始前に作成されている
- [ ] 評価基準（必須・加点）が検証前に定義されている
- [ ] 検証コードが最小限である（本番品質のコードを含まない）
- [ ] findings.md の判定にエビデンス（実行結果、数値）が紐付いている
- [ ] OSSでの採用事例が2件以上調査されている
- [ ] 不採用の場合でも理由と学びが記録されている
