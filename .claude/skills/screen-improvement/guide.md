# 変更仕様書作成ガイド

このガイドは、画面仕様書を起点に、UI/UXの改善内容をBefore/After変更仕様書として作成するための実践的な指針を提供します。

## 変更仕様書の目的

変更仕様書は、画面仕様書に対する変更を**Before/After形式で可視化**するドキュメントです。

**4つの役割**:

1. **可視化**: 何が変わるかをBefore/Afterで一目で把握できるようにする
2. **承認ゲート**: ユーザーが変更内容を確認・承認してから実装に進める
3. **実装ガイド**: 変更仕様書がそのまま実装の入力になる
4. **画面仕様書更新の入力**: Afterセクションの内容が画面仕様書の更新内容になる

## 影響範囲の特定（SKILL.md ステップ1に対応）

Before/Afterの作成に入る前に、変更の影響範囲を洗い出す。以下の4つの観点で整理する:

| 観点 | 確認内容 | 例 |
|------|----------|-----|
| **直接変更する画面** | ワイヤーフレーム・UI要素・操作挙動が変わる画面 | SCR-001のツールバーにボタン追加 |
| **間接的に影響する画面** | 遷移元・遷移先として関連する画面 | SCR-001から新画面への遷移が追加される場合のSCR-001 |
| **index.md** | 画面一覧・画面遷移図・PRD対応表の更新が必要か | 新画面追加時、遷移変更時 |
| **components.md** | 共通コンポーネント（`docs/screen-specification/components.md`）の変更が必要か | 複数画面で使われるUI部品の変更時 |

影響範囲が特定できたら、対象の画面仕様ファイルをすべて読み込んだ上で、Before/Afterの作成に進む。

## Beforeセクションの書き方

### 抜粋の原則

Beforeセクションは画面仕様書の**全体コピーではない**。変更に関連する部分のみを抜粋する。

**抜粋すべき範囲の判断基準**:

| 変更の種類 | 抜粋する範囲 |
|------------|-------------|
| UI要素の追加 | 追加箇所の周辺のワイヤーフレーム、既存のUI要素定義テーブル（番号の連続性確認用） |
| UI要素の変更 | 変更対象の要素を含むワイヤーフレーム、変更対象のUI要素定義行、変更対象の操作挙動 |
| UI要素の削除 | 削除対象を含むワイヤーフレーム、削除対象のUI要素定義行、削除対象に関連する操作挙動 |
| 操作フローの変更 | 変更対象の操作挙動仕様全体（正常系・異常系含む） |
| 画面遷移の変更 | 変更対象の操作挙動の「画面遷移」行 |

### ASCIIワイヤーフレームの注釈方法

変更対象箇所を明示するため、行末に `← ★変更対象` 注釈を付ける:

```
┌──────────────────────────────┐
│ ← 戻る    jiyucho      ✓ 保存│
├──────────────────────────────┤
│                              │
│  ┌──────────────────────┐   │
│  │ 何があった？          │   │
│  │                      │   │
│  └──────────────────────┘   │
│                              │
├──────────────────────────────┤
│  [写真] [カメラ] [マイク]    │  ← ★変更対象
├──────────────────────────────┤
│  [記録]        [タイムライン] │
└──────────────────────────────┘
```

### 出典の明記

Beforeセクションの冒頭に、抜粋元の画面仕様ファイルを必ず明記する。テンプレートでは以下の形式で記載されている:

```markdown
> 出典: `docs/screen-specification/scr-001-input.md`
```

この `> 出典:` 行は変更仕様書の**出力に含める内容**である（エージェント向け指示ではない）。実際のファイルパスに置き換えて出力すること。

## Afterセクションの書き方

### フォーマット統一の原則

Afterセクションは画面仕様書（`docs/screen-specification/scr-*.md`）と**同一フォーマット**で記述する。これにより:

- 承認後にAfterの内容をそのまま画面仕様書にマージできる
- フォーマット変換の手間とミスを防げる

### 変更箇所の注釈方法

Afterセクションでは、変更箇所に以下の注釈を付ける:

| 注釈 | 意味 | 使用場面 |
|------|------|----------|
| `← ★NEW` | 新規追加 | ワイヤーフレームの新しい要素、UI要素定義の新しい行、新しい操作挙動 |
| `← ★CHANGED` | 変更あり | ワイヤーフレームの変更箇所、UI要素定義の変更行、操作挙動の変更ステップ |

**ASCIIワイヤーフレームでの注釈例**:

```
┌──────────────────────────────┐
│ ← 戻る    jiyucho      ✓ 保存│
├──────────────────────────────┤
│                              │
│  ┌──────────────────────┐   │
│  │ 何があった？          │   │
│  │                      │   │
│  └──────────────────────┘   │
│                              │
│  ┌──────────────────────────┐│
│  │ 「保存しました」          ││  ← ★NEW（トースト通知）
│  └──────────────────────────┘│
│                              │
├──────────────────────────────┤
│  [写真] [カメラ] [マイク]    │
├──────────────────────────────┤
│  [記録]        [タイムライン] │
└──────────────────────────────┘
```

**UI要素定義テーブルでの注釈例**:

```markdown
| #   | 要素名       | 種類  | 説明                       | 初期値 | 制約・条件           |
| --- | ------------ | ----- | -------------------------- | ------ | -------------------- |
| 17  | 保存トースト | Toast | 「保存しました」、2秒後自動消去 | 非表示 | 保存成功時のみ表示 ← ★NEW |
```

**操作挙動仕様での注釈例**:

```markdown
**正常系の挙動**:

1. NetaService.createNeta() でネタを保存
2. ハプティックフィードバック（成功）が発生
3. 画面上部にトースト通知「保存しました」を2秒間表示  ← ★NEW
4. テキスト・写真・音声がすべてクリアされる
```

### ASCIIワイヤーフレームの原則

Afterセクションのワイヤーフレームは、画面仕様書と同じ原則に従う:

- **罫線文字を使用**: `┌ ┐ └ ┘ ├ ┤ ┬ ┴ ┼ ─ │` で枠線を描画
- **画面幅の統一**: 既存の画面仕様書と同じ幅で描画する
- **日本語は全角幅で計算**: 日本語1文字 = 半角2文字分
- **コメント**: ASCII図の右側に `←` でコメントを記述

詳細なASCIIワイヤーフレームの描画ルール（全角・半角の幅計算、レイアウトの再現方法等）については `.claude/skills/screen-specification/guide.md` の「ASCIIワイヤーフレーム」セクションを参照。

## 変更サマリの記入タイミング

変更サマリ（追加・変更・削除の箇条書き）は、Before/Afterの両セクションを作成した**後**に記入する。先にBefore/Afterを作成することで、変更内容を正確に把握した上でサマリを書ける。テンプレート上ではサマリがBefore/Afterの前に配置されているが、記入順序はBefore → After → サマリの順が推奨される。

## 変更の種類別ガイド

### パターン1: 小変更（UI要素の追加・変更・削除）

**特徴**: 1つの画面内の限定的な変更。ワイヤーフレーム・UI要素定義・操作挙動の一部が変わる。

**Beforeの書き方**:
- 変更対象のUI要素を含むワイヤーフレームを抜粋
- 変更対象のUI要素定義行を抜粋
- 変更対象に関連する操作挙動を抜粋

**Afterの書き方**:
- 変更後のワイヤーフレーム（変更箇所に `★NEW` / `★CHANGED` 注釈）
- 追加・変更するUI要素定義
- 追加・変更する操作挙動

**index.md への影響**: 通常「変更なし」
**components.md への影響**: 変更対象が複数画面で共有されるUI部品の場合は更新が必要（通常は影響なし）

### パターン2: 大変更（操作フローの大幅な変更）

**特徴**: 1つの画面の操作フローが大きく変わる。複数の操作挙動が連鎖的に変更される。

**Beforeの書き方**:
- 該当画面のワイヤーフレーム（変更に関連する状態すべて）
- 変更対象の全操作挙動を抜粋

**Afterの書き方**:
- 変更後のワイヤーフレーム（新しい状態が追加される場合はそれも含む）
- 変更後の全操作挙動（変更箇所に注釈）

**index.md への影響**: 画面遷移が変わる場合は画面遷移図・遷移一覧の更新が必要
**components.md への影響**: 変更対象が複数画面で共有されるUI部品の場合は更新が必要

### パターン3: 横断的変更（複数画面にまたがる変更）

**特徴**: 同一の変更が複数画面に適用される（例: 全画面でダークモード対応、共通ヘッダーの変更）。

**Beforeの書き方**:
- 各画面について、変更対象部分を個別に抜粋
- 画面ごとに `## SCR-[番号]: [画面名]` セクションを作成

**Afterの書き方**:
- 各画面について、変更後の状態を個別に記述
- 共通部分がある場合は最初の画面で詳細に記述し、他の画面では「SCR-001と同様」と参照可能

**index.md への影響**: 共通コンポーネントの変更がある場合は `components.md`（`docs/screen-specification/components.md`、複数画面で共通利用されるUI部品の仕様書）の更新も記載

### パターン4: 新画面追加（既存画面の改善に伴う派生）

**特徴**: 既存画面の改善に伴い、新しい画面が必要になるケース。

**新画面のBefore/After**:
- Beforeセクション: 「該当画面なし - 新規追加」と記載
- Afterセクション: `docs/screen-specification/` 配下の既存画面仕様書と同一フォーマットで、完全な画面仕様を記述する
- フォーマット参照先: `.claude/skills/screen-specification/templates/screen.md`
- 最低限含むべき要素:
  - ファイルヘッダー（仕様ファイルパス、対応PRD機能）
  - メタデータテーブル（作成日、最終更新日、参照ドキュメント）
  - 概要
  - ASCIIワイヤーフレーム（初期状態 + 必要な状態別）
  - Markdownワイヤーフレーム（UI要素定義）
  - 操作挙動仕様

**遷移元画面のBefore/After**（SKILL.md ステップ5に対応）:
- 新画面への遷移が追加される既存画面についても、通常のパターン（パターン1〜3）に従いBefore/Afterを記述する
- 遷移元画面の操作挙動に「画面遷移」の変更が含まれるため、該当する操作挙動をBefore/Afterで記述する

**index.md への影響**: 画面一覧への追加、画面遷移図の変更が必要
**components.md への影響**: 新画面が既存の共通UI部品を使用する場合、または新たな共通部品を導入する場合は更新が必要

## steeringファイルとの連携方法

変更仕様書は `.steering/` ディレクトリ内に配置され、steeringファイルと連携する:

```
.steering/[YYYYMMDDHHmmss]-improve-[名前]/
├── requirements.md       # 変更の要求内容（change-spec.mdへの参照を含む）
├── design.md             # 実装設計
├── tasklist.md           # タスクリスト（最終フェーズに「画面仕様書の更新」を含む）
└── change-spec.md        # 変更仕様書（Before/After）← このスキルの成果物
```

### requirements.md との関係

`requirements.md` には変更仕様書への参照を含める:

```markdown
## 変更仕様
詳細は `./change-spec.md` を参照。
```

### tasklist.md との関係

`tasklist.md` の最終フェーズに「画面仕様書の更新」タスクを必ず含める。これにより、実装完了後の画面仕様書更新が漏れなく実行される。

## 画面仕様書更新との連携方法

変更仕様書のAfterセクションは、画面仕様書の更新の**入力**になる:

1. **Afterセクションを参照**: 変更仕様書のAfter内容が画面仕様書の更新内容
2. **画面仕様書を編集**: Afterの内容を既存の画面仕様書にマージする
3. **注釈を除去**: `★NEW` / `★CHANGED` 注釈はマージ時に除去する
4. **最終更新日を更新**: 各ファイルのメタデータテーブルの最終更新日を更新する
5. **index.md の更新**: 変更仕様書の「index.md への影響」セクションに基づき更新する

### マージ時の注意点

- Afterセクションの内容は画面仕様書の**該当部分を置き換える**（追記ではない）
- UI要素定義テーブルの `#` 番号は、既存の画面仕様書の番号体系に合わせて調整する
- 新しい操作挙動は、画面仕様書の操作挙動セクションに適切な位置に挿入する

## まとめ

変更仕様書作成の成功のポイント:

1. **Beforeは関連部分のみ抜粋**: 全体コピーではなく変更箇所にフォーカス
2. **Afterは画面仕様書と同一フォーマット**: マージ可能な形で記述
3. **変更箇所に注釈**: `★NEW` / `★CHANGED` で変更点を一目で把握可能に
4. **影響範囲の網羅**: 直接変更する画面だけでなく、間接的な影響も記載
5. **index.md への影響を評価**: 画面一覧・画面遷移図・遷移一覧の更新要否を判断
