# 画面改善スキル（変更仕様書の作成）

このスキルは、画面仕様書（`docs/screen-specification/`）を起点に、UI/UXの改善・画面追加の内容を**Before/After変更仕様書**として定義します。

## スキルの目的

- 変更内容の**可視化**: Before/Afterで何が変わるかを一目で把握できるようにする
- **承認ゲート**の提供: ユーザーが変更内容を確認・承認してから実装に進む
- **実装ガイド**の提供: 変更仕様書がそのまま実装の入力になる
- **画面仕様書更新の入力**: Afterセクションが画面仕様書の更新内容になる

## いつ使うか

| シナリオ                              | このスキルを使う | `/add-feature` を使う |
| ------------------------------------- | :--------------: | :-------------------: |
| 既存画面のUI要素を追加・変更・削除    |        o         |                       |
| 既存画面の操作フローを変更            |        o         |                       |
| 複数画面にまたがるUI/UX改善           |        o         |                       |
| 既存画面の改善に伴い新画面を追加 ※1   |        o         |                       |
| PRDの新機能を実装（バックエンド含む） |                  |           o           |
| 独立した新画面の追加（PRD起点）       |                  |           o           |
| データモデルやビジネスロジックの追加  |                  |           o           |

※1: 既存画面の改善に伴い派生的に新画面が必要になるケース（例: 設定画面から詳細設定画面を分離）。

**判断基準**: 変更の起点が画面仕様書なら `/improve-screen`、PRDの機能要件なら `/add-feature`。

## 前提条件

変更仕様書の作成を開始する前に、以下が完了している必要があります:

1. **`docs/screen-specification/index.md` が存在すること**（必須）
   - 画面一覧・画面遷移図が定義されている
2. **変更対象の画面仕様ファイルが存在すること**（既存画面の改善の場合）
   - `docs/screen-specification/scr-[番号]-[name].md`
3. **変更内容がユーザーから指示されていること**
   - 自然言語による変更指示（例: 「保存後にトースト通知を追加」）

## テンプレートの参照

変更仕様書を作成する際は、以下のテンプレートを使用してください:

- **変更仕様書**: `./templates/change-spec.md`

### テンプレートの読み方

- `[...]` プレースホルダー: 具体的な内容に置き換えて出力する
- `> ` ブロック引用: 作成時の指示・判断基準。生成した変更仕様書には含めない

## 変更仕様書の作成プロセス

### ステップ1: 影響範囲の特定

変更内容を分析し、影響を受けるファイルを洗い出す:

1. `docs/screen-specification/index.md` を読み込み、全体構造を把握する
2. 変更内容に関連する画面仕様ファイルを特定し、読み込む
3. 以下の観点で影響範囲を整理する:
   - **直接変更する画面**: ワイヤーフレーム・UI要素・操作挙動が変わる画面
   - **間接的に影響する画面**: 遷移元・遷移先として関連する画面
   - **index.md**: 画面一覧・画面遷移図・PRD対応表の更新要否
   - **components.md**: 共通コンポーネントの変更要否

### ステップ2: Beforeセクションの作成

既存の画面仕様書から、**変更に関連する部分のみ**を抜粋する:

1. 変更対象のASCIIワイヤーフレームを抜粋する（変更箇所に `← ★変更対象` 注釈を付ける）
2. 変更対象のUI要素定義を抜粋する（変更・削除する要素のみ）
3. 変更対象の操作挙動仕様を抜粋する（変更・削除する操作のみ）

**重要**: 画面仕様書の全体をコピーしない。変更に関連する部分のみを抜粋し、変更箇所が明確になるようにする。

### ステップ3: Afterセクションの作成

変更後の状態を、**画面仕様書と同一のフォーマット**で記述する:

1. 変更後のASCIIワイヤーフレームを作成する（変更箇所に `← ★NEW` または `← ★CHANGED` 注釈を付ける）
2. 変更後のUI要素定義を作成する（追加・変更する要素）
3. 変更後の操作挙動仕様を作成する（追加・変更する操作）

**重要**: Afterセクションは画面仕様書（`docs/screen-specification/scr-*.md`）と同一フォーマットで記述する。これにより、承認後にAfterの内容をそのまま画面仕様書にマージできる。

### ステップ4: 変更サマリの作成

各画面について、変更内容を簡潔にまとめる:

- **追加**: 新たに加える要素・操作
- **変更**: 既存の要素・操作の変更点
- **削除**: 除去する要素・操作

### ステップ5: 新画面の場合の特別処理

変更に新しい画面の追加が含まれる場合:

1. Beforeセクション: 「該当画面なし - 新規追加」と記載
2. Afterセクション: `docs/screen-specification/` 配下の既存画面仕様書と同一フォーマットで完全な画面仕様を記述する（フォーマット参照先: `.claude/skills/screen-specification/templates/screen.md`）
3. index.md への影響セクション: 画面一覧への追加、画面遷移図の変更を記載

## 品質チェックリスト

変更仕様書の作成完了後、以下を確認する:

### 網羅性

- [ ] 変更内容に関連するすべての画面が変更仕様書に含まれているか
- [ ] Before/Afterの両方にASCIIワイヤーフレーム・UI要素定義・操作挙動が記述されているか
- [ ] index.md への影響が評価されているか

### 正確性

- [ ] Beforeセクションが現在の画面仕様書の内容と一致しているか
- [ ] Afterセクションがユーザーの変更指示を正確に反映しているか
- [ ] 変更箇所に `★NEW` / `★CHANGED` 注釈が付いているか

### 実装可能性

- [ ] Afterセクションの記述が実装者にとって曖昧さなく理解できるか
- [ ] 具体的な数値（サイズ、時間、文字数等）が定義されているか
- [ ] 技術的に実現不可能な要件が含まれていないか

### 整合性

- [ ] Afterセクションが画面仕様書と同一フォーマットで記述されているか
- [ ] ASCIIワイヤーフレームの原則（全角・半角考慮、レイアウトの忠実な再現）が守られているか
- [ ] 画面遷移の変更がindex.mdへの影響セクションに反映されているか
- [ ] 他の画面との遷移関係に矛盾がないか
