# AI向けドキュメントレビュー

`.claude/` 配下のAI向け設定ドキュメント（SKILL.md, agents/*.md, commands/*.md）をレビューし、改善点を報告する。

## 対象ドキュメント

`.claude/` 配下のAI向け設定ドキュメント（SKILL.md, agents/*.md, commands/*.md）をレビュー対象とする。

## スコープ外

- 対象ファイルの修正（レビュー結果の報告のみ行い、ファイルは変更しない）
- `.claude/` 外の一般ドキュメント（docs/ 配下など）のレビュー

## 引数の解釈

ユーザーが `/review-ai-doc` の後に指定した引数を以下のルールで解釈する。

| 引数 | 動作 |
|---|---|
| ファイルパス（例: `.claude/agents/foo.md`） | 指定ファイル1つをレビューする |
| フォルダパス（例: `.claude/skills/`） | フォルダ直下および再帰的に `*.md` を収集し、すべてレビューする |
| `unstaged` | `git diff --name-only` で未ステージの変更ファイルのうち `.md` ファイルをレビューする |
| `staged` | `git diff --cached --name-only` でステージ済みの `.md` ファイルをレビューする |
| 引数なし | エラー。レビュー対象を指定するようユーザーに案内する |

### エラー時の動作

| ケース | 動作 |
|---|---|
| 指定パスが存在しない | 「指定されたパスが見つかりません: {パス}」と報告し終了する |
| 対象の `.md` ファイルが0件 | 「レビュー対象の .md ファイルが見つかりませんでした」と報告し終了する |

## レビュー手順

### Step 1: 対象ファイルの収集

引数に応じてレビュー対象のファイル一覧を確定する。対象外の拡張子（`.md` 以外）はスキップし、スキップした旨を報告する。

### Step 2: 各ファイルのレビュー

ファイルごとに以下の7つの観点でレビューする。各観点について **OK / WARN / NG** の3段階で判定する。

#### 判定基準（全観点共通）

| 判定 | 基準 |
|---|---|
| OK | 問題なし、または文脈上一意に解釈可能 |
| WARN | 軽微な問題が1-2箇所あるが、致命的ではない |
| NG | 問題が3箇所以上、またはAIの動作に影響する致命的な問題がある |

#### 1. 指示の明瞭性

AIが指示を一意に解釈できるか。

- 「適切に」「必要に応じて」「なるべく」等の曖昧な修飾語が、具体的な条件や基準なしに使われていないか
- 「〜など」「〜等」で終わる列挙が、AIに網羅的な推測を強いていないか
- 条件分岐（「〜の場合は」）の分岐先がすべて明記されているか（暗黙のelseがないか）
- 主語・目的語の省略によって、動作主体や対象が曖昧になっていないか

#### 2. 構造の一貫性

ドキュメントの構造が予測可能で、AIがセクションの役割を正しく認識できるか。

- 見出しレベル（`#`, `##`, `###`）が階層構造として正しいか（レベル飛びがないか）
- 同じ粒度の情報が同じ見出しレベルで統一されているか
- リスト・テーブル・コードブロックの使い分けが情報の性質に合っているか
- フロントマター（`---`で囲まれたYAML）が正しい形式か（SKILL.mdの場合）

#### 3. コンテキスト効率

トークン消費を抑えつつ、必要な情報を漏れなく伝えているか。

- 同じ情報が複数箇所で重複して記載されていないか
- 冗長な説明文を、テーブルやリストで簡潔に置き換えられないか
- AIが処理する必要のない補足情報（人間向けの背景説明）が混入していないか
- ファイルの総行数が目的に対して過大でないか（目安: SKILL.md は200行以内、agents/*.md は150行以内）

#### 4. 実行可能性

AIが記載された手順を再現可能か。

- 手順が番号付きリストで順序明示されているか
- 各ステップの入力と出力（期待結果）が明確か
- 外部ファイルへの参照パスが正しく、存在するか（Globで検証する）
- ツール名やコマンド名が正確か

#### 5. スコープの明確性

ドキュメントの適用範囲と責務が明確か。

- ドキュメントの冒頭で、何を目的としたドキュメントかが1-2文で説明されているか
- 「このスキル/エージェントがやること」と「やらないこと」の境界が分かるか
- 前提条件（必要な入力、事前状態）が明記されているか

#### 6. 矛盾の検出

ドキュメント内で矛盾する記述がないか。

- 同じ概念に対して異なる名称が使われていないか（表記ゆれ）
- あるセクションの指示が別のセクションの指示と矛盾していないか
- 条件分岐のケースが重複・競合していないか

#### 7. エラーハンドリング

想定外のケースへの対処が記載されているか。

- 入力が不正・不足した場合の動作が定義されているか
- 外部ファイルが存在しない場合のフォールバックが記載されているか
- ワークフローの途中で失敗した場合のリカバリ手順があるか

### Step 3: レビュー結果の出力

以下のフォーマットで結果を出力する。

```
## レビュー結果: {ファイルパス}

| 観点 | 判定 | 概要 |
|---|---|---|
| 指示の明瞭性 | OK/WARN/NG | 一行サマリ |
| 構造の一貫性 | OK/WARN/NG | 一行サマリ |
| コンテキスト効率 | OK/WARN/NG | 一行サマリ |
| 実行可能性 | OK/WARN/NG | 一行サマリ |
| スコープの明確性 | OK/WARN/NG | 一行サマリ |
| 矛盾の検出 | OK/WARN/NG | 一行サマリ |
| エラーハンドリング | OK/WARN/NG | 一行サマリ |

### 総合判定: PASS / NEEDS_IMPROVEMENT / FAIL

### 指摘事項

（WARN/NG の項目について、以下を記載する）

#### {観点名}: {判定}

- **箇所**: 該当する行番号または見出し
- **問題**: 何が問題か
- **改善案**: 具体的な修正提案（可能な場合、修正後の文面を提示する）
```

**総合判定の基準:**

- **PASS**: すべてOK、またはWARNが2個以下でNGがゼロ
- **NEEDS_IMPROVEMENT**: WARNが3個以上、またはNGが1個
- **FAIL**: NGが2個以上

### Step 4: 複数ファイルの場合

複数ファイルをレビューした場合、個別結果の後に全体サマリを出力する。

```
## 全体サマリ

| ファイル | 総合判定 | NG数 | WARN数 |
|---|---|---|---|
| {パス} | PASS/NEEDS_IMPROVEMENT/FAIL | N | N |
| ... | ... | ... | ... |

### 共通して見られる傾向

（複数ファイルに共通する問題がある場合に記載する）
```
