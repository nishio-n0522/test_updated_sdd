# 開発ガイドラインスキル

開発ガイドラインを作成・参照・レビューする際のドメイン知識。テンプレートと品質基準を定義する。

## スコープ

- **提供するもの**: 開発ガイドラインのテンプレート、品質基準
- **提供しないもの**: 作成手順（Agent が定義）、プロジェクト固有の規約

## カバー範囲

開発ガイドラインは以下の2つの要素をカバーする:

| 要素 | 内容 | 配置先 |
|---|---|---|
| 共通プロセス | Git運用ルール、コミットメッセージ規約、PR/レビュープロセス、コードレビュー基準 | index.md |
| コーディング規約 | 命名規則、型定義、エラーハンドリング、コメント規約、テスト戦略 | {stack}.md |

## 入出力

| 項目 | 内容 |
|---|---|
| 入力 | `docs/architecture/` + `docs/repository-structure/` |
| 出力 | `docs/development-guidelines/` |

## ファイル構成

```
docs/development-guidelines/
├── index.md              # 共通ガイドライン（Git運用、コードレビュー、PR等）
└── {stack}.md            # スタック固有のコーディング規約（1スタック = 1ファイル）
```

### 概要（index.md）の役割

- 全スタック共通の開発プロセスを記載（Git運用、コミット規約、PR、レビュー）
- スタック一覧と各スタックの規約ファイルへの参照を記載
- Agentはindex.mdを参照し、タスクに該当するスタックのみを読み込む

### 個別スタックの命名規則

```
{stack}.md
```

`stack` はスタックを表す kebab-case の名前（例: `frontend.md`, `backend.md`, `infrastructure.md`）

## 読み込み戦略

開発ガイドラインは**選択的読み込み**で使用する。

```
実装タスク開始
  └─ development-guidelines/index.md を参照
       └─ タスクに該当するスタックのみ読み込み
```

## 既存ドキュメントの優先順位

| 優先度 | 対象 | 説明 |
|---|---|---|
| 1（最優先） | `docs/development-guidelines/` | プロジェクト固有の規約。このスキルのガイドより優先 |
| 2（参考） | このスキル | 汎用テンプレート。既存ガイドラインがない場合に使用 |

## テンプレート

開発ガイドラインを作成する際は次のテンプレートを使用する: ./template.md

テンプレートには index.md 用と個別スタック用の2種類が含まれる。

## 品質基準

### 概要（index.md）のチェック項目

- [ ] ブランチ戦略が決まっているか
- [ ] コミットメッセージ規約が明確か（Conventional Commits等）
- [ ] コードレビュープロセスが定義されているか
- [ ] PRテンプレートが定義されているか
- [ ] 全スタックが一覧に含まれ、詳細ファイルへの参照があるか

### 個別スタックのチェック項目

- [ ] コーディング規約が具体例付きで定義されているか
- [ ] 命名規則が明確か（変数、関数、クラス、ファイル）
- [ ] エラーハンドリング方針が定義されているか
- [ ] テストの種類（単体・統合・E2E）とカバレッジ目標が設定されているか
- [ ] 良い例・悪い例のコード例が含まれているか
