# アイデア壁打ちスキル

brainstormオーケストレーターが参照する知識スキル。対話的なアイデア壁打ちの方法論と質問フレームワークを定義する。

## 2つのモード

### モードA: 新規プロダクト構想

新規プロダクトのアイデアを壁打ちする際のフェーズ構成。

| フェーズ | 目的 | 主な問い |
|---|---|---|
| 1. 課題発見 | 解くべき課題を特定する | 誰が、どんな場面で、何に困っているか？ |
| 2. 解決策探索 | 課題に対する解決策を広げる | どうすれば課題を解決できるか？他のアプローチは？ |
| 3. ユーザー定義 | ターゲットユーザーを具体化する | 最も恩恵を受けるのは誰か？現在どう対処しているか？ |
| 4. スコープ定義 | MVP範囲を決める | 最小限必要な機能は？後回しにできるものは？ |
| 5. 実現可能性 | リスクと前提を洗い出す | 技術的に可能か？検証すべき仮説は？ |

### モードB: 既存プロダクト機能構想

既存プロダクトに機能を追加・改善する際のフェーズ構成。

| フェーズ | 目的 | 主な問い |
|---|---|---|
| 1. 現状理解 | 既存プロダクトの状況を確認する | 現在の主要機能は？ユーザーの使い方は？ |
| 2. 改善機会 | 改善・追加すべき点を特定する | ユーザーの不満・要望は？競合との差は？ |
| 3. インパクト評価 | 改善のインパクトを見積もる | 影響を受けるユーザー数は？改善幅は？ |
| 4. スコープ定義 | 実装範囲を決める | 既存機能への影響は？段階的に導入できるか？ |

## 質問フレームワーク

壁打ち中に使用する質問の6カテゴリ。フェーズに応じて適切なカテゴリの質問を選択する。

### 1. 明確化（Clarification）

曖昧な発言を具体化する。

- 「○○とは具体的にどういうことですか？」
- 「○○の例を挙げていただけますか？」
- 「○○と△△の違いは何ですか？」

### 2. 挑戦（Challenge）

前提や思い込みを揺さぶる。

- 「本当にそれは課題ですか？もしそうでなかったら？」
- 「それはユーザーの声ですか、それとも想像ですか？」
- 「逆のアプローチ（○○しない）を取ったらどうなりますか？」

### 3. 拡張（Expansion）

思考の範囲を広げる。

- 「他にどんな解決策が考えられますか？」
- 「この機能がなくても課題は解決できますか？」
- 「全く別の業界ではこの課題をどう解決していますか？」

### 4. 制約（Constraint）

現実的な制約を確認する。

- 「技術的な制約はありますか？」
- 「時間・予算の制約は？」
- 「既存システムとの互換性は必要ですか？」

### 5. ユーザー視点（User Perspective）

ユーザーの立場で考える。

- 「ユーザーはこの機能をどのタイミングで使いますか？」
- 「ユーザーが最初にやりたいことは何ですか？」
- 「ユーザーはこの機能が無い場合、どうしていますか？」

### 6. 優先順位（Prioritization）

重要度・緊急度を整理する。

- 「この中で最も重要なものはどれですか？」
- 「これがなくてもプロダクトは成立しますか？」
- 「まず1つだけ作るとしたら、何を選びますか？」

## 対話のアンチパターン

壁打ちでやってはいけないこと。

| アンチパターン | 問題 | 代わりにやるべきこと |
|---|---|---|
| 解決策への飛びつき | 課題を十分に理解する前に実装方針を決める | まず課題を深掘りし、複数の解決策を出してから絞る |
| Yes/No 質問の連発 | ユーザーの思考を引き出せない | オープンクエスチョン（「どのように」「なぜ」）を使う |
| 一度に大量の質問 | ユーザーが圧倒される | 1ターンにつき質問は2-3個に絞る |
| ユーザーの発言の鵜呑み | 表面的な要望にとどまる | 「なぜそう思いますか？」で背景を掘り下げる |
| 自分のアイデアの押しつけ | ユーザーのオーナーシップが失われる | ユーザーのアイデアを発展させる形で提案する |
| 抽象的なまま進行 | 具体性のないアイデアメモが生成される | 具体例やシナリオで常に地に足をつける |

## アイデアメモの品質基準

生成するアイデアメモが満たすべき条件。

- [ ] **課題が具体的**: 誰が、どんな場面で、何に困っているかが明確
- [ ] **ターゲットが特定可能**: 具体的なユーザー像が浮かぶ
- [ ] **解決策が課題に紐付いている**: 各機能が具体的な課題を解決している
- [ ] **MVPが絞られている**: 最小限の機能セットが定義されている
- [ ] **仮説が明示されている**: 検証が必要な前提が列挙されている
- [ ] **対話で得られた洞察が反映されている**: テンプレートを埋めるだけでなく、壁打ちで深掘りした内容が含まれている
