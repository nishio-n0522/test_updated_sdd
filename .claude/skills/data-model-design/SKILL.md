# データモデル設計スキル

データモデル設計書を作成・レビューする際に参照するドメイン知識。テンプレートと品質基準を定義する。

## スコープ

- **提供するもの**: データモデル設計書のテンプレート、品質基準、記述ガイドライン
- **提供しないもの**: 作成手順（Agent が定義）、プロジェクト固有の内容

## 入出力

| 項目 | WF2: 新規作成 | WF4: 既存変更 |
|---|---|---|
| 入力 | PRD + 機能設計書 | 要件 + 既存 `docs/data-model.md` |
| 出力 | `docs/data-model.md` | データモデル変更仕様 |

> WF2（新規作成）: プロジェクト立ち上げ時にドキュメントを新規作成するワークフロー
> WF4（既存変更）: 既存ドキュメントに対して変更・追記するワークフロー

## 責務の定義

データモデル設計書は「機能設計書のドメインモデル（概念レベル）を、実装可能なデータ構造に詳細化する」ドキュメントである。

**答える問い**: 各エンティティの具体的な構造は？

### 記載する内容

- 全エンティティのフィールド定義（名前・型・制約・説明）
- インターフェース / 型定義（実装言語に合わせた表現）
- ER図（物理設計レベル: 全フィールドを含む）
- エンティティ間の関係（カーディナリティ、外部キー）
- データバリデーションルール
- シリアライズ形式の構造（該当する場合: JSON スキーマ等）

### 記載しない内容

- 機能分解・ユーザーフロー・画面構成（functional-design.md の責務）
- 永続化方式の選定（どのストレージに保存するか）（architecture.md の責務）
- バックアップ・リカバリ戦略（architecture.md の責務）
- 各画面のUI仕様・操作挙動（screen-specification の責務）
- テストケース（test-specification の責務）

## 他ドキュメントとの関係

| ドキュメント | 関係 |
|---|---|
| 機能設計書 | 上流。ドメインモデル（概念レベル）を詳細化したものが本ドキュメント |
| アーキテクチャ設計書 | 並行。永続化方式は architecture.md が担い、データ構造は本ドキュメントが担う |
| 画面仕様書 | 並行。画面が扱うデータの構造定義を本ドキュメントが提供する |
| テスト仕様書 | 下流。本ドキュメントのバリデーションルール等がテスト観点となる |

## 既存ドキュメントの優先順位

| 優先度 | 対象 | 説明 |
|---|---|---|
| 1（最優先） | `docs/data-model.md` | プロジェクト固有の設計。このスキルのガイドより優先 |
| 2（参考） | このスキル | 汎用テンプレート。既存設計書がない場合に使用 |

## 品質基準

### 必須チェック項目

- [ ] 機能設計書のドメインモデルに存在する全エンティティがカバーされているか
- [ ] 各エンティティのフィールドに型・制約・説明が定義されているか
- [ ] ER図が物理設計レベル（全フィールド含む）で記載されているか
- [ ] エンティティ間の関係（カーディナリティ・外部キー）が明確か
- [ ] バリデーションルールが具体的な数値・条件で定義されているか
- [ ] インターフェース定義がプロジェクトの技術スタックに合った言語で記述されているか
- [ ] 責務外の内容（永続化方式、UI仕様、テストケース等）が混入していないか

### 記述ガイドライン

- フィールドの制約は具体的に書く（「適切な長さ」ではなく「最大100文字」）
- インターフェース定義はプロジェクトの技術スタック（architecture.md）に合わせる
- ER図はMermaid記法で記述する
- エンティティの並び順は、依存関係の上流（他に依存しないもの）から下流へ
- 機能設計書のドメインモデルとの対応が追跡できるようにする
