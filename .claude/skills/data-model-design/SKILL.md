# データモデル設計スキル

データモデル設計書を作成・レビューする際に参照するドメイン知識。テンプレートと品質基準を定義する。

## スコープ

- **提供するもの**: データモデル設計書のテンプレート、品質基準、記述ガイドライン
- **提供しないもの**: 作成手順（Agent が定義）、プロジェクト固有の内容

## 入出力

| 項目 | WF2: 新規作成 | WF4: 既存変更 |
|---|---|---|
| 入力 | PRD + 機能設計書 + 画面仕様書 | 要件 + 既存 `docs/data-model/` |
| 出力 | `docs/data-model/` | データモデル変更仕様 |

> WF2（新規作成）: プロジェクト立ち上げ時にドキュメントを新規作成するワークフロー
> WF4（既存変更）: 既存ドキュメントに対して変更・追記するワークフロー

## 責務の定義

データモデル設計書は「機能設計書のドメインモデル（概念レベル）と画面仕様書のUX要件を統合し、実装可能なデータ構造に詳細化する」ドキュメントである。

**答える問い**: 各エンティティの具体的な構造は？

### 記載する内容

- 全エンティティのフィールド定義（名前・型・制約・説明）
- インターフェース / 型定義（実装言語に合わせた表現）
- ER図（物理設計レベル: 全フィールドを含む）
- エンティティ間の関係（カーディナリティ、外部キー）
- データバリデーションルール（正規のルール定義。画面仕様書・実装コードはここを参照する）
- データ交換形式の構造（該当する場合: エクスポート/インポート用JSON スキーマ等）

### 記載しない内容

- 機能分解・ユーザーフロー・画面構成（functional-design.md の責務）
- 永続化方式の選定・テーブル定義・インデックス設計（architecture の責務）
- バックアップ・リカバリ戦略（architecture の責務）
- 各画面のUI仕様・操作挙動・エラー表示方法（screen-specification の責務）
- テストケース（test-specification の責務）

## 他ドキュメントとの関係

| ドキュメント | 関係 | 責務境界 |
|---|---|---|
| 機能設計書 | 上流。概念レベルのドメインモデルを詳細化 | 機能設計 = エンティティの存在と関係性。データモデル = フィールド定義・型・制約 |
| 画面仕様書 | 上流。UX要件からデータ要件を抽出 | 画面仕様 = UI操作・表示・エラー表示方法。データモデル = バリデーション制約の正規定義 |
| アーキテクチャ設計書 | 下流。データ構造を永続化方式に落とし込む | データモデル = 論理的なデータ構造。アーキテクチャ = 物理的な永続化（テーブル設計・ストレージ選定） |
| テスト仕様書 | 下流。バリデーションルール等がテスト観点となる | — |

### バリデーションルールの所有権

データモデル設計書がバリデーションルールの **Single Source of Truth** である。

- **データモデル設計書**: ルールの正規定義（例: 「name: 最大200文字、空文字不可」）
- **画面仕様書**: UIでの表現方法（例: 「200文字超過時、入力欄下部に赤字でエラー表示」）。制約値はデータモデルを参照し、重複定義しない
- **アーキテクチャ設計書**: 永続化層での制約実装方法（例: 「NOT NULL, VARCHAR(200)」）。データモデルの制約に対応する

### データ交換形式 vs 永続化方式

| 関心事 | 担当ドキュメント | 例 |
|---|---|---|
| データ交換形式（エクスポート/インポート） | データモデル設計書 | JSON構造、ファイル拡張子、エンコーディング |
| 永続化方式（ストレージ） | アーキテクチャ設計書 | SQLiteテーブル設計、IndexedDB構造、ファイルシステム配置 |
| 永続化のマイグレーション・バックアップ | アーキテクチャ設計書 | マイグレーション戦略、バックアップ頻度 |

## 配置形式

エンティティ数の増加に対応するため、ディレクトリ分割形式を採用する。

```
docs/data-model/
├── index.md          # エンティティ一覧・ER図（全体）・バリデーション概要・対応確認表
└── {layer}.md        # 層またはドメインごとの詳細エンティティ定義
```

### 分割の指針

- **層で分割**: グラフモデル層 / プロセッサ定義層 のように関心事が明確に分離される場合
- **ドメインで分割**: 認証 / 注文 / 在庫 のようにビジネスドメインで分離される場合
- 分割軸はプロジェクトのドメイン構造に依存する。エージェントが機能設計書の構造から判断する

### ファイルサイズ

- 各ファイル: 推奨300行以下、上限400行
- index.md には全体のER図・エンティティ一覧・対応確認表を配置し、個別の詳細定義は層/ドメイン別ファイルに記載

## 既存ドキュメントの優先順位

| 優先度 | 対象 | 説明 |
|---|---|---|

## 品質基準

### 必須チェック項目

- [ ] 機能設計書のドメインモデルに存在する全エンティティがカバーされているか
- [ ] 画面仕様書のUI要件から導出されるデータ要件が反映されているか
- [ ] 各エンティティのフィールドに型・制約・説明が定義されているか
- [ ] ER図が物理設計レベル（全フィールド含む）で記載されているか
- [ ] エンティティ間の関係（カーディナリティ・外部キー）が明確か
- [ ] バリデーションルールが具体的な数値・条件で定義されているか
- [ ] インターフェース定義がプロジェクトの技術スタックに合った言語で記述されているか
- [ ] 責務外の内容（永続化方式・テーブル定義、UI仕様、テストケース等）が混入していないか
- [ ] index.md + 層/ドメイン別ファイルの分割構成になっているか

### 記述ガイドライン

- フィールドの制約は具体的に書く（「適切な長さ」ではなく「最大100文字」）
- インターフェース定義はプロジェクトの技術スタック（architecture.md）に合わせる
- ER図はMermaid記法で記述する
- エンティティの並び順は、依存関係の上流（他に依存しないもの）から下流へ
- 機能設計書のドメインモデルとの対応が追跡できるようにする
- バリデーションルールは画面仕様書でのUI表現と矛盾しないこと
