# アーキテクチャレビュースキル

差分コードがプロジェクトの技術スタック・設計方針に準拠しているかをレビューするためのチェックリストと基準です。

## 参照ドキュメント

レビュー前に必ず以下を読み込むこと:

- `docs/architecture.md` — アーキテクチャ設計書
- `docs/repository-structure.md` — リポジトリ構造定義書
- `docs/design-patterns/` — 技術スタック固有のデザインパターン（存在する場合）
- `{steering_dir}/design.md` — 今回の設計書

## レビュー対象範囲

レビュー対象は `git diff main...HEAD` で取得される差分ファイルのみです。差分に含まれないファイルはレビュー対象外です。

## チェックリスト

### 1. レイヤー構造

- [ ] 定義されたレイヤー構造（presentation / application / domain / infrastructure 等）に従っているか
- [ ] 各レイヤーの責務が守られているか（UIロジックがドメイン層に入っていない等）
- [ ] Server Components / Client Components の分離が適切か（フロントエンドの場合）
- [ ] ページ / フィーチャー / 共有 の区別が明確か

### 2. 依存関係の方向性

- [ ] 上位レイヤーから下位レイヤーへの一方向依存になっているか
- [ ] 循環依存が発生していないか
- [ ] ドメイン層がインフラ層に直接依存していないか
- [ ] 依存性逆転の原則が守られているか（該当する場合）

### 3. モジュール境界・インポートパス

- [ ] フィーチャー間の依存が公開API（index.ts等）経由のみか
- [ ] 内部実装への直接参照がないか
- [ ] パスエイリアスが正しく使用されているか（@/ 等）
- [ ] 共有モジュールの配置が適切か（shared/ 等）
- [ ] 禁止されたインポートパスが使用されていないか

### 4. ディレクトリ構造

- [ ] 新規ファイルがrepository-structure.mdの定義に従った場所に配置されているか
- [ ] フィーチャーディレクトリ内のサブディレクトリ構成が既存パターンと一致しているか
- [ ] テストファイルの配置がプロジェクト規約に従っているか
- [ ] 設定ファイル・型定義ファイルの配置が適切か

### 5. 命名規則（アーキテクチャレベル）

- [ ] フィーチャー名の命名がプロジェクト規約に従っているか
- [ ] ファイル名の命名パターンが既存と一致しているか
- [ ] API層の関数名規則（動詞プレフィックス等）が守られているか
- [ ] ストア・フック・クエリキーの命名規則が守られているか（該当する場合）

### 6. design.mdとの整合性

- [ ] design.mdで定義されたコンポーネント構成が実装されているか
- [ ] design.mdで定義されたデータフローが守られているか
- [ ] design.mdで定義されたAPIインターフェースが実装通りか
- [ ] design.mdの実装順序に概ね沿っているか

### 7. データフロー・状態管理

- [ ] 状態管理パターンがプロジェクトの規約に従っているか（Redux, Zustand, Context等）
- [ ] データフェッチングパターンが規約に従っているか（SWR, TanStack Query等）
- [ ] グローバル状態とローカル状態の使い分けが適切か
- [ ] Props drilling が深すぎないか

### 8. API設計（バックエンドの場合）

- [ ] RESTful設計原則に従っているか（該当する場合）
- [ ] エンドポイントの命名が規約に従っているか
- [ ] リクエスト/レスポンスの型定義が適切か
- [ ] エラーレスポンスの形式が統一されているか

## 指摘の分類基準

### MUST（修正必須）

以下に該当する場合はMUSTとして指摘する:

- **レイヤー違反**: 定義されたレイヤー構造への違反
- **依存方向違反**: 下位レイヤーから上位レイヤーへの依存、循環依存
- **design.mdとの不整合**: 設計書と実装の重大な不一致
- **モジュール境界違反**: フィーチャーの内部実装への直接参照
- **禁止パスの使用**: 禁止されたインポートパスの使用

### SHOULD（修正推奨）

以下に該当する場合はSHOULDとして指摘する:

- 命名規則の軽微な逸脱
- ディレクトリ構造の軽微な改善提案
- より適切なパターンの提案
- design.mdとの軽微な相違

### MAY（検討事項）

以下に該当する場合はMAYとして指摘する:

- 将来的なリファクタリング提案
- アーキテクチャの発展的な改善
- パフォーマンス最適化のためのアーキテクチャ変更

## 出力フォーマット

レビュー結果は `.claude/skills/steering/templates/review-architecture.md` のテンプレートに従って出力すること。

全ての指摘に「根拠」フィールドを必ず含め、`docs/` 内のどのドキュメントのどの記述に基づくかを明記すること。

**総合判定ルール**: MUST指摘が1件でもあれば FAIL、0件なら PASS
