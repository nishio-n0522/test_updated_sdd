# アーキテクチャレビュースキル

差分コードがプロジェクトの技術スタック・設計方針に準拠しているかをレビューするためのチェックリストと基準。

## スコープ

- **提供するもの**: アーキテクチャレビューのチェックリスト、分類基準、出力フォーマット
- **提供しないもの**: レビュー手順（Agent が定義）、プロジェクト固有のアーキテクチャ定義（`docs/architecture.md` が定義）

## 参照ドキュメント

レビュー前に必ず以下を読み込むこと:

- `docs/architecture.md` — アーキテクチャ設計書
- `docs/repository-structure.md` — リポジトリ構造定義書
- `docs/design-patterns/` — デザインパターン（存在する場合）
- `.issue/{N}/design.md` — 今回の設計書

## レビュー対象範囲

`git diff main...HEAD` で取得される差分ファイルのみ。差分に含まれないファイルはレビュー対象外。

## チェックリスト

### 1. レイヤー構造

- [ ] 定義されたレイヤー構造に従っているか
- [ ] 各レイヤーの責務が守られているか（UIロジックがドメイン層に入っていない等）
- [ ] Server Components / Client Components の分離が適切か（フロントエンドの場合）

### 2. 依存関係の方向性

- [ ] 上位レイヤーから下位レイヤーへの一方向依存になっているか
- [ ] 循環依存が発生していないか
- [ ] ドメイン層がインフラ層に直接依存していないか

### 3. モジュール境界・インポートパス

- [ ] フィーチャー間の依存が公開API（index.ts等）経由のみか
- [ ] 内部実装への直接参照がないか
- [ ] パスエイリアスが正しく使用されているか
- [ ] 禁止されたインポートパスが使用されていないか

### 4. ディレクトリ構造

- [ ] 新規ファイルが `docs/repository-structure.md` の定義に従った場所に配置されているか
- [ ] テストファイルの配置がプロジェクト規約に従っているか

### 5. 命名規則（アーキテクチャレベル）

- [ ] フィーチャー名の命名がプロジェクト規約に従っているか
- [ ] ファイル名の命名パターンが既存と一致しているか
- [ ] API層の関数名規則が守られているか

### 6. design.mdとの整合性

- [ ] design.mdで定義されたコンポーネント構成が実装されているか
- [ ] design.mdで定義されたデータフローが守られているか
- [ ] design.mdで定義されたAPIインターフェースが実装通りか

### 7. データフロー・状態管理

- [ ] 状態管理パターンがプロジェクトの規約に従っているか
- [ ] データフェッチングパターンが規約に従っているか
- [ ] グローバル状態とローカル状態の使い分けが適切か

### 8. API設計（バックエンドの場合）

- [ ] RESTful設計原則に従っているか（該当する場合）
- [ ] リクエスト/レスポンスの型定義が適切か
- [ ] エラーレスポンスの形式が統一されているか

## 指摘の分類基準

### MUST（修正必須）

- **レイヤー違反**: 定義されたレイヤー構造への違反
- **依存方向違反**: 下位レイヤーから上位レイヤーへの依存、循環依存
- **design.mdとの不整合**: 設計書と実装の重大な不一致
- **モジュール境界違反**: フィーチャーの内部実装への直接参照
- **禁止パスの使用**: 禁止されたインポートパスの使用

### SHOULD（修正推奨）

- 命名規則の軽微な逸脱
- ディレクトリ構造の軽微な改善提案
- design.mdとの軽微な相違

### MAY（検討事項）

- 将来的なリファクタリング提案
- アーキテクチャの発展的な改善

## 出力フォーマット

```markdown
# アーキテクチャレビュー結果（ラウンド{N}）

## サマリ
- 総合判定: PASS / FAIL
- MUST指摘: {件数}件
- SHOULD指摘: {件数}件
- MAY指摘: {件数}件

## MUST（修正必須）

### 指摘1
- ファイル: {パス}
- カテゴリ: {レイヤー違反 / 依存方向違反 / design.mdとの不整合 / etc.}
- 問題: {問題の説明}
- 修正案: {具体的な修正提案}
- 根拠: {docs/内のどのドキュメントのどの記述に基づくか}

## SHOULD（修正推奨）

### 指摘1
- ファイル: {パス}
- カテゴリ: {カテゴリ}
- 問題: {問題の説明}
- 修正案: {具体的な修正提案}
- 根拠: {根拠}

## MAY（検討事項）

### 指摘1
- ファイル: {パス}
- カテゴリ: {カテゴリ}
- 提案: {提案内容}
- 根拠: {根拠}
```

**出力先**: `.issue/{N}/reviews/review-architecture-r{ラウンド番号}.md`

**総合判定ルール**: MUST指摘が1件でもあれば FAIL、0件なら PASS

全ての指摘に「根拠」フィールドを含め、`docs/` 内のどのドキュメントのどの記述に基づくかを明記すること。
