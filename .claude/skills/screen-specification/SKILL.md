# 画面仕様書作成スキル

画面仕様書を作成・レビューする際に参照するドメイン知識。テンプレート、品質基準、ファイル構成規約を定義する。

## スコープ

- **提供するもの**: 画面仕様書のテンプレート、品質基準、ファイル命名規則
- **提供しないもの**: 作成手順（Agent が定義）、プロジェクト固有の内容

## 入出力

| 項目 | WF2: 新規作成 | WF4: 既存変更 |
|---|---|---|
| 入力 | PRD + 機能設計書 | 要件 + 既存 `docs/screen-specification/` |
| 出力 | `docs/screen-specification/` | 画面変更仕様 |

> WF2（新規作成）: プロジェクト立ち上げ時にドキュメントを新規作成するワークフロー
> WF4（既存変更）: 既存ドキュメントに対して変更・追記するワークフロー

## 機能設計書（functional-design）との関係

画面仕様書は機能設計書の概要UI設計を**詳細化**したもの。

- 機能設計書の画面一覧・画面遷移図を引き継ぐ
- 概要レベルのワイヤーフレームをより詳細に描き直す
- 操作挙動・状態別ワイヤーフレーム・UI要素定義を追加する
- 矛盾がある場合は画面仕様書を正とし、機能設計書も更新を推奨

## ファイル構成

```
docs/screen-specification/
├── index.md              # 画面一覧・画面遷移図・共通挙動仕様
├── components.md         # 共通コンポーネント仕様
├── scr-001-[name].md     # 各画面の詳細仕様
├── scr-002-[name].md
└── ...
```

### ファイル命名規則

```
scr-[3桁連番]-[画面名の英語略称].md
```

例: `scr-001-input.md`, `scr-002-timeline.md`, `scr-003-detail.md`

### 分割の理由

- **コンテキスト節約**: `index.md` + 該当画面ファイルだけ読めば済む
- **更新の局所化**: 1画面の仕様変更が他画面に影響しない
- **レビューの容易さ**: 画面単位でレビュー・承認できる

## テンプレート

| 用途 | テンプレート |
|---|---|
| index.md | `./templates/index.md` |
| 各画面ファイル | `./templates/screen.md` |
| 共通コンポーネント | `./templates/components.md` |

## 品質基準

### 必須チェック項目

- [ ] PRDの機能要件との整合性が取れているか
- [ ] 機能設計書の概要UI設計と矛盾がないか（詳細化であること）
- [ ] すべての画面要素に対して操作時の挙動が定義されているか
- [ ] 画面遷移に抜け漏れがないか
- [ ] ASCIIワイヤーフレームが実際のUIレイアウトを正確に表現しているか
- [ ] 主要な状態（初期・入力中・空・エラー時）が網羅されているか

### ASCIIワイヤーフレームの原則

- 実際のレイアウトを忠実に再現する（要素の配置、サイズ比率）
- 全角・半角を考慮する（日本語テキストは全角幅で計算）
- 主要な状態を網羅する（初期状態、入力中、空状態、エラー時）

### 挙動定義の粒度

- ユーザーアクション単位で定義する（タップ、スワイプ、長押し等）
- 状態遷移を明確にする（操作前の状態 → 操作 → 操作後の状態）
- エッジケースも記述する（ネットワークエラー、空データ、権限拒否時の挙動）
