# 画面仕様書作成スキル

このスキルは、実装が進んだ段階で、各画面の詳細なレイアウトと操作挙動を定義する**画面仕様書**を作成します。

## functional-design（機能設計書）との違い

| | functional-design | screen-specification |
|---|------------------|---------------------|
| **作成タイミング** | プロジェクト初期（setup-project時） | 実装が進んだ後 |
| **UI設計の粒度** | 概要レベル（画面一覧、大まかなレイアウト、画面遷移の全体像） | 詳細レベル（全要素の定義、状態別ワイヤーフレーム、操作ごとの挙動） |
| **入力** | PRDのみ | PRD + functional-design.md + steeringファイル（実装履歴） |
| **視点** | システム設計（データモデル、コンポーネント、アルゴリズム） | ユーザー視点（何が見えて、何を操作して、何が起きるか） |
| **出力** | `docs/functional-design.md` | `docs/screen-specification/` ディレクトリ |

**段階的UI設計の流れ**:

```
1. functional-design（初期）
   → 「だいたいこんなアプリ」の概要UI設計
   → 画面一覧、大まかなレイアウト、画面遷移図

2. 実装を進める（steering）
   → 実際に触って、細かい挙動の理解が深まる

3. screen-specification（実装後）
   → 実装経験を踏まえた詳細画面仕様
   → 全UI要素の定義、状態別ワイヤーフレーム、操作挙動
```

## 前提条件

画面仕様書の作成を開始する前に、以下が完了している必要があります:

### 1. setup-project が実行済みであること

以下の永続ドキュメントが存在する必要があります:

- **`docs/product-requirements.md`**（必須）: プロダクト要求定義書
- **`docs/functional-design.md`**（必須）: 機能設計書（概要レベルのUI設計を含む）

### 2. 実装済み機能のsteeringファイルが存在すること（推奨）

`.steering/` 配下に過去の実装履歴がある場合、画面仕様の詳細化に活用します。実装を通じて得られた知見（UIの変更、操作フローの改善など）が反映されます。

## 既存ドキュメントの優先順位

**重要**: `docs/screen-specification/` に既存の画面仕様書がある場合、以下の優先順位に従ってください:

1. **既存の画面仕様書 (`docs/screen-specification/`)** - 最優先
   - プロジェクト固有の仕様が記載されている
   - このスキルのガイドより優先する

2. **このスキルのガイド** - 参考資料
   - 汎用的なテンプレートと例
   - 既存仕様書がない場合、または補足として使用

**新規作成時**: 「作成プロセス」に従い、テンプレートとガイドを参照
**更新時**: 「更新プロセス」に従い、既存仕様書の構造と内容を維持しながら更新

## 出力先

画面仕様書は以下のディレクトリ構造で保存してください:

```
docs/screen-specification/
├── index.md              ← 画面一覧・画面遷移図・共通挙動仕様
├── components.md         ← 共通コンポーネント仕様
├── scr-001-input.md      ← 入力画面の詳細仕様
├── scr-002-timeline.md   ← タイムライン画面の詳細仕様
├── scr-003-detail.md     ← ネタ詳細画面の詳細仕様
└── ...                   ← 画面ごとに1ファイル
```

### ファイル分割の理由

- **コンテキスト節約**: 特定画面の仕様を参照する際、`index.md` + 該当画面ファイルだけ読めば済む
- **更新の局所化**: 1画面の仕様変更が他の画面ファイルに影響しない
- **レビューの容易さ**: 画面単位でレビュー・承認できる

### ファイル命名規則

```
scr-[番号]-[画面名の英語略称].md
```

例: `scr-001-input.md`, `scr-002-timeline.md`, `scr-003-detail.md`

## テンプレートの参照

画面仕様書を作成する際は、以下のテンプレートを使用してください:

- **index.md用**: `./templates/index.md`
- **各画面ファイル用**: `./templates/screen.md`
- **共通コンポーネント用**: `./templates/components.md`

## 作成プロセス

### ステップ1: 入力ドキュメントの読み込み

以下のドキュメントを順序どおりに読み込みます:

```
1. Read('docs/product-requirements.md')
2. Read('docs/functional-design.md')
3. .steering/ 配下のディレクトリを日付順（昇順）に読み込む
```

#### steeringファイルの読み込み順序（重要）

steeringファイルは**必ず時系列順（日付の古い順）**に読み込んでください。

**理由**: steeringファイルは時系列で機能の追加・変更・削除を記録しています。新しいsteeringファイルが古いsteeringファイルの機能を上書き・削除している場合があるため、最終的な「現在のアプリの状態」を正確に把握するには時系列順の読み込みが必須です。

```
読み込み順の例:
1. .steering/20260212055903-text-note-input/    ← 最初に読む
2. .steering/20260212070155-fix-input-cancel/   ← 次に読む
3. .steering/20260213014322-photo-note-recording/  ← その次
...
```

各steeringディレクトリ内で読み込むファイル:
- `requirements.md` - 要求内容
- `design.md` - 実装設計
- `tasklist.md` - タスクリスト（振り返りセクションに実装結果が記録されている）

### ステップ2: 現在のアプリ状態の把握

読み込んだドキュメントから、以下を整理します:

1. **画面一覧**: 現在アプリに存在するすべての画面
2. **画面遷移**: 画面間のナビゲーション構造
3. **実装済み機能**: PRDの機能要件のうち、実装済みのもの
4. **UI要素**: 各画面に配置されているボタン・入力フィールド等

**重要**: functional-design.mdの概要レベルUI設計を出発点とし、steeringファイルで記録された実装時の変更を上書き適用して「現在の状態」を導き出します。

### ステップ3: index.md の作成

`./templates/index.md` テンプレートに従い、以下を記述します:

1. **画面一覧表**: 全画面のID・種別・概要・対応PRD機能
2. **画面遷移図**: Mermaid記法による画面遷移
3. **画面遷移一覧表**: 全遷移のトリガー・遷移方法
4. **共通挙動仕様**: アプリライフサイクル、エラーハンドリング、アプリ固有の共通仕様（該当する場合）
5. **PRD機能要件との対応表**: 機能→画面→操作のマッピング

**ユーザー確認（1回目）**:

```
「index.md（画面一覧・画面遷移図）の作成が完了しました。内容を確認してください。
承認いただけたら各画面の詳細仕様に進みます。」
```

### ステップ4: 各画面ファイルの作成

`./templates/screen.md` テンプレートに従い、**画面ごとに1ファイル**を作成します:

- ASCIIワイヤーフレーム（状態別）
- Markdownワイヤーフレーム（全UI要素の定義表）
- 操作挙動仕様（全タップ可能要素の挙動）

**1ファイルずつユーザーの承認を得てから次の画面に進む**:

```
「SCR-001（入力画面）の画面仕様書を作成しました。内容を確認してください。
承認いただけたら次の画面に進みます。」
```

### ステップ5: components.md の作成

`./templates/components.md` テンプレートに従い、共通コンポーネント仕様を作成します。

### ステップ6: レビューと改善

生成した仕様書を以下の観点でレビューします:

#### レビュー観点

1. PRDの機能要件との整合性が取れているか
2. functional-design.mdの概要UI設計と矛盾がないか（詳細化であること）
3. steeringファイルで実装済みの変更が反映されているか
4. すべての画面要素に対して操作時の挙動が定義されているか
5. 画面遷移に抜け漏れがないか
6. ASCIIワイヤーフレームが実際のUIレイアウトを正確に表現しているか

## 更新プロセス（既存仕様書がある場合）

既存の画面仕様書を更新する場合は、新規作成プロセスではなく以下の手順に従います。

### ステップ1: 更新対象の特定

1. `docs/screen-specification/index.md` を読み込み、全体構造を把握
2. 更新が必要な画面のファイル（`scr-[番号]-[name].md`）を読み込む
3. 変更の起点を確認する（新しいsteeringファイル、PRD変更、ユーザーからの指示など）

### ステップ2: 影響範囲の確認

以下の観点で、変更の影響範囲を洗い出します:

- **index.md**: 画面一覧・画面遷移図・PRD対応表の更新が必要か
- **該当画面ファイル**: ワイヤーフレーム・UI要素定義・操作挙動の変更が必要か
- **他画面ファイル**: 遷移元・遷移先の画面に影響があるか
- **components.md**: 共通コンポーネントの変更が必要か

### ステップ3: 更新の実施

1. 影響範囲のファイルを**1ファイルずつ**更新する
2. 各ファイルの「最終更新日」を更新する
3. **1ファイルずつユーザーの承認を得てから次に進む**

```
「SCR-001（入力画面）の画面仕様書を更新しました。変更点を確認してください。
承認いただけたら次のファイルに進みます。」
```

### ステップ4: 整合性の確認

更新完了後、以下を確認します:

1. index.md の画面遷移図が更新後の状態と一致しているか
2. 変更した画面から遷移する先・元の画面仕様に矛盾がないか
3. PRD機能要件との対応表が最新か

## 画面仕様の参照方法（実装時）

実装時に画面仕様を参照する場合は、**必要な画面のファイルだけ**を読み込んでください:

```
# 全体像の把握が必要な場合
Read('docs/screen-specification/index.md')

# 特定画面の実装時
Read('docs/screen-specification/index.md')       ← 画面遷移の確認
Read('docs/screen-specification/scr-001-input.md') ← 該当画面の仕様

# 共通コンポーネントの実装時
Read('docs/screen-specification/components.md')
```

## 重要なポイント

### 1. functional-design.md との関係

画面仕様書は functional-design.md の概要UI設計を**詳細化**したものです:

- functional-design.mdの画面一覧・画面遷移図を引き継ぐ
- 概要レベルのASCIIワイヤーフレームをより詳細に描き直す
- 概要にはなかった操作挙動・状態別ワイヤーフレーム・UI要素定義を追加する
- **矛盾がある場合は画面仕様書（実装後の知見）を正とする**が、functional-design.mdも合わせて更新することを推奨

### 2. ASCIIワイヤーフレームの原則

- **実際のレイアウトを忠実に再現する**: 要素の配置、サイズ比率をASCII文字で表現
- **全角・半角を考慮する**: 日本語テキストは全角幅で計算
- **主要な状態を網羅する**: 初期状態、入力中、空状態、エラー時など

### 3. 挙動定義の粒度

- **ユーザーアクション単位で定義する**: ボタンタップ、スワイプ、長押し等
- **状態遷移を明確にする**: 操作前の状態 → 操作 → 操作後の状態
- **エッジケースも記述する**: ネットワークエラー、空データ、権限拒否時の挙動
