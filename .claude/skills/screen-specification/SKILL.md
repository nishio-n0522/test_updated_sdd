# 画面仕様書作成スキル

画面仕様書を作成・レビューする際に参照するドメイン知識。テンプレート、品質基準、ファイル構成規約を定義する。

## スコープ

- **提供するもの**: 画面仕様書のテンプレート、品質基準、ファイル命名規則
- **提供しないもの**: 作成手順（Agent が定義）、プロジェクト固有の内容

## 入出力

| 項目 | WF2: 新規作成 | WF4: 既存変更 |
|---|---|---|
| 入力 | PRD + 機能設計書 | 要件 + 既存 `docs/screen-specification/` |
| 出力 | `docs/screen-specification/` | 画面変更仕様 |

> WF2（新規作成）: プロジェクト立ち上げ時にドキュメントを新規作成するワークフロー
> WF4（既存変更）: 既存ドキュメントに対して変更・追記するワークフロー

## 機能設計書（functional-design）との関係

画面仕様書は機能設計書の概要UI設計を**詳細化**したもの。

- 機能設計書の画面一覧・画面遷移図を引き継ぐ
- 概要レベルのワイヤーフレームをより詳細に描き直す
- 操作挙動・状態別ワイヤーフレーム・UI要素定義を追加する
- 矛盾がある場合は画面仕様書を正とし、機能設計書も更新を推奨

## ユーザー体験設計（user-experience.md）

画面仕様書の作成前に、**ユーザー体験（UX）を定義してからワイヤーフレーム・操作挙動に進む**。機能設計書はおおまかな機能一覧を目的としており、細かいユーザー体験までは定義しないため、画面仕様書のディレクトリにUXドキュメントとして残す。

### UX定義の目的

- ユーザーが画面上でどのような操作体験をするかを具体化する
- 画面構成・ビュー切替・モード切替などの構造を確定する
- 設計判断とその理由を記録し、後から参照可能にする

### UX定義のプロセス

1. **AIがUXシナリオのドラフトを提案する**: 機能設計書と類似ツールのドメイン知識から、具体的な操作フローを起草する
2. **ユーザーが承認・修正する**: ユーザーに設計を委ねるのではなく、AIが提案してユーザーがレビューする形をとる
3. **不明点はA/B/C案方式で質問する**: 自由記述を求めず、具体的な選択肢を提示して判断を仰ぐ
4. **確定したUXをドキュメント化する**: `user-experience.md` としてディレクトリに保存する

### user-experience.mdの位置づけ

- wireframes.md / behaviors.md の**作成プロセスで先に作る**入力ドキュメント
- wireframes.md / behaviors.md から参照する必要はない（UX検討結果は画面仕様に織り込み済み）
- 設計判断の経緯を後から確認するためのドキュメントとして残す

## UXレビュー（画面仕様作成後）

画面仕様（wireframes.md + behaviors.md）の作成後、`review-ux` スキルに基づくUXレビューを実施する。

### レビューの位置づけ

```
1. UX設計 (user-experience.md) → ユーザー承認
2. 画面仕様作成 (wireframes.md + behaviors.md) → ユーザー承認
3. UXレビュー (review-ux) → MUST指摘を修正 → PASS判定まで繰り返し
```

### レビューの観点

- `review-ux` スキルの8つのヒューリスティクス（H1〜H8）に基づく
- ユーザーフローの完結性、操作の一貫性、エラー防止などを検証する
- 詳細は `review-ux` スキルを参照

### PASS基準

- MUST指摘が0件であること
- SHOULD/MAY指摘はユーザー判断で対応可否を決定する

## ファイルサイズ制限

> **重要**: AIによるドキュメントの読み書き精度を維持するため、以下の制限を厳守する。

| 指標 | 推奨 | 上限 |
|---|---|---|
| 1ファイルあたりの行数 | 300行以下 | 400行 |
| 1ファイルあたりのトークン数 | ~8k tokens以下 | ~12k tokens |

### 上限を超える場合の対応

1ファイルで上限を超えると判断した場合、**ディレクトリ分割**を行う:

```
scr-XXX-[name]/
├── wireframes.md    # 概要 + ASCIIワイヤーフレーム（状態別）
└── behaviors.md     # UI要素定義表 + 操作挙動仕様 + ビジネスルール
```

### 分割判断の目安

| 条件 | 形式 |
|---|---|
| ワイヤーフレーム3状態以下 かつ 操作5個以下 | 単一ファイル `scr-XXX-[name].md` |
| ワイヤーフレーム4状態以上 または 操作6個以上 | ディレクトリ分割 |

## ファイル構成

```
docs/screen-specification/
├── index.md                     # 画面一覧・画面遷移図・共通挙動仕様
├── shared-components.md         # グローバル共通コンポーネント（複数SCRで共通）
├── user-experience.md           # ユーザー体験設計（UXシナリオ・設計判断）
├── scr-001-[name].md            # 小さい画面: 単一ファイル
├── scr-002-[name]/              # 大きい画面: ディレクトリ分割
│   ├── shared-components.md     # 画面内共通コンポーネント（この画面内で共通）
│   ├── wireframes-[view].md     # ビュー別ワイヤーフレーム
│   └── behaviors-[view].md      # ビュー別操作挙動
└── ...
```

> `user-experience.md` はUX検討が必要な画面でのみ作成する。単純な画面（ホーム画面等）では不要。

### ファイル命名規則

- 単一ファイル: `scr-[3桁連番]-[画面名の英語略称].md`
- ディレクトリ分割: `scr-[3桁連番]-[画面名の英語略称]/wireframes.md` + `behaviors.md`

### 共通コンポーネント（2層構造）

共通コンポーネントは**グローバル共通**と**画面内共通**の2層で管理する。**同じUI要素定義を複数ファイルにコピーしない。**

#### グローバル共通（shared-components.md）

アプリケーション全体で共通のUIコンポーネント（複数のSCR-xxxをまたいで使用されるもの）を定義する。

- 配置: `docs/screen-specification/shared-components.md`
- 例: グローバルサイドバー、プロジェクトヘッダー、トースト通知
- COMP番号（COMP-001, COMP-002, ...）で採番する

#### 画面内共通（scr-xxx-yyy/shared-components.md）

特定の画面ディレクトリ内の複数ビュー・サブ画面で共通のコンポーネントを定義する。グローバルには共通化されないが、画面内では重複を避けたいものが対象。

- 配置: `docs/screen-specification/scr-xxx-yyy/shared-components.md`
- 例: タスク画面内の詳細ドロワー、ビュー切替ボタングループ
- グローバルCOMP番号とは別体系とし、名称で参照する
- グローバル共通コンポーネントへの参照は `../shared-components.md` を指定する

#### 参照の記載形式

各画面のUI要素定義表では、共通コンポーネントのセクションに参照先を明記する:

```markdown
### グローバル共通コンポーネント
> グローバル左サイドバーの UI 要素定義・操作は ../shared-components.md (COMP-001) を参照。

### 画面内共通コンポーネント
> タスク詳細ドロワーの UI 要素定義・操作は shared-components.md（タスク画面内）を参照。
```

### 分割の理由

- **コンテキスト節約**: `index.md` + 該当画面ファイルだけ読めば済む
- **更新の局所化**: 1画面の仕様変更が他画面に影響しない
- **レビューの容易さ**: 画面単位でレビュー・承認できる
- **AI操作の安全性**: ファイルサイズを制限することで読み書きの失敗を防ぐ

## テンプレート

| 用途 | テンプレート |
|---|---|
| index.md | `./templates/index.md` |
| 画面仕様（単一ファイル） | `./templates/screen.md` |
| 画面仕様（分割: ワイヤーフレーム） | `./templates/wireframes.md` |
| 画面仕様（分割: 操作挙動） | `./templates/behaviors.md` |

## 品質基準

### 必須チェック項目

- [ ] PRDの機能要件との整合性が取れているか
- [ ] PRDの非機能要件との対応がindex.mdに記載されているか
- [ ] 機能設計書の概要UI設計と矛盾がないか（詳細化であること）
- [ ] すべての画面要素に対して操作時の挙動が定義されているか
- [ ] 画面遷移に抜け漏れがないか
- [ ] ASCIIワイヤーフレームが実際のUIレイアウトを正確に表現しているか
- [ ] 主要な状態（初期・入力中・空・エラー時）が網羅されているか
- [ ] 共通コンポーネントが重複定義されていないか（shared-components.mdへの参照になっているか）
- [ ] 各ファイルが400行以下に収まっているか

### ASCIIワイヤーフレームの原則

- 実際のレイアウトを忠実に再現する（要素の配置、サイズ比率）
- 全角・半角を考慮する（日本語テキストは全角幅で計算）
- 主要な状態を網羅する（初期状態、入力中、空状態、エラー時）
- 共通コンポーネント（サイドバー等）はワイヤーフレーム内にアイコン表現で含め、凡例でshared-components.mdを参照する

### 挙動定義の粒度

- ユーザーアクション単位で定義する（タップ、スワイプ、長押し等）
- 状態遷移を明確にする（操作前の状態 → 操作 → 操作後の状態）
- エッジケースも記述する（ネットワークエラー、空データ、権限拒否時の挙動）
