# PRD作成スキル

PRD（プロダクト要求定義書）を作成・レビューする際に参照するドメイン知識。テンプレートと品質基準を定義する。

## スコープ

- **提供するもの**: PRDのテンプレート、品質基準、記述ガイドライン
- **提供しないもの**: 作成手順（Agent が定義）、プロジェクト固有の内容

## 入出力

| 項目 | 内容 |
|---|---|
| 入力 | アイデアメモ（`docs/ideas/YYYYMMDD-memo.md`） |
| 出力 | `docs/product-requirements.md` |

## 既存ドキュメントの優先順位

| 優先度 | 対象 | 説明 |
|---|---|---|
| 1（最優先） | `docs/product-requirements.md` | プロジェクト固有の要件。このスキルのガイドより優先 |
| 2（参考） | このスキル | 汎用テンプレート。既存PRDがない場合に使用 |

## PRDの責務

PRDは「**なぜ作るか？何を作るか？**」に答えるドキュメントである。

### 記載する内容

- 背景・課題、プロダクトの概要・目的
- ターゲットユーザー（ペルソナ）
- KPI・成功指標
- 機能要求（ユーザーストーリー + 概要説明 + 優先度）
- 非機能要求（測定可能な数値で定義）
- スコープ外の定義

### 記載しない内容

- **具体的なUI仕様**: 画面レイアウト、操作の詳細挙動、undo/redo回数などは画面仕様書の責務
- **技術的な実現方法**: 具体的なフォーマット（ONNX等）、フレームワーク名、通信プロトコルなどはアーキテクチャ設計書の責務
- **詳細な受け入れ条件**: チェックリスト形式の受け入れ条件は機能設計書の責務

### 抽象度の基準

PRDの機能要求は **KPIを除き抽象的な記述にとどめる**。

| 抽象度 | PRDに書く（適切） | PRDに書かない（詳細すぎる） |
|---|---|---|
| 機能概要 | 「ノードベースUIで検査フローを視覚的に設計できる」 | 「undo/redoが10ステップ以上対応」 |
| 技術要素 | 「AIモデルを検査フローに組み込める」 | 「ONNX形式＋メタ情報で管理」 |
| データ規模 | 「大量の画像データセットを管理できる」 | 「フォルダ一括インポートで画像を取り込める」 |

## 品質基準

### 必須チェック項目

- [ ] プロダクトビジョンが明確で測定可能か
- [ ] ターゲットユーザー（ペルソナ）が具体的に定義されているか
- [ ] 成功指標（KPI）がSMART原則に従っているか
- [ ] 全機能にユーザーストーリーと概要説明があるか
- [ ] 全機能に優先度（P0/P1/P2）が設定されているか
- [ ] 非機能要件が測定可能な数値で定義されているか
- [ ] スコープ外が明示されているか
- [ ] 責務境界を逸脱していないか（具体的なUI仕様・技術的実現方法・詳細な受け入れ条件が含まれていないか）

### 優先度の定義

| 優先度 | 意味 | 基準 |
|---|---|---|
| P0（必須） | MVPに含める | これがないとプロダクトとして成立しない |
| P1（重要） | 初期リリース後すぐに追加 | ユーザー体験に大きく影響する |
| P2（できれば） | 将来的に検討 | あると便利だが必須ではない |

### 記述ガイドライン

#### 具体性と測定可能性

KPIと非機能要件は具体的で測定可能でなければならない。
機能要求は抽象的な記述にとどめ、詳細は機能設計書に委ねる。

| 種別 | 悪い例 | 良い例 |
|---|---|---|
| パフォーマンス | システムは高速である | コマンド実行時間: 100ms以内 |
| ユーザビリティ | 使いやすい | 新規ユーザーが5分以内に基本操作を習得できる |
| 機能要求 | undo/redoが10ステップ以上対応 | 直感的な操作で検査フローを構築できる |

#### ユーザーストーリー形式

```
[ユーザー]として、[目的]のために、[機能]が欲しい
```

#### KPIのSMART原則

- **S**pecific（具体的）: 何を測定するか明確
- **M**easurable（測定可能）: 数値で測定できる
- **A**chievable（達成可能）: 現実的な目標
- **R**elevant（関連性）: ビジネス目標と関連
- **T**ime-bound（期限）: 達成期限を設定
