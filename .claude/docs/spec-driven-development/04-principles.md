# 4. Skill / Agent の設計原則

## 4.1. 3つの関心事の分離

| 層 | 担うもの | 定義場所 | 内容 |
|---|---|---|---|
| **Agent** | **プロセス**（どう働くか） | `.claude/agents/` | 作業手順、判断ロジック、入出力の制約、ファイルスコープ |
| **Skill（知識型）** | **ドメイン知識**（何を知っているか） | `.claude/skills/` | テンプレート、チェックリスト、設計ガイドライン、品質基準 |
| **入力ドキュメント** | **タスク固有の情報**（何を作るか） | `docs/`, `.issue/` | PRD、機能設計書、issue内容 |

### なぜ分離するか

1. **再利用性**: 同じ知識（skill）を複数のエージェントが参照できる
2. **保守性**: 知識の更新が skill 1ファイルで完結し、全エージェントに反映される
3. **コンテキスト効率**: agent定義は常に読み込まれるが、skillは必要なものだけ選択的に読み込める
4. **テスト容易性**: skill単体で内容の正しさを検証できる

## 4.2. Agent定義に書くもの / 書かないもの

```
Agent定義（.claude/agents/xxx.md）
  ├─ 書くもの
  │    ├─ 役割の定義（何者か）
  │    ├─ 作業プロセス（どんな手順で進めるか）
  │    ├─ 判断基準（どんな状況で何を選ぶか）
  │    ├─ 入出力の制約（何を読み、何を書くか）
  │    ├─ 参照すべきskillの指定（コア / 状況依存）
  │    └─ 他エージェントとの境界（やらないこと）
  │
  └─ 書かないもの（= skillに置く）
       ├─ ドキュメントのテンプレート・フォーマット
       ├─ 品質基準・チェックリスト
       ├─ 設計パターン・ベストプラクティス
       └─ 用語定義・命名規則
```

## 4.3. Agentが参照するSkillの2パターン

| パターン | 説明 | 例 |
|---|---|---|
| **コアskill** | そのエージェントの役割に不可欠。agent定義に明記し常時参照する | PRD作成エージェント → `prd-writing` skill |
| **状況依存skill** | タスク内容に応じて選択的に読み込む。カタログ（index）経由で判断する | 実装仕様エージェント → `design-patterns/index.md` を見て該当パターンのみ読み込み |

## 4.4. Skill / Agent の選択基準

オーケストレーターを skill にするか agent にするかは、**ユーザーとの対話が必要か**で判断する。

| 条件 | 選択 | 理由 |
|---|---|---|
| ユーザーとの対話・承認が中心 | **Skill**（メイン会話で実行） | 直接対話・承認ゲートが自然にできる |
| 自律的に実行（承認なし） | **Agent**（別プロセスで実行） | 独立コンテキストで長時間の自律実行に適する |

```
WF1〜4: User → skill(オーケストレーター) [→ Agent(s)] → 成果物
                  ↑ 直接対話・承認

WF5:    User → skill(入り口) → agent(autopilot-manager) → Agent(s) → 成果物
                                    ↑ 自律実行（承認なし）
```

## 4.5. 関係図

```
ユーザー → skill(オーケストレーター型) → agent(s) → 成果物
                                           ↓
                                     skill(専門知識型)を参照
```
