# アイデアメモ

## メタ情報

- **作成日**: 2026-03-15
- **種別**: 新規プロダクト

## 概要

マインドマップ駆動のプロジェクト管理アプリケーション。マインドマップで視覚的にプロジェクトを分解し、各ノードをそのままタスクへ変換できる。カンバンボードやガントチャートによる進捗管理、Wiki機能によるナレッジ管理も統合した、計画から実行までをワンストップで行えるWebアプリケーション。

## 背景・課題

### 現状

- プロジェクト計画時にマインドマップツールでタスクを分解し、別のタスク管理ツールに手動で転記している
- タスクの粒度や網羅性の判断が属人的で、抜け漏れが発生しやすい
- 計画（全体構造）と実行（タスク管理）が別ツールに分断されており、全体像と進捗の関係が見えにくい

### 課題

- **ツール間の分断**: マインドマップ（計画）とタスク管理（実行）が別ツールのため、転記の手間・情報の不一致が発生する
- **タスク分解の困難さ**: 漠然としたプロジェクト目標を適切な粒度のタスクに落とし込むのが難しく、経験の浅いメンバーは特に苦戦する
- **構造と進捗の乖離**: タスク管理ツールではフラットなリスト表示が中心で、プロジェクト全体の構造の中で各タスクの位置付けや進捗を把握しにくい

## ターゲットユーザー

### ユーザー属性

- **誰**: エンジニア、プロジェクトマネージャー、チームリーダー
- **経験レベル**: プロジェクト管理初心者〜中級者。マインドマップの直感的UIにより、複雑なツールに慣れていないユーザーもカバー
- **規模感**: 小〜中規模チーム（個人利用も可）

### 利用シーン

- プロジェクト管理を任されたジュニアエンジニアが、マインドマップでタスクを視覚的に分解し、そのまま実行管理に移行する
- プロジェクトマネージャーが、マインドマップ上でプロジェクト構造と進捗状況を俯瞰し、ボトルネックを特定する
- チームメンバーがカンバンボードやガントチャートで自分のタスクを確認し、ステータスを更新する

### 現在の代替手段

- マインドマップツール（Miro、MindMeister等）で計画し、タスク管理ツール（Jira、Asana等）に手動転記
- スプレッドシートでWBSを作成し、タスク管理ツールと併用
- **代替手段の問題点**: ツール間の転記コスト、情報の二重管理による不整合、全体構造と個別タスクの関係が可視化されない

## 提案するアプローチ

### 基本アイデア

マインドマップをプロジェクト構造の中心に据え、各ノードをタスクとして直接管理できるWebアプリケーション。マインドマップ（構造ビュー）、カンバンボード（ステータスビュー）、ガントチャート（タイムラインビュー）を同一データ上の異なるビューとして提供し、計画と実行をシームレスに統合する。

### 主要機能

- **マインドマップエディタ**: ノードの作成・編集・親子関係管理、ズーム・パン操作、自動保存。プロジェクトの全体構造を視覚的に構築
- **ノード→タスク変換**: マインドマップのノードにステータス・優先度・期限・担当者を設定し、タスクとして管理可能にする
- **カンバンボード**: タスクをステータス別カラムに表示。ドラッグ＆ドロップでステータス変更
- **ガントチャート**: タスクをタイムライン上に表示し、スケジュールを可視化
- **プロジェクト管理**: 複数プロジェクト対応、メンバー招待、ロール管理（オーナー/編集者/閲覧者）
- **プロジェクトダッシュボード**: タスク状況の集計、ツリーベースの進捗可視化、期限超過アラート
- **Wiki/ドキュメント**: Markdownベースのナレッジベース。プロジェクトに紐づく階層構造のドキュメント管理
- **マインドマップ履歴**: スナップショット保存と自動保存タイムラインによる状態復元
- **メッセージング・通知**: アプリ内コミュニケーション、タスク変更通知

## MVPスコープ

### 含めるもの

- **マインドマップエディタ**: プロジェクト構造の視覚的構築（ノード作成・編集・親子関係・自動保存）
- **ノード→タスク変換**: ノードへのステータス・優先度・期限・担当者の設定
- **カンバンボード**: ステータス別タスク表示、ドラッグ＆ドロップによるステータス変更
- **ガントチャート**: タスクのタイムライン表示
- **プロジェクト管理**: 複数プロジェクト、メンバーロール管理
- **ユーザー認証**: メール/パスワード認証、パスワードリセット

### 含めないもの（将来検討）

- プロジェクトダッシュボード: MVP後の第2優先
- Wiki/ドキュメント: ダッシュボードの次に対応
- マインドマップ履歴: Wiki機能と連動して提供
- メッセージング・通知: 履歴機能の次に対応
- ネイティブモバイルアプリ: スコープ外
- 外部ツール連携（Jira/Slack等）: スコープ外
- AI機能: スコープ外
- オフライン対応: スコープ外
- リアルタイム共同編集: スコープ外

## 前提・仮説・リスク

### 前提条件

- ユーザーはWebブラウザからアクセスする（デスクトップ利用が主）
- チーム規模は数名〜数十名程度を想定
- インターネット接続環境で利用する

### 検証すべき仮説

- マインドマップからタスクへのシームレスな変換が、従来のツール分断よりも効率的か
- マインドマップベースのプロジェクト構造表示が、タスクの網羅性・粒度の改善に寄与するか
- 複数ビュー（マインドマップ/カンバン/ガント）の統合が、ユーザーにとって直感的に使えるか
- プロジェクト管理初心者がマインドマップUIを通じて適切なタスク分解を行えるか

### リスク

- **マインドマップUI操作性リスク（高）**: ノード数が増えた際の操作性・視認性が低下する可能性。レイアウトアルゴリズムやフィルタリング機能で対処が必要
- **描画パフォーマンスリスク（高）**: 大量ノード（100+）のマインドマップ描画・操作時のパフォーマンス。React Flowの最適化とデータ構造の工夫が必要
- **データ同期の複雑性（中）**: マインドマップ状態とタスクデータの双方向同期における整合性維持。楽観的更新と競合解決の設計が重要
- **既存ツールからの移行障壁（中）**: ユーザーが既にJira等のツールに慣れている場合、移行コストが障壁になる可能性

## 補足メモ

- **技術スタック**: Next.js 16（App Router）+ React 19、Supabase（PostgreSQL + Auth）、React Flow（マインドマップ）、dnd-kit（カンバン）、Milkdown（Wikiエディタ）
- **状態管理**: TanStack Query（サーバー状態）+ Zustand（クライアント状態）
- **データ永続化**: マインドマップ状態をPostgreSQLのJSONカラムに保存、タスクデータはwork_itemsテーブルで管理（Ltreeによる階層クエリ対応）
- **多言語対応**: next-intlによる日本語/英語対応（Cookieベースのロケール管理）
- **マインドマップの参考UI**: React Flow公式サンプル
