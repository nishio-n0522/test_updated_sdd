# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称

**FlowInspect** - ノードベース画像検査プラットフォーム [要確認: プロダクト名は仮称]

### プロダクトコンセプト

- **ビジュアルパイプライン構築**: Node-RED風のノードベースUIで、コードを書かずに検査フローを視覚的に設計できる
- **従来処理とAIのシームレス統合**: 従来型画像処理ノードとAI推論ノードを同一フロー上で自由に組み合わせられる
- **現場主導の検査自動化**: 製造技術担当やオペレーターが自らフローを構築・調整し、目視検査からの段階的な自動化を実現する

### プロダクトビジョン

製造現場における画像検査の自動化は、これまで専門家による複雑な開発作業を必要としてきた。FlowInspectは、ノードベースのビジュアルエディタを通じて、製造技術担当者が自らの知識と経験を検査フローとして直接表現できる環境を提供する。従来型画像処理とAI推論を柔軟に組み合わせることで、簡単な検査から複雑な目視代替検査まで対応できる。最終的には、品質のばらつきを排除し、検査コストを削減しながら、現場のデジタル化・知識の体系化を推進する。

### 目的

- 内製アプリの前処理フロー制限を解消し、複雑な検査要件に対応できる柔軟な検査パイプラインを実現する
- 難易度の高い検査における人による目視検査への依存を低減し、品質ばらつきと検査コストを削減する
- 従来型画像処理とAIモデルを統合した検査フローの構築手段を提供し、既存ツールの限界を超える検査精度を達成する

- 期待する解決策: ドラッグ＆ドロップで直感的にノードを接続し、試行錯誤しながら最適な検査フローを短期間で構築できること

## 成功指標 (KPI)

### プライマリーKPI

- **目視検査代替率**: AIフローを活用した検査ラインにおいて、目視検査工数を50%以上削減（MVP後12ヶ月以内）[要確認: 目標値は現場の実態に合わせて調整が必要]
- **フロー構築リードタイム**: 検査ラインの立ち上げ時、フロー構築にかかる時間を既存内製アプリ比で30%短縮（初期展開後6ヶ月以内）[要確認]
- **検査精度**: AIフロー適用ラインにおいて、不良検出率95%以上・偽陽性率5%以下を達成 [要確認: 検査対象によって異なるため要調整]

### セカンダリーKPI

- **ユーザー習熟時間**: 画像処理基礎知識を持つユーザーが、初めてのフロー構築（基本的な検査パイプライン）を2時間以内に完了できる（リリース後3ヶ月以内に検証）
- **内製アプリ移行率**: クラウド外の機能が揃った段階で、既存内製アプリ適用ラインの70%以上をFlowInspectへ移行（移行フェーズ開始後12ヶ月以内）[要確認]
- **テンプレート活用率**: 新規フロー構築のうち40%以上がテンプレートを起点として作成される [要確認]

## 機能要件

### コア機能 (MVP)

#### ノードベースフローエディタ

**ユーザーストーリー**:
製造技術担当として、検査ラインの立ち上げ時に最適な検査フローを短期間で構築するために、ドラッグ＆ドロップで処理ノードを接続して検査パイプラインを視覚的に設計できる機能が欲しい

**受け入れ条件**:

- [ ] ノードをキャンバス上にドラッグ＆ドロップで配置できる
- [ ] ノード間をポートの接続線（エッジ）でつなぎ、データフローを定義できる
- [ ] 各ノードのパラメータをサイドパネルまたはモーダルで編集できる
- [ ] 条件分岐ノードを使って、検査結果に応じた分岐フローを構築できる
- [ ] フローの保存・読み込みができる
- [ ] フローの実行（テスト実行を含む）ができる
- [ ] キャンバスの拡大縮小・スクロールが直感的に操作できる
- [ ] undo/redo が10ステップ以上対応している

**優先度**: P0（必須）

#### 従来型画像処理ノード群

**ユーザーストーリー**:
製造技術担当として、適切な前処理を組み合わせることで検査精度を高めるために、フィルタリング・二値化・エッジ検出・形状マッチングなどの画像処理をノードとして選択して使える機能が欲しい

**受け入れ条件**:

- [ ] 以下のノードカテゴリが提供されている: 画像入力、前処理（フィルタリング・二値化・エッジ検出・モルフォロジー変換等）、特徴抽出（形状マッチング・輪郭検出等）、判定、結果出力
- [ ] 各ノードにパラメータが設定でき、変更がリアルタイムにプレビューへ反映される [要確認: リアルタイム反映の範囲はパフォーマンスに依存]
- [ ] ノードの入出力ポートの型（画像・数値・フラグ等）が視覚的に区別できる
- [ ] 非対応の接続を行った場合にエラーが表示される
- [ ] MVPとして最低20種類以上の基本ノードが提供される [要確認: ノード種類の優先度は別途定義が必要]

**優先度**: P0（必須）

### 将来的な機能 (Post-MVP)

#### 画像アセット管理

プロジェクト内で使用する画像データ（サンプル画像・テスト画像等）を整理・管理する機能。フォルダ構造での整理、タグ付け、検索に対応する。

**優先度**: P1（重要）

#### アノテーション機能

AI学習用に画像へのラベル付け・バウンディングボックス・セグメンテーションマスクのアノテーションをアプリ内で実施できる機能。

**優先度**: P1（重要）

#### AI学習機能

アプリ内でAIモデルの学習を実行する機能。ローカルGPU（CUDA対応）またはクラウドGPUを利用した学習に対応。対象AIタスク: 分類、物体検出、セグメンテーション、良品学習、OCR。

**優先度**: P1（重要）

#### AI推論ノード

学習済みAIモデルをノードとして検査フローに組み込む機能。分類、物体検出、セグメンテーション、良品学習、OCRの各AIタスクをノードとして配置可能。

**優先度**: P1（重要）

#### 画像処理フロー評価機能

作成した検査フローの精度を定量的に評価する機能。テスト用画像セットをフローに通して実行し、混同行列・確率分布・精度/再現率等の統計情報を可視化する。AI推論ノードが利用可能になった段階で提供。

**ユーザーストーリー**:
品質管理担当者として、検査フローを現場展開する前に必要な精度を達成しているか確認するために、テスト画像セットを使ったフロー評価と精度指標の可視化機能が欲しい

**受け入れ条件**:

- [ ] テスト用画像セット（正常品・不良品ラベル付き）を指定してフローを一括実行できる
- [ ] 混同行列（TP/FP/TN/FN）が表示される
- [ ] 精度（Accuracy）・適合率（Precision）・再現率（Recall）・F1スコアが表示される
- [ ] 確率分布グラフ（AI推論ノード使用時）が表示される
- [ ] 評価結果をCSVまたはPDF形式でエクスポートできる

**優先度**: P1（重要）

#### 検査実行画面（現場運用向け）

工場現場のオペレーターが日常的に検査を実行するためのシンプルな操作画面。フロー構築画面とは分離した、実行専用のUI。

**優先度**: P1（重要）

#### テンプレート機能

よくある検査パターン（外観検査、寸法測定、文字認識等）を検査フローのテンプレートとして提供し、画像処理知識が少ないユーザーでも起点として利用できる機能。

**優先度**: P2（できれば）[要確認: 社内展開フェーズでの優先度は高い可能性があるため要調整]

#### クラウド連携機能

検査データのクラウド保存、検査設定のバックアップ、クラウドGPUを使ったAI学習。外販・協力会社展開フェーズで対応。

**優先度**: P2（できれば）

## 非機能要件

### パフォーマンス

- **フローエディタの操作レスポンス**: ノードの追加・移動・接続操作が100ms以内に画面反映される
- **画像処理ノードのプレビュー更新**: パラメータ変更後のプレビュー画像更新が1秒以内に完了する（1080p画像を基準）[要確認: 処理の複雑度によって要調整]
- **フロー実行スループット**: 基本的な従来型画像処理フロー（5ノード以下）で、カメラ入力30fps相当の処理速度を維持できる [要確認: 検査対象・ノード数によって異なるため要調整]
- **アプリ起動時間**: コールドスタートから操作可能状態まで10秒以内

### ユーザビリティ

- **習熟時間**: 画像処理の基礎知識を持つユーザーが、チュートリアルなしで基本的なフローを構築できるよう、2時間以内に操作を習得できる
- **エラーメッセージ**: フロー実行エラー発生時、原因ノードとエラー内容がユーザーに分かる形で表示される
- **操作の取り消し**: undo/redo で誤操作を10ステップ以上遡れる
- **アクセシビリティ**: UIテキストの最小フォントサイズ12px以上。カラーのみに依存しない情報表現（アイコン・ラベルを併用）

### 信頼性

- **フローデータの自動保存**: 編集中のフローを5分ごとに自動保存し、アプリのクラッシュ時にデータを復元できる
- **フロー実行の安定性**: 画像処理フローの実行中に未処理例外が発生した場合、アプリ全体をクラッシュさせずにエラーを通知してフローを停止する
- **後方互換性**: フローファイルのフォーマットは後方互換性を維持し、旧バージョンで保存したフローを新バージョンで読み込める

### セキュリティ

- **ローカルデータ保護**: フローファイルおよび画像データはローカルに保存し、ユーザーの許可なく外部に送信しない
- **クラウド通信の暗号化**: クラウド連携機能使用時はTLS 1.2以上での通信を保証する（クラウド連携機能実装時に適用）
- **ライセンス管理**: アプリのライセンス認証を実装し、不正利用を防止する [要確認: 外販フェーズでの要件として詳細化が必要]

### スケーラビリティ

- **ノード数の上限**: 単一フロー内で100ノード以上を扱えるパフォーマンスを維持する
- **画像データ規模**: プロジェクト内で10,000枚以上の画像アセットを管理できる（画像アセット管理機能実装時に適用）[要確認]
- **マルチライン対応**: 複数の検査ラインに対してそれぞれ独立したフローを管理できる（プロジェクト単位での分離）

## スコープ外

明示的にスコープ外とする項目（現時点）:

- クラウド機能全般（検査データのクラウド保存・バックアップ・クラウド学習）: 最終フェーズで対応
- Webアプリ版・モバイル版: デスクトップアプリ（Tauri）のみ対象
- マルチテナント・ライセンス管理の詳細設計: 外販フェーズで検討
- 外部システム（MES・ERPなど）とのAPI連携: 将来検討
- リモートカメラ・PLCなどの産業機器との直接通信インターフェース: 将来検討 [要確認]
- Webベースの検査結果ダッシュボード（管理者向け）: 将来検討

## 付録: 前提条件とリスク

### 前提条件

- ユーザー（製造技術担当）は画像処理の基礎知識（フィルタリング・二値化等）を持っている。知識のないユーザーは教育プログラムで補完する
- AI推論・学習機能の利用時はユーザーのローカルマシンにGPU（CUDA対応）が搭載されている
- アプリケーションは工場のオンプレミスネットワーク環境でデスクトップアプリとして動作する

### リスクと対応方針

| リスク | 重要度 | 対応方針 |
|---|---|---|
| ノードUI操作性リスク: ノードベースUIが現場ユーザーに受け入れられない | 高 | 早期プロトタイプでユーザビリティテストを実施。テンプレートで操作コストを下げる |
| 描画パフォーマンスリスク: TauriベースのUIでリアルタイム検査時の描画速度が不足する | 高 | C++バックエンドとの通信オーバーヘッドを早期計測。処理パイプラインの分離設計を検討 |
| 技術スタックの複雑性: React+Tauri・C++・Pythonの3言語構成による開発・保守コスト増大 | 中 | 言語間インターフェースを最小化する設計。各言語の責務を明確に分離する |
| 外販時のマルチテナント対応: ライセンス管理・テナント分離の設計が後付けになる | 低（将来） | 外販検討フェーズ開始前にアーキテクチャレビューを実施 |
